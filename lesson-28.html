CHƯƠNG 28

Securing Web App (Bảo mật ứng dụng Web)

Triển khai toàn bộ vòng đời bảo mật cho ứng dụng web MVC: Cookie Authentication, tạo và giải mã authentication ticket, phân quyền theo claim, đăng xuất và bật HTTPS.

1 Tổng quan Authentication & Authorization

Bài 1 · Authentication and Authorization Overview

Chương này tập trung vào hai khái niệm cốt lõi của bảo mật ứng dụng web:

  • Authentication (Xác thực): xác định người dùng là ai. Người dùng cung cấp thông tin đăng nhập (email + mật khẩu), server xác minh và cấp một authentication ticket (thẻ xác thực).
  • Authorization (Phân quyền): xác định người dùng được làm gì. Sau khi đã xác thực, mỗi request tiếp theo gửi kèm authentication ticket; server kiểm tra ticket đó để quyết định có cho phép truy cập tài nguyên hay không.

Luồng hoạt động tổng thể

  1. Người dùng điền thông tin đăng nhập và gửi form Login.
  2. Server xác minh thông tin. Nếu hợp lệ, server tạo authentication ticket và trả về cho trình duyệt dưới dạng cookie (hoặc JWT).
  3. Với mỗi request tiếp theo, trình duyệt tự động đính kèm cookie vào HTTP header.
  4. Middleware Authentication đọc cookie, giải mã thành đối tượng ClaimsPrincipal và gán vào HttpContext.User.
  5. Middleware Authorization kiểm tra HttpContext.User: nếu thoả điều kiện (đã đăng nhập, đúng role...) thì cho phép; ngược lại trả về 401/403.
  6. Khi đăng xuất, server xoá cookie khỏi trình duyệt — ticket không còn tồn tại, người dùng phải đăng nhập lại.

🗝️Ba nhóm việc chính: (1) Tạo authentication ticket khi đăng nhập, (2) Giải mã ticket trong mỗi request (authentication middleware), (3) Phân quyền dựa trên thông tin trong ticket (authorization middleware).

Hai scheme phổ biến

SchemeĐặc điểmDùng cho
Cookie AuthenticationTicket lưu trong cookie trình duyệt, server-side session-likeWeb App (MVC, Razor Pages)
JWT (Bearer Token)Ticket là chuỗi JSON Web Token trong Authorization headerWeb API, SPA, mobile

Chương này tập trung vào Cookie Authentication — phù hợp nhất cho ứng dụng web MVC truyền thống. JWT sẽ được trình bày trong chương tiếp theo (Securing Web API).

2 Authentication Ticket là gì

Bài 2 · Authentication Ticket

Authentication ticket là một tập hợp thông tin được mã hoá và gửi đến trình duyệt sau khi đăng nhập thành công. Nó hoạt động giống như tấm vé vào cổng: bạn mua vé một lần rồi dùng vé đó để vào sự kiện — tương tự, bạn đăng nhập một lần rồi dùng ticket cho mọi request tiếp theo.

Ticket chứa những gì?

Thành phầnVai tròKiểu trong ASP.NET Core
User information (Claims)Thông tin người dùng: tên, email, role, quyền...Claim, ClaimsIdentity, ClaimsPrincipal
ExpiryThời điểm ticket hết hạnAuthenticationTicket.Properties.ExpiresUtc
Authentication SchemeCách ticket được tạo/giải mã (Cookie hay JWT?)Tên scheme đăng ký: "CookieScheme"

Mô hình Claims trong ASP.NET Core

Thông tin người dùng được biểu diễn theo cấu trúc 3 tầng:

  • Claim — một cặp kiểu/giá trị đơn lẻ, ví dụ Name = "admin@company.com", Role = "Admin".
  • ClaimsIdentity — tập hợp các Claim tạo thành một "danh tính". Phải chỉ định authenticationType (tên scheme) thì IsAuthenticated mới trả về true.
  • ClaimsPrincipal — đối tượng đại diện cho người dùng, có thể chứa nhiều ClaimsIdentity. Đây chính là HttpContext.User.
Cấu trúc Claimscsharp
// Một Claim đơn lẻ
var claim = new Claim("Role", "Admin");

// ClaimsIdentity = tập hợp claims + authentication scheme
var identity = new ClaimsIdentity(
    new[] { claim },
    authenticationType: "CookieScheme"   // bắt buộc để IsAuthenticated = true
);

// ClaimsPrincipal = đại diện người dùng
var principal = new ClaimsPrincipal(identity);

// Kiểm tra
bool isAuth = principal.Identity?.IsAuthenticated;   // true
string? name = principal.Identity?.Name;             // lấy claim ClaimTypes.Name

Tại sao cần Authentication Scheme?

Khi ticket quay về server, middleware Authentication cần biết cách giải mã nó — là Cookie hay JWT. Authentication scheme chính là tên định danh cho cặp "cách tạo / cách giải mã" đó. Scheme được đăng ký trong Program.cs và phải khớp với tên dùng khi gọi SignInAsync / SignOutAsync.

ℹ️ ASP.NET Core cung cấp hằng số CookieAuthenticationDefaults.AuthenticationScheme có giá trị là chuỗi "Cookies". Bạn có thể dùng tên tuỳ chỉnh (như "CookieScheme") miễn là nhất quán.

3 Tạo Authentication Ticket (Sign In)

Bài 3 · Generate Authentication Ticket

Để tạo authentication ticket, cần làm ba việc: (1) đăng ký dịch vụ Authentication trong Program.cs, (2) tạo ClaimsPrincipal với thông tin người dùng, (3) gọi HttpContext.SignInAsync().

Bước 1 — Đăng ký dịch vụ Authentication trong Program.cs

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllersWithViews();

// Đăng ký Authentication với scheme "CookieScheme"
builder.Services.AddAuthentication("CookieScheme")
    .AddCookie("CookieScheme", options =>
    {
        options.Cookie.Name    = "CookieScheme";
        options.ExpireTimeSpan = TimeSpan.FromMinutes(5);
    });

var app = builder.Build();

app.UseStaticFiles();
app.UseRouting();

// UseAuthentication PHẢI đứng trước UseAuthorization
app.UseAuthentication();
app.UseAuthorization();

app.MapDefaultControllerRoute();
app.Run();

⚠️ Ở bài này chưa có UseAuthentication() trong pipeline — middleware chưa được thêm vào. Đây là lý do ticket được tạo nhưng HttpContext.User.Identity.IsAuthenticated vẫn trả về false (sẽ sửa ở bài 4).

Bước 2 — Tạo ViewModel cho form Login

Model/CredentialViewModel.cscsharp
using System.ComponentModel.DataAnnotations;

namespace WebApp.Model
{
    public class CredentialViewModel
    {
        [Required]
        [Display(Name = "Email Address")]
        [EmailAddress]
        public string? EmailAddress { get; set; }

        [Required]
        [DataType(DataType.Password)]
        public string? Password { get; set; }
    }
}

Bước 3 — AccountController: xác minh và SignInAsync

Controllers/AccountController.cscsharp
using Microsoft.AspNetCore.Authentication;
using Microsoft.AspNetCore.Mvc;
using System.Security.Claims;
using WebApp.Model;

namespace WebApp.Controllers
{
    public class AccountController : Controller
    {
        // GET /Account/Login — hiển thị form
        public IActionResult Login()
        {
            return View(new CredentialViewModel());
        }

        // POST /Account/Login — xử lý đăng nhập
        [HttpPost]
        public async Task<IActionResult> Login(CredentialViewModel credential)
        {
            if (!ModelState.IsValid)
                return View(credential);

            // Xác minh thông tin (thực tế nên truy vấn database)
            if (credential.EmailAddress == "admin@company.com"
                && credential.Password == "password")
            {
                // 1. Tạo danh sách Claims
                var claims = new List<Claim>
                {
                    new Claim("Name", credential.EmailAddress),
                    new Claim("Role", "Admin")
                };

                // 2. Gói claims vào ClaimsIdentity — phải truyền tên scheme
                var identity = new ClaimsIdentity(claims, "CookieScheme");

                // 3. Tạo ClaimsPrincipal (đại diện người dùng)
                var userPrincipal = new ClaimsPrincipal(identity);

                // 4. Gọi SignInAsync — tạo cookie chứa authentication ticket
                await HttpContext.SignInAsync("CookieScheme", userPrincipal);

                return RedirectToAction("Index", "Home");
            }

            ModelState.AddModelError("Error", "Không thể đăng nhập.");
            return View(credential);
        }
    }
}

4 Giải mã Authentication Ticket

Bài 4 · Interpret Authentication Ticket

Khi cookie được gửi về server trong request tiếp theo, cần có một middleware để đọc và giải mã nó, rồi nạp thông tin người dùng vào HttpContext.User. Đây là nhiệm vụ của Authentication Middleware.

Thêm UseAuthentication() vào pipeline

Program.cscsharp
var app = builder.Build();

app.UseStaticFiles();
app.UseRouting();

app.UseAuthentication();   // <-- giải mã ticket, nạp vào HttpContext.User
// UseAuthorization sẽ thêm ở bài tiếp theo

app.MapDefaultControllerRoute();
app.Run();

Cách hoạt động bên trong

  1. Request đến — cookie CookieScheme được đọc từ header.
  2. Middleware tra cứu cấu hình scheme "CookieScheme" đã đăng ký.
  3. Cookie handler giải mã (decrypt) giá trị cookie thành ClaimsPrincipal.
  4. HttpContext.User được gán = ClaimsPrincipal đó.
  5. Nếu không có cookie hoặc cookie không hợp lệ, HttpContext.User.Identity.IsAuthenticated = false.

Kiểm tra kết quả trong Controller

Controllers/HomeController.cs (minh hoạ debug)csharp
public IActionResult Index()
{
    // Sau khi UseAuthentication() được thêm vào pipeline:
    var user = HttpContext.User;                            // ClaimsPrincipal
    bool isAuth   = user.Identity?.IsAuthenticated ?? false; // true nếu đã đăng nhập
    string? name  = user.Identity?.Name;                    // giá trị claim ClaimTypes.Name
    string? role  = user.FindFirst("Role")?.Value;          // "Admin" hoặc "User"

    return View();
}

ℹ️ HttpContext.User luôn tồn tại (không null) nhưng khi chưa xác thực nó là một ClaimsPrincipal với AnonymousIdentity — IsAuthenticated = false. Chỉ sau khi middleware Authentication giải mã cookie thành công mới là true.

Thứ tự middleware là bắt buộc

Thứ tựMiddlewareLý do
1UseRouting()Xác định endpoint đích
2UseAuthentication()Giải mã ticket → nạp HttpContext.User
3UseAuthorization()Cần HttpContext.User đã được nạp
4MapControllerRoute()Chuyển đến handler thực sự

5 Authorization dựa trên Authentication Ticket

Bài 5 · Authorization based on Authentication Ticket

Sau khi ticket được giải mã và nạp vào HttpContext.User, bước tiếp theo là phân quyền: quyết định xem người dùng có được phép truy cập resource hay không. ASP.NET Core hỗ trợ nhiều mức phân quyền.

Thêm UseAuthorization() và đăng ký policy

Program.cscsharp
builder.Services.AddAuthentication("CookieScheme")
    .AddCookie("CookieScheme", options =>
    {
        options.LoginPath      = "/Account/Login"; // redirect khi chưa đăng nhập
        options.Cookie.Name    = "CookieScheme";
        options.ExpireTimeSpan = TimeSpan.FromMinutes(5);
    });

// Đăng ký authorization policy
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("AdminOnly", policy =>
    {
        policy.RequireClaim("Role", "Admin");  // ticket phải có claim Role=Admin
    });
});

// ...

app.UseAuthentication();
app.UseAuthorization();   // <-- thêm vào pipeline

Ba mức phân quyền

Mức 1 — Không yêu cầu xác thực (mặc định)

Controller/action không gắn [Authorize] → bất kỳ ai cũng truy cập được, kể cả chưa đăng nhập.

Controllers/HomeController.cscsharp
// Không có [Authorize] → công khai
public class HomeController : Controller
{
    public IActionResult Index() => View();
}

Mức 2 — Chỉ cần đã đăng nhập ([Authorize] đơn giản)

Gắn [Authorize] lên controller/action: chỉ cần ticket tồn tại và hợp lệ, không kiểm tra claim cụ thể nào.

Controllers/DepartmentsController.cscsharp
using Microsoft.AspNetCore.Authorization;
using Microsoft.AspNetCore.Mvc;

[Authorize]   // toàn bộ controller yêu cầu đăng nhập
public class DepartmentsController : Controller
{
    public IActionResult Index() => View();

    [HttpGet]
    public async Task<IActionResult> Details(int id) { /*...*/ return View(); }

    // Action này cần thêm quyền Admin
    [HttpPost]
    [Authorize(Policy = "AdminOnly")]
    public async Task<IActionResult> Create(Department department) { /*...*/ return RedirectToAction(nameof(Index)); }

    [HttpPost]
    [Authorize(Policy = "AdminOnly")]
    public async Task<IActionResult> Edit(Department department) { /*...*/ return RedirectToAction(nameof(Index)); }

    [HttpPost]
    [Authorize(Policy = "AdminOnly")]
    public async Task<IActionResult> Delete(int id) { /*...*/ return RedirectToAction(nameof(Index)); }
}

Mức 3 — Phân quyền theo Policy / Claim

Gắn [Authorize(Policy = "AdminOnly")]: ngoài việc phải đăng nhập, claim Role trong ticket còn phải có giá trị "Admin".

ℹ️ Có thể kết hợp: đặt [Authorize] ở cấp controller (yêu cầu đăng nhập chung) và [Authorize(Policy = "AdminOnly")] tại từng action nhạy cảm. Hai attribute này cộng dồn (AND logic) — cả hai đều phải thoả mãn.

Xử lý khi truy cập bị từ chối

Khi người dùng chưa đăng nhập và truy cập resource có [Authorize]:

  • Middleware tự động redirect về trang Login (đường dẫn cấu hình qua options.LoginPath).

Khi người dùng đã đăng nhập nhưng không đủ quyền (ví dụ không phải Admin):

  • Middleware redirect về trang Access Denied (mặc định /Account/AccessDenied).

    return View();
}

AccountController đầy đủ (với phân biệt Admin / User)

Controllers/AccountController.cscsharp
using Microsoft.AspNetCore.Authentication;
using Microsoft.AspNetCore.Mvc;
using System.Security.Claims;
using WebApp.Model;

namespace WebApp.Controllers
{
    public class AccountController : Controller
    {
        public IActionResult Login() => View(new CredentialViewModel());

        [HttpPost]
        public async Task<IActionResult> Login(CredentialViewModel credential)
        {
            if (!ModelState.IsValid) return View(credential);

            bool isAdmin = credential.EmailAddress == "admin@company.com"
                           && credential.Password == "password";
            bool isUser  = credential.EmailAddress == "user@company.com"
                           && credential.Password == "password";

            if (isAdmin || isUser)
            {
                var claims = new List<Claim>();
                claims.Add(new Claim("Name", credential.EmailAddress!));
                claims.Add(new Claim("Role", isAdmin ? "Admin" : "User"));

                var identity      = new ClaimsIdentity(claims, "CookieScheme");
                var userPrincipal = new ClaimsPrincipal(identity);

                await HttpContext.SignInAsync("CookieScheme", userPrincipal);
                return RedirectToAction("Index", "Home");
            }

            ModelState.AddModelError("Error", "Not able to login.");
            return View(credential);
        }

        public IActionResult AccessDenied() => View();
    }
}

6 Đăng xuất (Sign Out)

Bài 6 · Remove the Authentication Ticket when Signing Out

Khi người dùng đăng xuất, server cần xoá cookie khỏi trình duyệt. Nếu không, cookie vẫn tồn tại và người dùng vẫn được coi là đã đăng nhập cho đến khi hết hạn.

Thêm action LogOut vào AccountController

Controllers/AccountController.cs — thêm LogOutcsharp
[HttpPost]
public async Task<IActionResult> LogOut()
{
    // SignOutAsync xoá cookie khỏi trình duyệt
    await HttpContext.SignOutAsync("CookieScheme");

    // Redirect về trang chủ (không yêu cầu xác thực)
    return RedirectToAction("Index", "Home");
}

⚠️ Action đăng xuất phải dùng [HttpPost] để tránh tấn công CSRF — kẻ tấn công có thể khiến trình duyệt gửi GET request ngầm đến URL đăng xuất, ép người dùng bị logout ngoài ý muốn. POST kết hợp AntiForgery token ngăn chặn điều này.

Partial View hiển thị trạng thái đăng nhập (_LoginStatus.cshtml)

Thay vì hardcode nút Login/Logout ở mọi nơi, tạo một partial view kiểm tra User.Identity.IsAuthenticated và hiển thị phù hợp:

Views/Shared/_LoginStatus.cshtmlrazor
@if (User is not null && User.Identity is not null && User.Identity.IsAuthenticated)
{
    <form method="post" class="form-inline"
          asp-action="LogOut" asp-controller="Account">
        Welcome @User.Identity.Name
        <button type="submit" class="ml-2 btn btn-link">Logout</button>
    </form>
}
else
{
    <a class="btn btn-link"
       asp-controller="Account" asp-action="Login">Login</a>
}

Nhúng _LoginStatus vào Navbar

Views/Shared/_Navbar.cshtml (trích)razor
<nav class="navbar navbar-expand-lg bg-body-tertiary">
    <div class="container-fluid">
        <!-- ... các nav-item ... -->
        <div class="mr-2">
            <partial name="_LoginStatus" />
        </div>
    </div>
</nav>

💡 User trong Razor view là shorthand cho HttpContext.User. Thuộc tính này luôn khả dụng vì Razor view kế thừa RazorPage<T> vốn có sẵn property User.

Luồng hoàn chỉnh sau khi thêm Sign Out

  1. User chưa đăng nhập → navbar hiện nút Login.
  2. User đăng nhập thành công → SignInAsync tạo cookie → navbar hiện Welcome user@company.com + nút Logout.
  3. User nhấn Logout → POST đến /Account/LogOut → SignOutAsync xoá cookie → redirect về trang chủ → navbar lại hiện nút Login.

7 Bật HTTPS

Bài 7 · Enable HTTPS

Nếu ứng dụng chỉ chạy qua HTTP thuần, toàn bộ dữ liệu — bao gồm email và mật khẩu — được gửi dưới dạng plain text. Bất kỳ ai nghe lén mạng (ví dụ dùng công cụ Fiddler) đều có thể đọc được thông tin đó. HTTPS thêm một lớp mã hoá (TLS/SSL) lên trên HTTP, ngăn chặn việc đọc trộm.

HTTPS là gì?

ProtocolMã hoáAn toàn?
HTTPKhôngKhông — dữ liệu plain text
HTTPSTLS (Transport Layer Security)Có — dữ liệu mã hoá end-to-end

Bật HTTPS trong ASP.NET Core

ASP.NET Core cung cấp hai middleware để xử lý HTTPS:

  • UseHsts() — gửi HTTP header Strict-Transport-Security, yêu cầu trình duyệt luôn dùng HTTPS với domain này trong tương lai.
  • UseHttpsRedirection() — tự động redirect mọi request HTTP (80) thành HTTPS (443).
Program.cs — bật HTTPScsharp
var app = builder.Build();

// Chỉ bật HSTS trong môi trường production
if (!app.Environment.IsDevelopment())
{
    app.UseHsts();
}

app.UseHttpsRedirection();   // redirect HTTP → HTTPS

app.UseStaticFiles();
app.UseRouting();

app.UseAuthentication();
app.UseAuthorization();

app.MapDefaultControllerRoute();
app.Run();

Khi phát triển local, .NET đã tự tạo sẵn một development certificate (chứng chỉ tự ký). Chạy lệnh sau để tin tưởng certificate đó trên máy:

    });

// 2. Đăng ký Authorization policies
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("AdminOnly", policy =>
        policy.RequireClaim("Role", "Admin"));
});

var app = builder.Build();

// 3. HTTPS
if (!app.Environment.IsDevelopment())
    app.UseHsts();

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();

// 4. Authentication & Authorization middleware (thứ tự quan trọng)
app.UseAuthentication();
app.UseAuthorization();

app.MapDefaultControllerRoute();
app.Run();
