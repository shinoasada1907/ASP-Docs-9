CHƯƠNG 06

Minimal API — Producing Results

Tìm hiểu cách tạo và trả về kết quả từ Minimal API endpoint: từ trả chuỗi/object đơn giản đến IResult, Results/TypedResults, status code chính xác, chuẩn Problem Details (RFC 7807), và tùy biến response với class riêng.

1 Các kiểu trả về của Minimal API

Bài 1 · Minimal API Return Types

Trong Minimal API, endpoint handler là một lambda (hoặc phương thức) được gọi khi request khớp route. Giá trị mà handler trả về sẽ được framework chuyển thành HTTP response. ASP.NET Core hỗ trợ nhiều kiểu trả về khác nhau, mỗi kiểu có hành vi mặc định riêng.

Trả về trực tiếp: string và object

Cách đơn giản nhất là trả về ngay từ lambda:

  • Trả về string → status 200 OK, Content-Type: text/plain.
  • Trả về bất kỳ object nào (class, record, anonymous type) → framework tự serialize thành JSON, status 200 OK, Content-Type: application/json.
Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Trả về string → 200 OK, text/plain
app.MapGet("/", () => "Hello World!");

// Trả về object → 200 OK, application/json
app.MapGet("/employees", () =>
{
    var employees = EmployeesRepository.GetEmployees();
    return employees; // tự động serialize JSON
});

app.Run();

Vấn đề khi chỉ trả về object

Trả thẳng object hoạt động tốt khi mọi thứ thành công. Nhưng khi cần báo lỗi — ví dụ request không hợp lệ — nếu vẫn trả về object thông thường, status code vẫn là 200 OK dù thực ra là lỗi. Client không thể phân biệt thành công hay thất bại chỉ qua status code.

⚠️ Trả về chuỗi lỗi dạng return "Employee is not valid" với status 200 là sai về ngữ nghĩa HTTP. Client mong đợi 400/404/422 cho các tình huống lỗi, không phải 200.

Giao diện IResult

Để kiểm soát hoàn toàn response (status code, header, body), handler nên trả về IResult. Đây là interface cốt lõi:

IResult — định nghĩacsharp
// Interface nội bộ của ASP.NET Core (đơn giản hóa)
public interface IResult
{
    Task ExecuteAsync(HttpContext httpContext);
}

Khi handler trả về IResult, framework gọi ExecuteAsync để ghi response. Bạn không cần tự implement interface này vì framework cung cấp sẵn hai lớp helper tĩnh: Results và TypedResults.

Helper class Results — các phương thức thường dùng

Phương thứcStatus CodeMô tả
Results.Ok(obj)200Thành công, trả kèm body JSON
Results.Created(uri, obj)201Tạo resource mới thành công
Results.NoContent()204Thành công, không có body
Results.BadRequest(msg)400Request không hợp lệ
Results.NotFound()404Resource không tồn tại
Results.Json(obj)200Serialize object thành JSON (tùy chỉnh options)

Program.cscsharp
using WebApi.Models;

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/employees", () =>
{
    var employees = EmployeesRepository.GetEmployees();
    return TypedResults.Ok(employees); // 200 OK + JSON array
});

app.MapPost("/employees", (Employee employee) =>
{
    if (employee is null || employee.Id <= 0)
    {
        // 400 Bad Request + thông báo lỗi
        return Results.BadRequest("Employee is not provided or is not valid.");
    }

    EmployeesRepository.AddEmployee(employee);
    return Results.Ok("Employee added successfully."); // 200 OK

}).WithParameterValidation();

app.Run();

💡 WithParameterValidation() là extension method từ package MinimalApis.Extensions. Khi được gọi, nó tự động chạy Data Annotation validation trên các tham số của endpoint trước khi gọi handler.

2 Kết hợp Results và TypedResults

Bài 2 · Mix and Match Results and TypedResults

ASP.NET Core cung cấp hai lớp helper song song: Results (non-generic) và TypedResults (generic/strongly-typed). Hiểu được sự khác biệt và cách dùng đúng mỗi loại giúp code rõ ràng và an toàn hơn.

Results vs TypedResults

Tiêu chíResultsTypedResults
Kiểu trả vềIResult (non-generic)Ok<T>, Created<T>... (strongly-typed)
Unit testingKhó kiểm tra kiểu cụ thểDễ dàng assert kiểu chính xác
OpenAPI metadataKhông tự động generateTự động expose kiểu trong Swagger
Khi nên dùngKhi trả nhiều kiểu khác nhau trong cùng branchKhi branch thành công có kiểu rõ ràng

Vấn đề type consistency khi dùng TypedResults

Khi một handler có nhiều nhánh trả về, compiler yêu cầu tất cả các nhánh phải có cùng kiểu trả về. Với TypedResults, TypedResults.Ok<Employee> và TypedResults.Created<Employee> là hai kiểu khác nhau, dẫn đến lỗi biên dịch nếu dùng cả hai trong cùng handler mà không khai báo union type.

Program.cs — lỗi biên dịchcsharp
// SAI: compiler báo lỗi vì hai nhánh có kiểu khác nhau
app.MapPost("/employees", (Employee employee) =>
{
    if (employee is null)
        return TypedResults.BadRequest("invalid"); // BadRequest<string>

    EmployeesRepository.AddEmployee(employee);
    return TypedResults.Created($"/employees/{employee.Id}", employee); // Created<Employee>
    // Compiler: cannot implicitly convert BadRequest<string> to Created<Employee>
});

Giải pháp: Results<T1, T2> — union type

Khi handler có thể trả nhiều kiểu IResult cụ thể, khai báo kiểu trả về là Results<T1, T2, ...>. Cách này giữ được lợi ích strongly-typed của TypedResults đồng thời cho phép nhiều kiểu trả về:

Program.cscsharp
using Microsoft.AspNetCore.Http.HttpResults;
using WebApi.Models;

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/employees", () =>
{
    var employees = EmployeesRepository.GetEmployees();
    return TypedResults.Ok(employees); // Ok<List<Employee>>
});

// Khai báo kiểu trả về tường minh với Results<T1,T2>
app.MapPost("/employees", (Employee employee) =>
    Results<BadRequest<string>, Created<Employee>>
{
    if (employee is null)
        return TypedResults.BadRequest("Employee is not provided or is not valid.");

    EmployeesRepository.AddEmployee(employee);
    return TypedResults.Created($"/employees/{employee.Id}", employee);

}).WithParameterValidation();

app.Run();

Phương thức Created — ngữ nghĩa đúng cho POST

Khi client gửi POST để tạo resource mới, response đúng chuẩn HTTP là 201 Created, kèm header Location trỏ đến URL của resource vừa tạo và body chứa resource đó. TypedResults.Created(uri, value) thực hiện đúng điều này:

Program.cs — POST đúng chuẩncsharp
app.MapPost("/employees", (Employee employee) =>
{
    if (employee is null)
        return Results.BadRequest("Employee is not provided.");

    // Cần đảm bảo employee.Id hợp lệ trước khi thêm
    int maxId = EmployeesRepository.GetEmployees().Max(e => e.Id);
    employee = employee with { Id = maxId + 1 };

    EmployeesRepository.AddEmployee(employee);

    // 201 Created + Location: /employees/{id} + body JSON employee
    return TypedResults.Created($"/employees/{employee.Id}", employee);

}).WithParameterValidation();

🎯Khi nào dùng gì? — Với nhánh thành công (happy path): ưu tiên TypedResults. Với nhánh lỗi hoặc khi handler trả nhiều kiểu: dùng Results<T1,T2> để khai báo union, hoặc dùng Results nếu không cần strongly-typed.

3 Chuẩn Problem Details (RFC 7807)

Bài 3 · The Problem Details Standard

Khi API trả về lỗi, nếu mỗi endpoint tự định nghĩa format JSON riêng thì client phải xử lý nhiều cấu trúc khác nhau — rất phức tạp và dễ gây bug. RFC 7807 — Problem Details for HTTP APIs là tiêu chuẩn quốc tế định nghĩa một format JSON thống nhất cho mọi lỗi API.

Cấu trúc Problem Details JSON

Mỗi response lỗi theo RFC 7807 là một JSON object với các trường chuẩn:

Problem Details response (ví dụ)json
{
  "type": "https://tools.ietf.org/html/rfc7807",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "errors": {
    "Position": ["The Position field is required."],
    "id": ["Employee is not provided or is not valid."]
  }
}
TrườngÝ nghĩa
typeURI tham chiếu đến tài liệu mô tả loại lỗi
titleMô tả ngắn gọn, con người đọc được
statusHTTP status code của lỗi
detailChi tiết cụ thể về lỗi (tuỳ chọn)
instanceURI của request gây ra lỗi (tuỳ chọn)
errorsMap field → danh sách lỗi (cho validation)

Tại sao cần chuẩn này?

Xem xét tình huống thực tế: validation từ Data Annotations (qua WithParameterValidation()) trả về format khác với lỗi bạn tự trả về bằng Results.BadRequest("some string"). Client sẽ không biết cần parse cấu trúc nào.

ℹ️ RFC 7807 (nay cập nhật thành RFC 9457) được định nghĩa tại https://tools.ietf.org/html/rfc7807. Đây là tiêu chuẩn công nghiệp — bất kỳ API nào công khai đều nên tuân theo.

Results.ValidationProblem và Results.Problem

ASP.NET Core cung cấp sẵn hai helper để tạo response theo chuẩn RFC 7807 mà không cần tự build JSON:

Program.cscsharp
// ValidationProblem: lỗi 400 do dữ liệu không hợp lệ
app.MapPost("/employees", (Employee employee) =>
{
    if (employee is null || employee.Id < 0)
    {
        return Results.ValidationProblem(new Dictionary<string, string[]>
        {
            { "id", new[] { "Employee is not provided or is not valid." } }
        });
        // Trả về 400 + JSON Problem Details
    }

    EmployeesRepository.AddEmployee(employee);
    return TypedResults.Created($"/employees/{employee.Id}", employee);

}).WithParameterValidation();

// Problem: lỗi server chung (500 hoặc status khác)
app.MapGet("/danger", () =>
{
    return Results.Problem(
        title: "Unexpected server error",
        detail: "Something went wrong processing your request.",
        statusCode: 500
    );
});

💡 Khi bạn cần báo lỗi, ưu tiên Results.ValidationProblem (lỗi do input) hoặc Results.Problem (lỗi phía server) thay vì Results.BadRequest("chuỗi lỗi"). Đây là cách chuẩn và nhất quán nhất.

4 Chuẩn hóa kết quả API

Bài 4 · Standardize API Results

Chỉ dùng Results.ValidationProblem trong handler chưa đủ để đảm bảo mọi response lỗi đều nhất quán. Còn hai tình huống cần xử lý: 404 không có body (khi route không khớp) và exception chưa được bắt (500 với stack trace HTML).

Tình huống 1: 404 không có body

Khi client gọi URL không tồn tại (ví dụ /employees/99999 khi route tồn tại nhưng không tìm thấy dữ liệu, hoặc URL route không tồn tại hoàn toàn), ASP.NET Core mặc định trả về 404 với body rỗng — không theo chuẩn Problem Details.

Tình huống 2: Exception không được bắt

Khi code throw exception, ở môi trường production server trả về trang lỗi HTML hoặc JSON không theo chuẩn — cũng không nhất quán.

Giải pháp: AddProblemDetails + UseStatusCodePages + UseExceptionHandler

Ba bước để đảm bảo toàn bộ response lỗi đều theo chuẩn RFC 7807:

  1. builder.Services.AddProblemDetails() — đăng ký service Problem Details vào DI container. Khi được đăng ký, các middleware lỗi sẽ biết cách format JSON theo chuẩn.
  2. app.UseStatusCodePages() — middleware này bắt mọi response có status code lỗi (4xx/5xx) nhưng không có body, rồi tự động điền JSON Problem Details vào body.
  3. app.UseExceptionHandler() — middleware bắt exception chưa được xử lý, trả về 500 dạng Problem Details JSON thay vì stack trace.
Program.cscsharp
using WebApi.Models;

var builder = WebApplication.CreateBuilder(args);

// Bước 1: Đăng ký Problem Details service
builder.Services.AddProblemDetails();

var app = builder.Build();

// Bước 2: Exception handler (chỉ khi không phải development)
// Ở development, developer exception page hữu ích hơn
if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler(); // format exception → Problem Details JSON
}

// Bước 3: Điền body cho mọi response lỗi không có body (404, 405...)
app.UseStatusCodePages();

app.MapGet("/employees", () =>
{
    var employees = EmployeesRepository.GetEmployees();
    return TypedResults.Ok(employees);
});

app.MapPost("/employees", (Employee employee) =>
{
    if (employee is null || employee.Id < 0)
    {
        return Results.ValidationProblem(new Dictionary<string, string[]>
        {
            { "id", new[] { "Employee is not provided or is not valid." } }
        });
    }

    EmployeesRepository.AddEmployee(employee);
    return TypedResults.Created($"/employees/{employee.Id}", employee);

}).WithParameterValidation();

app.Run();

Kết quả sau khi chuẩn hóa

Với cấu hình trên, mọi tình huống lỗi đều trả về JSON Problem Details nhất quán:

  • URL không tồn tại → 404 + Problem Details body (nhờ UseStatusCodePages)
  • Exception chưa bắt → 500 + Problem Details body (nhờ UseExceptionHandler)
  • Validation lỗi → 400 + Problem Details body (nhờ Results.ValidationProblem)
  • Request thành công → 200/201 + JSON data bình thường

🎯Thứ tự middleware quan trọng. UseExceptionHandler phải đặt trước các middleware khác để bắt được exception từ toàn bộ pipeline. UseStatusCodePages nên đặt sớm tương tự.

5 Tùy biến kết quả với IResult

Bài 5 · Customize Results by Implementing IResult

Tất cả các helper trong Results và TypedResults đều tối ưu cho việc trả về JSON — phù hợp với API. Nhưng đôi khi bạn cần trả về định dạng khác (HTML, CSV, file nhị phân, XML, v.v.) với logic đặc thù. Lúc này hãy tự implement IResult.

Interface IResult

IResult chỉ có một phương thức cần implement:

IResult (simplified)csharp
public interface IResult
{
    // Framework gọi phương thức này để ghi response
    Task ExecuteAsync(HttpContext httpContext);
}

Bên trong ExecuteAsync, bạn có toàn quyền truy cập vào HttpContext — tức là có thể set bất kỳ header, status code, hay content type nào, rồi ghi trực tiếp vào response body.

Ví dụ: HtmlResult

Tạo class HtmlResult để trả về nội dung HTML thay vì JSON:

Results/HtmlResult.cscsharp
using System.Text;

namespace WebApi.Results
{
    public class HtmlResult : IResult
    {
        private readonly string html;

        public HtmlResult(string html)
        {
            this.html = html;
        }

        public async Task ExecuteAsync(HttpContext httpContext)
        {
            // Set Content-Type để browser biết đây là HTML
            httpContext.Response.ContentType = "text/html";

            // Set Content-Length để tránh chunked transfer
            httpContext.Response.ContentLength = Encoding.UTF8.GetByteCount(html);

            // Ghi HTML vào body
            await httpContext.Response.WriteAsync(html);
        }
    }
}

Sử dụng HtmlResult trong endpoint

Khi dùng IResult tùy biến, nên khai báo kiểu trả về tường minh để framework và Swagger biết kiểu response chính xác:

Program.cscsharp
using WebApi.Models;
using WebApi.Results;

var builder = WebApplication.CreateBuilder(args);
builder.Services.AddProblemDetails();

var app = builder.Build();

if (!app.Environment.IsDevelopment())
    app.UseExceptionHandler();

app.UseStatusCodePages();

// Khai báo kiểu trả về tường minh: HtmlResult
app.MapGet("/", HtmlResult () =>
{
    string html = "<h2>Welcome to our API</h2> Our API is used to learn ASP.NET Core.";
    return new HtmlResult(html);
});

app.MapGet("/employees", () =>
{
    var employees = EmployeesRepository.GetEmployees();
    return TypedResults.Ok(employees);
});

app.MapPost("/employees", (Employee employee) =>
{
    if (employee is null || employee.Id < 0)
    {
        return Results.ValidationProblem(new Dictionary<string, string[]>
        {
            { "id", new[] { "Employee is not provided or is not valid." } }
        });
    }

    EmployeesRepository.AddEmployee(employee);
    return TypedResults.Created($"/employees/{employee.Id}", employee);

}).WithParameterValidation();

app.Run();

ℹ️ Cú pháp HtmlResult () => { ... } (khai báo kiểu trả về tường minh trước lambda) là tính năng C# 10+ cho phép chỉ định kiểu trả về của lambda expression. Điều này giúp framework inference đúng kiểu response khi generate OpenAPI docs.

Khi nào nên implement IResult?

  • Trả về HTML từ endpoint Minimal API (ví dụ trang chào mừng, health page).
  • Trả về CSV/XML với Content-Type tùy chỉnh.
  • Trả về file tải về với Content-Disposition header.
  • Cần thêm custom header vào mọi response của một loại kết quả.
  • Logic formatting phức tạp cần tái sử dụng qua nhiều endpoint.

6 Bài tập: CRUD cho Employees

Bài 6 · Assignment 5 — Implement CRUD for Employees

Đây là bài tập thực hành tổng hợp toàn bộ kiến thức của chương. Bạn cần hoàn thiện một Web API quản lý nhân viên với đầy đủ CRUD operations, sử dụng TypedResults cho happy path và Results.ValidationProblem cho mọi tình huống lỗi.

Yêu cầu

Triển khai các endpoint còn thiếu cho resource /employees:

MethodRouteMô tảSuccess Response
GET/employeesLấy danh sách tất cả nhân viên200 + JSON array
GET/employees/{id:int}Lấy nhân viên theo ID200 + JSON employee
POST/employeesThêm nhân viên mới201 + JSON employee
PUT/employees/{id:int}Cập nhật nhân viên204 No Content
DELETE/employees/{id:int}Xóa nhân viên200 + JSON employee đã xóa

Yêu cầu kỹ thuật

  • Đăng ký AddProblemDetails(), dùng UseStatusCodePages() và UseExceptionHandler().
  • Dùng TypedResults khi thành công (Ok, Created, NoContent).
  • Dùng Results.ValidationProblem khi có lỗi, kèm statusCode tương ứng (400 hoặc 404).
  • Dùng WithParameterValidation() cho endpoint POST.
  • Validate: ID trong route phải trùng ID trong body (cho PUT); employee phải tồn tại (cho GET by id, PUT, DELETE).

Model và Repository có sẵn

Models/Employee.cscsharp
using System.ComponentModel.DataAnnotations;

namespace WebApi.Models
{
    public class Employee
    {
        public int Id { get; set; }
        public string Name { get; set; }

        [Required]
        public string Position { get; set; }

        public double Salary { get; set; }

        public Employee(int id, string name, string position, double salary)
        {
            Id = id; Name = name; Position = position; Salary = salary;
        }
    }
}
Models/EmployeesRepository.cscsharp
namespace WebApi.Models
{
    public static class EmployeesRepository
    {
        private static List<Employee> employees = new()
        {
            new(1, "John Doe", "Engineer", 60000),
            new(2, "Jane Smith", "Manager", 75000),
            new(3, "Sam Brown", "Technician", 50000)
        };

        public static List<Employee> GetEmployees() => employees;

        public static Employee? GetEmployeeById(int id)
            => employees.FirstOrDefault(x => x.Id == id);

        public static void AddEmployee(Employee? employee)
        {
            if (employee is not null) employees.Add(employee);
        }

        public static bool UpdateEmployee(Employee? employee)
        {
            if (employee is null) return false;
            var emp = employees.FirstOrDefault(x => x.Id == employee.Id);
            if (emp is null) return false;
            emp.Name = employee.Name;
            emp.Position = employee.Position;
            emp.Salary = employee.Salary;
            return true;
        }

        public static bool DeleteEmployee(Employee? employee)
        {
            if (employee is null) return false;
            return employees.Remove(employee);
        }
    }
}

💡 Hãy cố tự làm trước khi xem đáp án ở bài 7. Thực hành là cách tốt nhất để nắm vững cú pháp và logic.

7 Đáp án: CRUD cho Employees

Bài 7 · Assignment 5 (Answer) — Implement CRUD for Employees

Dưới đây là giải pháp hoàn chỉnh với đầy đủ CRUD, chuẩn hóa lỗi, và custom HtmlResult. Đây là code thực từ project minh họa của khóa học.

Program.cscsharp
using WebApi.Models;
using WebApi.Results;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddProblemDetails();

var app = builder.Build();

if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler();
}

app.UseStatusCodePages();

// GET / — trả về HTML (dùng custom IResult)
app.MapGet("/", HtmlResult () =>
{
    string html = "<h2>Welcome to our API</h2> Our API is used to learn ASP.NET Core.";
    return new HtmlResult(html);
});

// GET /employees — lấy tất cả nhân viên
app.MapGet("/employees", () =>
{
    var employees = EmployeesRepository.GetEmployees();
    return TypedResults.Ok(employees);
});

// GET /employees/{id} — lấy nhân viên theo ID
app.MapGet("/employees/{id:int}", (int id) =>
{
    var employee = EmployeesRepository.GetEmployeeById(id);

    return employee is not null
        ? TypedResults.Ok(employee)
        : Results.ValidationProblem(
            new Dictionary<string, string[]>
            {
                { "id", new[] { $"Employee with the id {id} doesn't exist." } }
            },
            statusCode: 404);
});

// POST /employees — thêm nhân viên mới
app.MapPost("/employees", (Employee employee) =>
{
    if (employee is null || employee.Id < 0)
    {
        return Results.ValidationProblem(
            new Dictionary<string, string[]>
            {
                { "id", new[] { "Employee is not provided or is not valid." } }
            },
            statusCode: 400);
    }

    EmployeesRepository.AddEmployee(employee);
    return TypedResults.Created($"/employees/{employee.Id}", employee);

}).WithParameterValidation();

// PUT /employees/{id} — cập nhật nhân viên
app.MapPut("/employees/{id:int}", (int id, Employee employee) =>
{
    if (id != employee.Id)
    {
        return Results.ValidationProblem(
            new Dictionary<string, string[]>
            {
                { "id", new[] { "Employee id is not the same as route id." } }
            },
            statusCode: 400);
    }

    return EmployeesRepository.UpdateEmployee(employee)
        ? TypedResults.NoContent()
        : Results.ValidationProblem(
            new Dictionary<string, string[]>
            {
                { "id", new[] { "Employee doesn't exist." } }
            },
            statusCode: 404);
});

// DELETE /employees/{id} — xóa nhân viên
app.MapDelete("/employees/{id:int}", (int id) =>
{
    var employee = EmployeesRepository.GetEmployeeById(id);

    return EmployeesRepository.DeleteEmployee(employee)
        ? TypedResults.Ok(employee)
        : Results.ValidationProblem(
            new Dictionary<string, string[]>
            {
                { "id", new[] { $"Employee with the id {id} doesn't exist." } }
            },
            statusCode: 404);
});

app.Run();

Phân tích từng endpoint

GET /employees/{id:int}

Dùng pattern matching ternary để chọn giữa 200 OK (tìm thấy) và 404 ValidationProblem (không tìm thấy). Lưu ý statusCode: 404 được truyền vào Results.ValidationProblem để override status code mặc định 400.

PUT /employees/{id:int}

Xác minh ID trong route khớp với ID trong body trước khi cập nhật — đây là best practice quan trọng tránh cập nhật nhầm resource. TypedResults.NoContent() trả về 204 — đúng chuẩn REST cho update thành công không có body.

DELETE /employees/{id:int}

Trả về 200 OK kèm object employee đã xóa — hữu ích cho client biết chính xác resource nào được xóa. Một số API chọn 204 No Content cho DELETE; cả hai đều hợp lệ theo REST.

Results/HtmlResult.cscsharp
using System.Text;

namespace WebApi.Results
{
    public class HtmlResult : IResult
    {
        private readonly string html;

        public HtmlResult(string html)
        {
            this.html = html;
        }

        public async Task ExecuteAsync(HttpContext httpContext)
        {
            httpContext.Response.ContentType = "text/html";
            httpContext.Response.ContentLength = Encoding.UTF8.GetByteCount(html);
            await httpContext.Response.WriteAsync(html);
        }
    }
}

Tổng kết chương

Chương này đã đi từ cách trả về đơn giản nhất (string/object) đến bức tranh hoàn chỉnh về producing results trong Minimal API:

  1. Trả về string/object → tự động 200 OK (text hoặc JSON).
  2. IResult interface → nền tảng của mọi kết quả có thể kiểm soát status code.
  3. Results/TypedResults helpers → bộ toolkit phong phú: Ok, Created, BadRequest, NotFound, NoContent, Redirect, File, Json...
  4. Results<T1,T2> → strongly-typed union cho handler có nhiều kiểu trả về.
  5. Problem Details (RFC 7807) → format JSON chuẩn cho mọi lỗi API.
  6. AddProblemDetails + UseStatusCodePages + UseExceptionHandler → đảm bảo 100% response lỗi nhất quán.
  7. Custom IResult → tùy biến response cho nhu cầu đặc thù (HTML, CSV, custom headers...).

🎯Best practice tóm tắt: dùng TypedResults cho happy path, Results.ValidationProblem cho lỗi input, đăng ký AddProblemDetails và UseStatusCodePages để phủ toàn bộ các trường hợp còn lại.
