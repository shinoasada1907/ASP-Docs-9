Securing Web API (Bảo mật Web API)

Bảo mật Web API bằng JWT — tìm hiểu lý thuyết JSON Web Token, sinh và ký JWT trên server, client lưu & gửi token qua header Authorization Bearer, cấu hình Web API giải mã và kiểm tra JWT, và phân quyền endpoint bằng policy dựa trên claims trong JWT.

1 JWT là gì? Tại sao dùng cho Web API?

Bài 1 · JWT Authentication Ticket

Ở chương trước (Securing Web App), authentication ticket được tạo ra dưới dạng cookie và trình duyệt tự động lưu, tự động gửi kèm mỗi request. Cơ chế này hoạt động tốt khi client là trình duyệt web.

Tuy nhiên, khi server phơi ra một Web API, client có thể là bất kỳ loại nào: ứng dụng di động, smart TV, thiết bị IoT, hoặc một ứng dụng web khác. Những client này không có cơ chế cookie giống trình duyệt — vì vậy chúng ta cần một định dạng authentication ticket khác, phổ quát hơn.

Bearer Token và JSON Web Token

Giải pháp là dùng Bearer Token: thay vì lưu ticket trong cookie, client lưu một chuỗi token và đính kèm nó vào header Authorization của mỗi request:

HTTP Headerhttp
Authorization: Bearer <token_string>

JSON Web Token (JWT) là tiêu chuẩn phổ biến nhất để đóng gói thông tin xác thực thành một chuỗi Bearer Token. JWT có cấu trúc gồm 3 phần cách nhau bởi dấu chấm:

PhầnTênNội dung
xxxxxHeaderThuật toán ký (ví dụ: HS256) và loại token (JWT) — Base64Url encoded.
yyyyyPayloadCác claims: thông tin về người dùng (name, role, email...) và metadata (thời gian hết hạn exp, thời gian tạo iat...) — Base64Url encoded.
zzzzzSignatureChữ ký số — bảo đảm token không bị giả mạo. Được tạo bằng cách ký header + payload với secret key (hoặc private key RSA).

🔑Điểm mấu chốt: JWT không được mã hóa — ai cũng có thể decode phần Payload để đọc claims. Tuy nhiên, JWT được ký số — server kiểm tra chữ ký để biết token có bị giả mạo hay không. Vì vậy, không lưu thông tin nhạy cảm (mật khẩu, thẻ tín dụng...) trong JWT.

Luồng hoạt động tổng quát

  1. Client gửi credentials (ClientId + ClientSecret, hoặc username + password) đến endpoint POST /account/login của Web API.
  2. Web API xác thực credentials, tạo danh sách claims (tên, vai trò...) rồi sinh JWT có thời hạn, ký bằng secret key và trả về cho client.
  3. Client lưu JWT (trong session, localStorage, hoặc static repository tuỳ platform).
  4. Mỗi request tiếp theo, client đính kèm JWT vào header Authorization: Bearer <token>.
  5. Web API giải mã và xác thực JWT: kiểm tra chữ ký, kiểm tra thời hạn. Nếu hợp lệ, lấy claims ra để dùng cho authorization.
  6. Authorization kiểm tra claims trong JWT (ví dụ: Role = Admin) để cho phép hoặc từ chối truy cập endpoint.

ℹ️Trong khóa học này, project WebApp (MVC) đóng vai trò client gọi đến project WebApi. WebApp dùng cookie để xác thực người dùng cuối, còn WebApi dùng JWT để xác thực các client ứng dụng gọi vào API.

2 Sinh JWT Authentication Ticket

Bài 2 · Generate JWT Authentication Ticket

Bước đầu tiên ở phía Web API là xây dựng endpoint đăng nhập (POST /account/login) nhận credentials từ client, xác thực, rồi tạo và ký JWT.

Cấu hình secret key trong appsettings.json

Secret key dùng để ký JWT phải đủ dài (ít nhất 32 ký tự cho HMAC-SHA256) và được lưu trong cấu hình, không hard-code trong code:

appsettings.json (WebApi)json
{
  "ClientAuthentication": {
    "SecurityKey": "n4T5@h$W3*9vz!Lk1dX#FpJ8&7qRg2YeS"
  }
}

Model CredentialViewModel

Web API dùng client credentials (ClientId / ClientSecret) thay vì email/password của người dùng cuối:

Models/CredentialViewModel.cscsharp
using System.ComponentModel.DataAnnotations;

namespace WebApp.Model
{
    public class CredentialViewModel
    {
        [Required]
        public string? ClientId { get; set; }

        [Required]
        public string? ClientSecret { get; set; }
    }

AccountEndpoints — sinh JWT

Toàn bộ logic tạo JWT nằm trong AccountEndpoints.cs. Phương thức CreateToken tạo SymmetricSecurityKey từ secret key, tạo SigningCredentials với HMAC-SHA256, rồi dùng JsonWebTokenHandler (từ package Microsoft.IdentityModel.JsonWebTokens) để tạo token string:

Endpoints/AccountEndpoints.cscsharp
using Microsoft.IdentityModel.JsonWebTokens;
using Microsoft.IdentityModel.Tokens;
using System.Security.Claims;
using System.Text;
using WebApp.Model;

namespace WebApi.Endpoints
{
    public static class AccountEndpoints
    {
        public static void MapAccountEndpoints(this WebApplication app)
        {
            app.MapPost("/account/login",
                [Tags("Web API - Accounts")]
                (CredentialViewModel credential, IConfiguration configuration) =>
            {
                // Xác thực client credentials
                if (credential.ClientId == "web-app-001" && credential.ClientSecret == "n4T5@h$W3*9vz!Lk1dX#FpJ8&7qRg2" ||
                    credential.ClientId == "web-app-002" && credential.ClientSecret == "n4T5@h$W3*9vz!Lk1dX#FpJ8&7qRg2")
                {
                    // Tạo danh sách claims
                    var claims = new List<Claim>();
                    claims.Add(new Claim(ClaimTypes.Name, credential.ClientId));

                    if (credential.ClientId == "web-app-001")
                        claims.Add(new Claim("Role", "Admin"));
                    else
                        claims.Add(new Claim("Role", "User"));

                    var expiresAt = DateTime.UtcNow.AddMinutes(20);

                    return TypedResults.Ok(new
                    {
                        access_token = CreateToken(claims, expiresAt, configuration),
                        expires_at = expiresAt,
                    });
                }
                else
                {
                    return Microsoft.AspNetCore.Http.Results.ValidationProblem(
                        new Dictionary<string, string[]>
                        {
                            { "ClientId",     new[] { "Errors in ClientId or ClientSecret." } },
                            { "ClientSecret", new[] { "Errors in ClientId or ClientSecret." } },
                        },
                        statusCode: 401);
                }
            }).WithParameterValidation();
        }

        public static string CreateToken(
            IEnumerable<Claim> claims,
            DateTime expiresAt,
            IConfiguration configuration)
        {
            // 1. Tạo SymmetricSecurityKey từ secret key trong cấu hình
            var securityKey = new SymmetricSecurityKey(
                Encoding.UTF8.GetBytes(
                    configuration["ClientAuthentication:SecurityKey"] ?? string.Empty));

            // 2. Tạo SigningCredentials với thuật toán HMAC-SHA256
            var signingCredentials = new SigningCredentials(
                securityKey,
                SecurityAlgorithms.HmacSha256Signature);

            // 3. Chuyển List<Claim> thành Dictionary cho SecurityTokenDescriptor
            var claimsDictionary = new Dictionary<string, object>();
            foreach (var claim in claims)
                claimsDictionary[claim.Type] = claim.Value;

            // 4. Mô tả token
            var tokenDescriptor = new SecurityTokenDescriptor
            {
                Claims = claimsDictionary,
                Expires = expiresAt,
                SigningCredentials = signingCredentials
            };

            // 5. Sinh token string
            var tokenHandler = new JsonWebTokenHandler();
            return tokenHandler.CreateToken(tokenDescriptor);
        }
    }
}

💡JsonWebTokenHandler vs JwtSecurityTokenHandler: JsonWebTokenHandler là API mới hơn trong .NET, không cần cast về SecurityToken rồi gọi WriteToken() như JwtSecurityTokenHandler — trả về chuỗi token trực tiếp từ CreateToken(). Khuyến nghị dùng JsonWebTokenHandler trong các project .NET 7+.

Response trả về

3 Client lưu và gửi JWT

Bài 3 · Store and Send JWT

Sau khi Web API sinh ra JWT và trả về, client phải tự lưu token và tự đính kèm vào mọi request tiếp theo — khác hoàn toàn với cookie (trình duyệt làm tự động). Trong project này, WebApp (client) xử lý việc đó thông qua một DelegatingHandler.

Model JsonWebToken

Tạo một class để deserialize JSON response từ API đăng nhập:

Model/JsonWebToken.cs (WebApp)csharp
using System.Text.Json.Serialization;

namespace WebApp.Model
{
    public class JsonWebToken
    {
        [JsonPropertyName("access_token")]
        public string? AccessToken { get; set; }

        [JsonPropertyName("expires_at")]
        public DateTime ExpiresAt { get; set; }
    }
}

JWTAuthenticationHandler — DelegatingHandler

DelegatingHandler là một message handler được cắm vào pipeline của HttpClient. Mọi request gửi đi qua HttpClient sẽ đi qua handler này trước. JWTAuthenticationHandler thực hiện:

  1. Kiểm tra session: nếu đã có token, lấy ra và parse.
  2. Nếu token hết hạn hoặc chưa có, gọi POST /account/login để lấy token mới và lưu vào session.
  3. Gắn token vào header Authorization: Bearer ... của request.
  4. Chuyển tiếp request bằng base.SendAsync().
MessageHandlers/JWTAuthenticationHandler.cs (WebApp)csharp
using System.Net.Http.Headers;
using System.Text.Json;
using WebApp.Model;

namespace WebApp.MessageHandlers
{
    public class JWTAuthenticationHandler : DelegatingHandler
    {
        private readonly IHttpContextAccessor httpContextAccessor;
        private readonly IConfiguration configuration;

        public JWTAuthenticationHandler(
            IHttpContextAccessor httpContextAccessor,
            IConfiguration configuration)
        {
            this.httpContextAccessor = httpContextAccessor;
            this.configuration = configuration;
        }

        protected override async Task<HttpResponseMessage> SendAsync(
            HttpRequestMessage request,
            CancellationToken cancellationToken)
        {
            var session = httpContextAccessor.HttpContext?.Session;

            // Lấy token từ session (hoặc đăng nhập mới)
            JsonWebToken token;
            string? strToken = session?.GetString("access_token");

            if (string.IsNullOrWhiteSpace(strToken))
                token = await Login();
            else
                token = JsonSerializer.Deserialize<JsonWebToken>(strToken) ?? new JsonWebToken();

            // Nếu token hết hạn, đăng nhập lại
            if (string.IsNullOrWhiteSpace(token.AccessToken) ||
                token.ExpiresAt <= DateTime.UtcNow)
            {
                token = await Login();
            }

            // Gắn token vào Authorization header
            if (!string.IsNullOrWhiteSpace(token.AccessToken))
            {
                request.Headers.Authorization =
                    new AuthenticationHeaderValue("Bearer", token.AccessToken);
            }

            return await base.SendAsync(request, cancellationToken);
        }

        private async Task<JsonWebToken> Login()
        {
            using var client = new HttpClient();
            var response = await client.PostAsJsonAsync(
                $"{configuration["WebApi:Url"]}account/login",
                new
                {
                    ClientId     = configuration["WebApi:ClientId"],
                    ClientSecret = configuration["WebApi:ClientSecret"]
                });
            response.EnsureSuccessStatusCode();

            string strJwt = await response.Content.ReadAsStringAsync();

            // Lưu token vào session để tái sử dụng
            httpContextAccessor.HttpContext?.Session.SetString("access_token", strJwt);

            return JsonSerializer.Deserialize<JsonWebToken>(strJwt) ?? new JsonWebToken();
        }
    }
}

Đăng ký handler và cấu hình HttpClient

Trong Program.cs của WebApp, đăng ký JWTAuthenticationHandler vào DI và cắm nó vào HttpClient factory:

Program.cs (WebApp) — trích đoạncsharp
builder.Services.AddHttpContextAccessor();
builder.Services.AddTransient<JWTAuthenticationHandler>();

builder.Services.AddHttpClient("ApiEndpoints", client =>
{
    client.BaseAddress = new Uri("http://localhost:5065/");
})
.AddHttpMessageHandler<JWTAuthenticationHandler>();

// Cần Session để lưu token
builder.Services.AddSession(options =>
{
    options.Cookie.HttpOnly = true;
    options.IdleTimeout = TimeSpan.FromMinutes(20);
    options.Cookie.IsEssential = true;
});

// ...
app.UseSession(); // phải có trước khi dùng session

ℹ️Cấu hình WebApi:Url, WebApi:ClientId, WebApi:ClientSecret nên để trong appsettings.json của WebApp (hoặc user-secrets khi phát triển) để tránh hard-code credentials trong code.

4 Lưu JWT trong static repository

Bài 4 · Store JWT in static repository

Ở bài 3, JWT được lưu trong Session (vốn được duy trì bởi cookie phía WebApp). Đây là cách hợp lý khi client là ứng dụng web chạy trên server. Tuy nhiên, với các client khác như ứng dụng di động, smart TV, hoặc IoT device, không có khái niệm session hay cookie.

Giải pháp thay thế: static repository

Một cách tiếp cận đơn giản cho môi trường học tập là lưu JWT trong một static repository — tức là một class với field static tồn tại suốt vòng đời ứng dụng:

Models/JwtRepository.cs (ví dụ minh họa)csharp
namespace WebApp.Models
{
    /// <summary>
    /// Repository tĩnh — lưu JWT trong bộ nhớ ứng dụng.
    /// Dùng cho demo; trong production hãy dùng Distributed Cache hoặc
    /// cơ chế lưu trữ phù hợp với platform (Keychain, SecureStorage...).
    /// </summary>
    public static class JwtRepository
    {
        public static string? AccessToken { get; set; }
        public static DateTime ExpiresAt { get; set; }

        public static bool IsValid()
            => !string.IsNullOrWhiteSpace(AccessToken) &&
               ExpiresAt > DateTime.UtcNow;
    }
}

Phần logic trong handler khi đó thay session.GetString / session.SetString bằng đọc/ghi trực tiếp JwtRepository.AccessToken và JwtRepository.ExpiresAt:

MessageHandlers/JWTAuthenticationHandler.cs — dùng static repocsharp
protected override async Task<HttpResponseMessage> SendAsync(
    HttpRequestMessage request,
    CancellationToken cancellationToken)
{
    // Kiểm tra token trong static repository
    if (!JwtRepository.IsValid())
    {
        var jwt = await Login();
        JwtRepository.AccessToken = jwt.AccessToken;
        JwtRepository.ExpiresAt   = jwt.ExpiresAt;
    }

    // Gắn token vào header
    request.Headers.Authorization =
        new AuthenticationHeaderValue("Bearer", JwtRepository.AccessToken);

    return await base.SendAsync(request, cancellationToken);
}

⚠️Chỉ dùng static repository cho mục đích demo. Trong production: (1) nếu server có nhiều instance, static field không chia sẻ giữa các instance; (2) không thread-safe khi nhiều request đồng thời; (3) dữ liệu mất khi ứng dụng restart. Hãy dùng Distributed Cache (Redis), IMemoryCache có lock, hoặc cơ chế lưu trữ phù hợp với platform.

Khi nào dùng session, khi nào dùng static repo?

Phương pháp lưu JWTPhù hợp vớiGhi chú
Session (cookie-backed)WebApp trên server (.NET MVC, Razor Pages)Mỗi user có session riêng; token tự hết hạn cùng session.

localStorage / SecureStorageSPA (React, Angular), mobile appLưu phía client; phải cẩn thận XSS (SPA) hoặc dùng Keychain/Keystore (mobile).

5 Giải mã và xác thực JWT trên Web API

Bài 5 · Interpret JWT

Client đã gửi JWT trong header Authorization: Bearer <token>, nhưng Web API chưa làm gì với nó. Bước tiếp theo: cấu hình Web API để giải mã và xác thực JWT — chuyển chuỗi token thành ClaimsPrincipal có thể dùng cho authorization.

Cài đặt NuGet package

Terminalbash
dotnet add package Microsoft.AspNetCore.Authentication.JwtBearer

Cấu hình AddAuthentication + AddJwtBearer trong Program.cs

Đây là phần cốt lõi: đăng ký JWT Bearer scheme và cấu hình TokenValidationParameters — bộ tham số kiểm tra khi server nhận được một JWT:

Program.cs (WebApi)csharp
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.IdentityModel.Tokens;
using System.Text;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddAuthentication(options =>
{
    options.DefaultAuthenticateScheme = JwtBearerDefaults.AuthenticationScheme;
    options.DefaultChallengeScheme    = JwtBearerDefaults.AuthenticationScheme;
    options.DefaultSignInScheme       = JwtBearerDefaults.AuthenticationScheme;
})
.AddJwtBearer(options =>
{
    options.TokenValidationParameters = new TokenValidationParameters
    {
        // Không kiểm tra Audience và Issuer (đơn giản cho demo)
        ValidateAudience         = false,
        ValidateIssuer           = false,

        // Kiểm tra thời hạn token
        ValidateLifetime         = true,
        ClockSkew                = TimeSpan.Zero, // Không có dung sai thời gian

        // Kiểm tra chữ ký — bắt buộc
        ValidateIssuerSigningKey = true,
        IssuerSigningKey         = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes(
                builder.Configuration["ClientAuthentication:SecurityKey"] ?? string.Empty))
    };
});

builder.Services.AddAuthorization();

// ... các dịch vụ khác ...

var app = builder.Build();

// ...

// QUAN TRỌNG: UseAuthentication TRƯỚC UseAuthorization
app.UseAuthentication();
app.UseAuthorization();

app.MapAccountEndpoints();
app.MapEmployeeEndpoints();
app.MapDepartmentEndpoints(apiVersionSet);

app.Run();

Giải thích TokenValidationParameters

Tham sốÝ nghĩa
ValidateIssuerSigningKey = trueBắt buộc kiểm tra chữ ký — ngăn token bị giả mạo.
IssuerSigningKeyKhoá dùng để xác minh chữ ký; phải khớp với khoá đã dùng để ký token.
ValidateLifetime = trueTừ chối token đã hết hạn (exp claim).
ClockSkew = TimeSpan.ZeroMặc định .NET cho phép trễ 5 phút. Đặt về 0 để kiểm tra thời hạn chính xác.
ValidateIssuer = falseKhông kiểm tra iss claim. Trong production nên đặt true và cấu hình ValidIssuer.
ValidateAudience = falseKhông kiểm tra aud claim. Trong production nên đặt true và cấu hình ValidAudience.

💡Middleware app.UseAuthentication() đọc header Authorization: Bearer ..., giải mã JWT và nếu hợp lệ, gán HttpContext.User thành ClaimsPrincipal chứa các claims của token. Nhờ đó, các middleware phía sau (kể cả UseAuthorization() và các endpoint) có thể truy cập thông tin người dùng qua HttpContext.User.

6 Authorization dựa trên JWT

Bài 6 · Authorization based on JWT

Sau khi Web API đã xác thực JWT và populate HttpContext.User, bước cuối cùng là dùng claims trong JWT để phân quyền endpoint — tức là kiểm soát endpoint nào được phép truy cập với vai trò nào.

Cơ chế này hoàn toàn giống với authorization dựa trên cookie ở chương 28 — vì về bản chất, cả hai đều sử dụng ClaimsPrincipal làm nguồn thông tin phân quyền. Sự khác biệt chỉ là nguồn gốc của ClaimsPrincipal: từ cookie (chương 28) hay từ JWT (chương 29).

Cấu hình Authorization Policy trong Program.cs

Program.cs (WebApi)csharp
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("AdminOnly", policy =>
    {
        // Yêu cầu claim "Role" có giá trị "Admin"
        policy.RequireClaim("Role", "Admin");
    });
});

Bảo vệ endpoint bằng [Authorize]

Áp dụng attribute [Authorize] lên endpoint handler trong Minimal API. Có thể yêu cầu xác thực đơn thuần, hoặc yêu cầu một policy cụ thể:

Endpoints/EmployeeEndpoints.cscsharp
using Microsoft.AspNetCore.Authorization;

public static class EmployeeEndpoints
{
    public static void MapEmployeeEndpoints(this WebApplication app)
    {
        // Endpoint này yêu cầu JWT hợp lệ VÀ claim Role = Admin
        app.MapGet("/employees/search",
            [Tags("Web API - Employees")]
            [Authorize(Policy = "AdminOnly")]
            (string? filter, int? departmentId, IEmployeesRepository repo) =>
            {
                if (string.IsNullOrWhiteSpace(filter)) filter = null;
                if (departmentId == 0) departmentId = null;

                var employees = repo.GetEmployees(filter, departmentId);
                return TypedResults.Ok(employees);
            });

        // Endpoint này yêu cầu JWT hợp lệ (bất kỳ role nào)
        app.MapGet("/employees",
            [Tags("Web API - Employees")]
            [Authorize]
            (IEmployeesRepository repo) =>
            {
                return TypedResults.Ok(repo.GetEmployees());
            });

        // Endpoint công khai — không cần JWT
        app.MapGet("/", HtmlResult () =>
        {
            return new HtmlResult("<h2>Welcome to our API</h2>");
        });
    }
}

Luồng hoạt động đầy đủ

  1. WebApp gọi GET /employees/search kèm header Authorization: Bearer eyJ....
  2. Middleware UseAuthentication() phân tích header, xác minh chữ ký JWT, kiểm tra exp. Nếu hợp lệ, tạo ClaimsPrincipal và gán vào HttpContext.User.
  3. Middleware UseAuthorization() kiểm tra [Authorize(Policy = "AdminOnly")]: tìm claim "Role" = "Admin" trong HttpContext.User.
  4. Nếu claim tồn tại → cho phép, endpoint handler được gọi và trả về 200 OK.
  5. Nếu không có JWT hoặc JWT không hợp lệ → 401 Unauthorized.
  6. Nếu JWT hợp lệ nhưng thiếu claim cần thiết → 403 Forbidden.

🔑Tổng kết chuỗi bảo mật Web API với JWT:

  • WebApi cấu hình AddAuthentication().AddJwtBearer() để biết cách đọc và kiểm tra JWT.
  • WebApi cấu hình AddAuthorization() với các policy dựa trên claims.
  • WebApi sinh JWT tại endpoint /account/login bằng JsonWebTokenHandler + SymmetricSecurityKey.
  • WebApp (client) dùng DelegatingHandler để tự động lấy, lưu và gửi JWT trong header Authorization: Bearer.
  • Endpoint được bảo vệ bằng [Authorize] hoặc [Authorize(Policy = "...")].

💡Production checklist: (1) Luôn đặt ValidateIssuer = true và ValidateAudience = true; (2) Lưu secret key trong Azure Key Vault hoặc biến môi trường, không trong appsettings.json commit vào Git; (3) Dùng thời hạn token ngắn (5–20 phút) kết hợp với Refresh Token; (4) Luôn dùng HTTPS để bảo vệ token khi truyền qua mạng.
