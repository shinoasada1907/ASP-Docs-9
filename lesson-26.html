CHƯƠNG 26

Logging trong ASP.NET Core

Tìm hiểu hệ thống logging tích hợp sẵn trong ASP.NET Core — cách hoạt động, các thành phần của một log entry, log providers mặc định, cách tự ghi log với ILogger<T>, và cách tích hợp Serilog với sink Console/File.

1 Logging hoạt động như thế nào

Bài 1 · How Logging Works in ASP.NET Core

ASP.NET Core có hệ thống logging tích hợp sẵn (built-in), không cần cài thêm package nào để bắt đầu. Hiểu cơ chế hoạt động của nó giúp bạn sử dụng và mở rộng một cách chính xác.

Ba thành phần cốt lõi

Hệ thống logging trong ASP.NET Core xoay quanh ba thành phần chính:

Thành phầnVai trò
ILogger / ILogger<T> Interface mà code ứng dụng gọi để ghi log. Mỗi logger được tạo ra cho một category (thường là tên class đầy đủ).
ILoggerFactory Factory trung tâm, tạo ra các instance ILogger và phân phối log message đến tất cả các provider đã đăng ký.
ILoggerProvider Mỗi provider biết cách ghi log đến một đích cụ thể: console, file, debug output, event log, database… Có thể có nhiều provider đồng thời.

Luồng hoạt động

  1. Khi ứng dụng khởi động, WebApplication.CreateBuilder(args) tự động đăng ký ILoggerFactory và các provider mặc định vào DI container.
  2. Class cần ghi log khai báo dependency ILogger<TClass> trong constructor. DI framework tạo logger với category = TClass.
  3. Khi gọi logger.LogInformation(...), ILoggerFactory gọi tất cả các provider đã đăng ký — mỗi provider có logger riêng của nó ghi ra đích tương ứng.
  4. Mỗi provider lọc log theo log level được cấu hình — chỉ ghi những message đạt ngưỡng severity tối thiểu.
Program.cscsharp
var builder = WebApplication.CreateBuilder(args);

// ILoggerFactory + providers được đăng ký tự động.
// Có thể tuỳ chỉnh nếu cần:
builder.Logging.ClearProviders();          // xoá toàn bộ provider mặc định
builder.Logging.AddConsole();              // chỉ giữ lại Console

var app = builder.Build();

app.MapGet("/", (ILogger<Program> logger) =>
{
    logger.LogInformation("Root endpoint called");
    return "Hello World!";
});

app.Run();

ℹ️ Vì ILoggerFactory đã có trong DI container, bạn chỉ cần khai báo ILogger<T> là nhận được logger đúng category mà không cần khởi tạo thủ công.

2 Các thành phần của một log

Bài 2 · The different parts of a Log

Mỗi dòng log được in ra (ví dụ ở console) bao gồm ba phần quan trọng: Log Level, Category, và Message.

Log Level (Mức độ nghiêm trọng)

Log level cho biết mức độ quan trọng / nghiêm trọng của sự kiện. ASP.NET Core định nghĩa 6 level theo thứ tự từ thấp đến cao:

LevelEnum valueKhi nào dùng
Trace0Thông tin cực kỳ chi tiết, chỉ bật khi cần trace từng bước (rất ảnh hưởng hiệu năng).
Debug1Thông tin hữu ích khi debug — luồng xử lý, giá trị biến trung gian.
Information2Sự kiện bình thường của ứng dụng: request đến, dịch vụ khởi động…
Warning3Tình huống không mong đợi nhưng ứng dụng vẫn xử lý được.
Error4Lỗi khiến một operation thất bại, cần điều tra.
Critical5Lỗi nghiêm trọng, ứng dụng có thể không tiếp tục hoạt động.

💡 Còn có LogLevel.None (giá trị 6) — dùng trong cấu hình để tắt hoàn toàn logging cho một category.

Category (Nhóm / Nguồn gốc)

Category xác định nơi log được ghi ra. Theo quy ước, category là tên đầy đủ của class (bao gồm namespace) sử dụng ILogger<T>. Ví dụ:

  • MyApp.Controllers.EmployeesController
  • Microsoft.AspNetCore.Hosting.Diagnostics (log của framework)

Category giúp bạn lọc log theo namespace trong appsettings.json — ví dụ chỉ bật Debug cho namespace của riêng bạn mà không làm ngập console bởi log của Microsoft.

Message (Thông điệp)

Chuỗi văn bản mô tả sự kiện. Message có thể chứa placeholder cho structured logging (sẽ đề cập ở bài 4). Ngoài ra, mỗi log còn có Event ID — một số nguyên tùy chọn giúp tra cứu log theo loại sự kiện.

Console output (ví dụ)text
info: MyApp.Controllers.EmployeesController[0]
      Getting employee list for department Sales

Trong đó: info = Log Level, MyApp.Controllers.EmployeesController = Category, [0] = Event ID, dòng tiếp theo = Message.

3 Ghi log với các provider mặc định

Bài 3 · Write Logs with default logger providers

Khi tạo project ASP.NET Core Empty, framework tự động đăng ký bốn provider vào DI container mà không cần cấu hình thêm:

ProviderĐích ghi logGhi chú
ConsoleCửa sổ terminal / consoleLuôn bật mặc định
DebugOutput window của Visual Studio (khi debug)Luôn bật mặc định
EventSourceETW / dotnet-traceDùng cho profiling chuyên sâu
EventLogWindows Event ViewerTắt mặc định, phải bật trong cấu hình; chỉ hoạt động trên Windows

Cấu hình Logging trong appsettings.json

Phần "Logging" trong appsettings.json điều khiển ngưỡng log level theo category prefix. Cấu hình trong appsettings.Development.json sẽ ghi đè lên cấu hình chính khi chạy Development.

appsettings.jsonjson
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning"
    }
  }
}

Giải thích:

  • "Default": "Information" — mọi category không khớp quy tắc nào → ghi từ Information trở lên.
  • "Microsoft.AspNetCore": "Warning" — các category bắt đầu bằng Microsoft.AspNetCore → chỉ ghi từ Warning trở lên, giúp giảm log "nhiễu" của framework.

Bật EventLog provider (Windows)

Provider EventLog cần được bật tường minh bằng cách thêm section Providers vào cấu hình Logging:

appsettings.Development.jsonjson
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning"
    },
    "EventLog": {
      "LogLevel": {
        "Default": "Information",
        "Microsoft.AspNetCore": "Warning"
      }
    }
  }
}

⚠️ appsettings.Development.json ghi đè appsettings.json ở môi trường Development. Nếu chỉ thêm cấu hình EventLog vào file chính nhưng file Development không có phần đó, cài đặt Development sẽ không thấy thay đổi.

Cấu hình provider cụ thể vs. cấu hình chung

Ngoài section LogLevel chung, bạn có thể cấu hình riêng cho từng provider (như EventLog ở trên). Cấu hình provider-specific sẽ ưu tiên hơn cấu hình chung khi provider đó xử lý log.

💡 Khi cần debug một vấn đề cụ thể, hãy hạ LogLevel xuống Debug hoặc Trace chỉ cho category liên quan thay vì bật toàn bộ — tránh bị ngập log không cần thiết.

4 Tự ghi log của riêng bạn

Bài 4 · Write your own logs

Có hai mục đích chính khi tự ghi log trong ứng dụng:

  • Audit (Kiểm tra): ghi lại ai, lúc nào, thực hiện hành động gì — để truy vết hoạt động người dùng.
  • Troubleshooting: ghi thông tin luồng xử lý, giá trị tham số giúp tìm nguyên nhân lỗi.

Inject ILogger<T> qua DI

Cách chuẩn là khai báo ILogger<T> trong constructor. Type parameter T sẽ trở thành category của logger, giúp phân biệt log từ nhiều class.

EmployeesController.cscsharp
using Microsoft.Extensions.Logging;

public class EmployeesController : ControllerBase
{
    private readonly ILogger<EmployeesController> _logger;

    public EmployeesController(ILogger<EmployeesController> logger)
    {
        _logger = logger;
    }

    [HttpGet("{department}")]
    public IActionResult GetByDepartment(string department)
    {
        // Structured logging: {Department} là named placeholder
        _logger.LogInformation(
            "Getting employee list for department {Department}",
            department);

        // ... xử lý nghiệp vụ ...
        return Ok();
    }
}

Trong Minimal API

Program.cscsharp
app.MapGet("/{name}", (string name, ILogger<Program> logger) =>
{
    logger.LogInformation(
        "Request received from user {UserName} at {RequestTime}",
        name,
        DateTime.UtcNow);

    return $"Hello, {name}!";
});

Các phương thức log theo level

ILogger cung cấp extension method tương ứng cho từng level:

Ví dụ các log methodcsharp
_logger.LogTrace("Trace detail: entering method {Method}", nameof(GetByDepartment));
_logger.LogDebug("Debug: querying DB with filter {Filter}", filter);
_logger.LogInformation("Employee list returned, count: {Count}", employees.Count);
_logger.LogWarning("Department {Dept} not found, returning empty list", department);
_logger.LogError(ex, "Error fetching employees for {Department}", department);
_logger.LogCritical("Database connection lost — application cannot continue");

Structured Logging với named placeholder

Sử dụng cú pháp {PlaceholderName} thay vì nối chuỗi giúp log provider lưu trữ dữ liệu dưới dạng key-value, cho phép truy vấn log theo thuộc tính (đặc biệt hữu ích với Seq, Elastic…).

🎯Quy tắc quan trọng: không dùng string interpolation ($"User {name}") trong log message — hãy dùng template với placeholder ("User {Name}", name). String interpolation bỏ mất thông tin cấu trúc và gây cấp phát bộ nhớ thừa.

LoggerMessage Source Generator (.NET 6+)

Khi hiệu năng là ưu tiên, dùng [LoggerMessage] attribute để trình biên dịch sinh code log tối ưu (không boxing, không allocation thừa):

EmployeesController.cs (source generator)csharp
public partial class EmployeesController : ControllerBase
{
    private readonly ILogger<EmployeesController> _logger;

    public EmployeesController(ILogger<EmployeesController> logger)
        => _logger = logger;

    // Source generator sinh method log hiệu năng cao
    [LoggerMessage(
        EventId = 1001,
        Level = LogLevel.Information,
        Message = "Getting employees for department {Department}")]
    private partial void LogGettingEmployees(string department);

    [HttpGet("{department}")]
    public IActionResult GetByDepartment(string department)
    {
        LogGettingEmployees(department);  // gọi method đã được generate
        return Ok();
    }
}

5 Cơ bản về Serilog

Bài 5 · The Basics of Using SeriLog

Serilog là thư viện logging bên thứ ba phổ biến nhất cho .NET, nổi bật với:

  • Structured logging tốt hơn built-in — lưu log dưới dạng object, không chỉ chuỗi.
  • Hệ thống sink phong phú: Console, File, MSSqlServer, Elasticsearch, Azure Blob, Seq…
  • Enricher: tự động gắn thêm thông tin (tên máy, thread, correlation ID…) vào mỗi log entry.

Cài đặt package

Terminalbash
dotnet add package Serilog.AspNetCore

Package Serilog.AspNetCore đã bao gồm sẵn sink Console và tích hợp DI với ASP.NET Core. Để ghi ra file cần thêm Serilog.Sinks.File (thường được kéo vào tự động).

Cấu hình Serilog trong Program.cs (cách cũ — static Log)

Serilog có static class Log cho phép tạo logger toàn cục trước khi DI sẵn sàng (hữu ích để log lỗi ngay giai đoạn startup):

Program.cscsharp
using Serilog;

// Tạo logger tạm để bắt lỗi trong quá trình khởi động
Log.Logger = new LoggerConfiguration()
    .MinimumLevel.Debug()                          // ngưỡng tối thiểu toàn cục
    .WriteTo.Console()                             // sink: console
    .WriteTo.File(
        path: "Logs/log-.txt",                     // tên file (Serilog thêm timestamp)
        rollingInterval: RollingInterval.Day)      // tạo file mới mỗi ngày
    .CreateLogger();

try
{
    var builder = WebApplication.CreateBuilder(args);

    // Tích hợp Serilog vào host — thay thế toàn bộ built-in logging
    builder.Host.UseSerilog();

    var app = builder.Build();

    app.MapGet("/", () =>
    {
        Log.Information("Root endpoint hit");
        return "Hello World!";
    });

    app.Run();
}
catch (Exception ex)
{
    Log.Fatal(ex, "Application failed to start");
}
finally
{
    Log.CloseAndFlush();   // đảm bảo flush buffer trước khi thoát
}

RollingInterval — file log xoay vòng

Tham số rollingInterval quyết định tần suất tạo file log mới:

Giá trịÝ nghĩa
RollingInterval.MinuteTạo file mới mỗi phút (chỉ dùng khi demo/debug)
RollingInterval.HourTạo file mới mỗi giờ
RollingInterval.DayTạo file mới mỗi ngày (khuyến nghị cho production)
RollingInterval.MonthTạo file mới mỗi tháng

Tên file sẽ tự động có timestamp ở cuối, ví dụ: log-20241215.txt.

Log levels của Serilog

Serilog dùng tên level hơi khác so với built-in ASP.NET Core:

SerilogTương đương ASP.NET Core
VerboseTrace
DebugDebug
InformationInformation
WarningWarning
ErrorError
FatalCritical

ℹ️ Serilog hỗ trợ nhiều sink hơn: Serilog.Sinks.MSSqlServer, Serilog.Sinks.Elasticsearch, Serilog.Sinks.AzureTableStorage… Chọn sink phù hợp giúp giảm chi phí lưu trữ và tăng khả năng truy vấn log.

6 Dùng Serilog như built-in logger

Bài 6 · Use SeriLog similar to the built-in logger

Ở bài trước, chúng ta dùng Log.Information() — static class của Serilog. Cách này hoạt động nhưng không nhất quán với kiến trúc DI của ASP.NET Core. Cách tốt hơn là đăng ký Serilog vào DI container và inject ILogger<T> như bình thường — giữ code thống nhất, dễ thay thế sau này.

Đăng ký Serilog qua AddSerilog (khuyến nghị)

Program.cscsharp
using Serilog;

Log.Logger = new LoggerConfiguration()
    .MinimumLevel.Debug()
    .WriteTo.Console()
    .WriteTo.File("Logs/log-.txt", rollingInterval: RollingInterval.Day)
    .Enrich.FromLogContext()                       // enricher: thêm context tự động
    .CreateLogger();

var builder = WebApplication.CreateBuilder(args);

// Cách 1: dùng Host.UseSerilog() — thay thế toàn bộ logging pipeline
builder.Host.UseSerilog();

// Cách 2 (tương đương, .NET 8+): dùng Services.AddSerilog()
// builder.Services.AddSerilog();

var app = builder.Build();

Inject ILogger<T> — giống hệt built-in

Sau khi đăng ký, mọi nơi trong ứng dụng đều inject ILogger<T> y hệt như khi dùng built-in logging:

EmployeesController.cscsharp
public class EmployeesController : ControllerBase
{
    private readonly ILogger<EmployeesController> _logger;

    public EmployeesController(ILogger<EmployeesController> logger)
    {
        _logger = logger;
    }

    [HttpGet("{department}")]
    public IActionResult GetByDepartment(string department)
    {
        // Structured logging — hoàn toàn giống built-in ILogger
        _logger.LogInformation(
            "Getting employees for {Department} requested by {User}",
            department,
            User.Identity?.Name ?? "anonymous");

        var employees = new[] { "Alice", "Bob" }; // demo
        return Ok(employees);
    }
}
Program.cs (Minimal API với DI logger)csharp
// Inject ILogger<Program> qua tham số endpoint handler
app.MapGet("/{name}", (string name, ILogger<Program> logger) =>
{
    logger.LogInformation(
        "Greeting requested for {UserName} at {Time}",
        name,
        DateTime.UtcNow);

    return $"Hello, {name}!";
});

Tại sao nên dùng ILogger<T> thay vì Log.xxx static?

Tiêu chíStatic Log.xxxInject ILogger<T>
Nhất quán với ASP.NET CoreKhôngCó
Dễ unit test (mock)KhóDễ (mock interface)
Thay thế logging libraryPhải sửa nhiều nơiChỉ sửa đăng ký DI
Category tự động theo classPhải chỉ định thủ côngTự động qua T

🎯Best practice: Luôn dùng ILogger<T> qua DI trong controllers, services, pages. Chỉ dùng Log static ở trước khi DI container sẵn sàng (ví dụ: bắt lỗi ngay khi khởi động trong khối try/catch của Program.cs). Nhờ đó, nếu mai này muốn bỏ Serilog để dùng NLog hay OpenTelemetry, bạn chỉ cần thay một dòng đăng ký.

Cấu hình Serilog qua appsettings.json

Thay vì hard-code trong C#, có thể đọc cấu hình từ file — cần thêm package Serilog.Settings.Configuration:

appsettings.jsonjson
{
  "Serilog": {
    "MinimumLevel": {
      "Default": "Information",
      "Override": {
        "Microsoft.AspNetCore": "Warning",
        "System": "Warning"
      }
    },
    "WriteTo": [
      { "Name": "Console" },
      {
        "Name": "File",
        "Args": {
          "path": "Logs/log-.txt",
          "rollingInterval": "Day"
        }
      }
    ],
    "Enrich": [ "FromLogContext", "WithMachineName", "WithThreadId" ]
  }
}
Program.cs (đọc cấu hình từ appsettings)csharp
using Serilog;

var builder = WebApplication.CreateBuilder(args);

builder.Host.UseSerilog((context, config) =>
    config.ReadFrom.Configuration(context.Configuration));

var app = builder.Build();
// ... phần còn lại như bình thường
app.Run();
