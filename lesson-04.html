CHƯƠNG 04

Minimal API — Routing

Định tuyến (routing) trong Minimal API: hiểu endpoint là gì, cách routing middleware hoạt động, route parameters (bắt buộc, mặc định, tùy chọn), route constraints dựng sẵn và tự viết, cùng bài tập CRUD đầy đủ.

1 Hiểu về Endpoint

Bài 1 · Understand Endpoints

Trước khi đi vào routing, cần nắm vững khái niệm endpoint. Khi một HTTP request đến ứng dụng web, ứng dụng thường có nhiều endpoint — mỗi endpoint là một "điểm đến" xử lý một loại request nhất định.

Endpoint là gì?

Một endpoint gồm hai phần không thể tách rời:

  • Route template — mẫu URL để nhận dạng request (ví dụ /employees, /employees/{id}).
  • Request delegate (endpoint handler) — hàm/lambda được gọi khi request khớp với route template đó.

Nói cách khác, routing là cơ chế ánh xạ một request đến đúng endpoint handler, để handler đó xử lý và trả về response.

Đăng ký Endpoint trong Minimal API

Trong Minimal API (.NET 6+), endpoint được đăng ký trực tiếp thông qua các phương thức Map* trên đối tượng WebApplication:

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Đăng ký endpoint: GET /  →  trả về chuỗi "Welcome!"
app.MapGet("/", () => "Welcome!");

// Đăng ký endpoint: GET /employees  →  trả về danh sách nhân viên
app.MapGet("/employees", () => EmployeesRepository.GetEmployees());

// Đăng ký endpoint: POST /employees
app.MapPost("/employees", (Employee emp) => {
    EmployeesRepository.AddEmployee(emp);
    return Results.Created($"/employees/{emp.Id}", emp);
});

app.Run();

ℹ️ Các phương thức MapGet, MapPost, MapPut, MapDelete, MapPatch tương ứng với từng HTTP method. Có thể dùng MapMethods khi muốn hỗ trợ nhiều method cùng lúc.

Cách hoạt động nội bộ

Mỗi lần gọi Map*, ASP.NET Core tạo một đối tượng RouteEndpoint chứa route pattern và delegate. Toàn bộ các endpoint được lưu trong EndpointDataSource — một "kho" được routing middleware tra cứu khi có request đến.

MapPutPUT/employees/{id}

MapPatchPATCH/employees/{id}

2 Routing Middleware

Bài 2 · Use the routing middleware

Định nghĩa endpoint chỉ là bước đầu — ứng dụng còn cần routing middleware để thực sự tra cứu và điều phối request đến đúng handler.

Hai middleware routing cốt lõi

Routing trong ASP.NET Core được chia làm hai middleware riêng biệt, thêm theo đúng thứ tự:

  1. UseRouting() — phân tích URL của request, so khớp với danh sách endpoint đã đăng ký, rồi gắn kết quả vào HttpContext.GetEndpoint(). Middleware này chỉ chọn endpoint, chưa thực thi.
  2. UseEndpoints() (hoặc đặt Map* sau pipeline) — thực sự gọi endpoint handler đã được chọn.
Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// 1. Middleware routing: phân tích URL và chọn endpoint
app.UseRouting();

// 2. Middleware khác có thể đặt ở đây (auth, cors, v.v.)
// Chúng có thể đọc endpoint đã chọn qua context.GetEndpoint()

// 3. Thực thi endpoint handler
app.UseEndpoints(endpoints =>
{
    endpoints.MapGet("/", async (HttpContext context) =>
    {
        await context.Response.WriteAsync("Welcome to the home page.");
    });

    endpoints.MapGet("/employees", async (HttpContext context) =>
    {
        var employees = EmployeesRepository.GetEmployees();
        context.Response.ContentType = "application/json";
        await context.Response.WriteAsJsonAsync(employees);
    });
});

app.Run();

💡 Trong .NET 6+ với Minimal API, WebApplication tự động thêm UseRouting() và UseEndpoints() khi bạn gọi app.MapGet(...) trực tiếp — bạn không cần gọi tường minh trong hầu hết trường hợp. Tuy nhiên, khi cần đặt middleware phụ thuộc vào endpoint (như authorization) giữa hai bước routing, bạn phải gọi tường minh theo đúng thứ tự.

Tại sao thứ tự quan trọng?

Vì middleware pipeline chạy tuần tự, middleware nằm giữa UseRouting() và UseEndpoints() có thể truy cập thông tin endpoint đã chọn. Đây là nền tảng để authorization middleware hoạt động: nó kiểm tra policy của endpoint được chọn trước khi cho phép handler chạy.

Program.cs — thứ tự chuẩn với authcsharp
app.UseRouting();       // 1. Chọn endpoint
app.UseAuthentication(); // 2. Xác thực danh tính
app.UseAuthorization();  // 3. Kiểm tra quyền truy cập endpoint
app.MapGet("/secret", [Authorize] () => "Bí mật!");

3 Middleware 404 Not Found

Bài 3 · 404 Not Found Middleware component

Điều gì xảy ra khi routing middleware không tìm thấy endpoint nào khớp với request? Câu trả lời là: một middleware mặc định sẽ tự động trả về HTTP 404 Not Found.

Cơ chế hoạt động

ASP.NET Core cài sẵn một middleware cuối pipeline xử lý trường hợp không có endpoint nào khớp. Middleware này:

  • Đặt status code = 404.
  • Trả về body rỗng hoặc trang lỗi mặc định.
  • Không ném exception — đây là hành vi expected của framework.

ℹ️ Bạn không thấy middleware 404 trong code vì nó được thêm ngầm định bởi framework. Tuy nhiên nếu muốn tùy chỉnh trang 404 (ví dụ trả JSON thay vì trang HTML trắng), bạn có thể dùng app.UseStatusCodePages() hoặc viết middleware tùy chỉnh.

Tùy chỉnh trang 404

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Tùy chỉnh phản hồi cho các status code lỗi
app.UseStatusCodePages(async statusCodeContext =>
{
    var response = statusCodeContext.HttpContext.Response;
    if (response.StatusCode == 404)
    {
        response.ContentType = "application/json";
        await response.WriteAsJsonAsync(new
        {
            error = "Không tìm thấy tài nguyên yêu cầu.",
            statusCode = 404
        });
    }
});

app.MapGet("/employees", () => EmployeesRepository.GetEmployees());

app.Run();

⚠️ UseStatusCodePages() chỉ can thiệp khi response chưa có body. Nếu handler đã ghi nội dung vào response trước khi set status code 404, middleware này sẽ không được kích hoạt.

4 Route Parameter bắt buộc

Bài 4 · Required Route Parameters

Thay vì dùng query string (/employees?id=1), routing cho phép nhúng giá trị trực tiếp vào URL (/employees/1). Phần giá trị động đó gọi là route parameter.

Cú pháp khai báo

Route parameter được đặt trong cặp ngoặc nhọn {tên} trong route template:

Program.cscsharp
app.UseRouting();

app.UseEndpoints(endpoints =>
{
    // {id} là route parameter bắt buộc
    // URL khớp: /employees/1, /employees/42, ...
    endpoints.MapGet("/employees/{id}", async (HttpContext context) =>
    {
        // Đọc giá trị route parameter
        var id = context.Request.RouteValues["id"];
        var employeeId = int.Parse(id!.ToString()!);

        var employee = EmployeesRepository.GetEmployeeById(employeeId);

        if (employee is not null)
        {
            await context.Response.WriteAsJsonAsync(employee);
        }
        else
        {
            context.Response.StatusCode = 404;
            await context.Response.WriteAsync("Không tìm thấy nhân viên.");
        }
    });
});

Đọc parameter trong Minimal API hiện đại

Kể từ .NET 6, Minimal API hỗ trợ parameter binding trực tiếp từ lambda — gọn hơn nhiều so với đọc qua RouteValues:

Program.cs — cách hiện đạicsharp
// ASP.NET Core tự động bind {id} vào tham số int id
app.MapGet("/employees/{id}", (int id) =>
{
    var employee = EmployeesRepository.GetEmployeeById(id);
    return employee is not null
        ? Results.Ok(employee)
        : Results.NotFound("Không tìm thấy nhân viên.");
});

🔑 Route parameter là bắt buộc theo mặc định — nếu URL thiếu segment tương ứng, route sẽ không khớp và routing middleware sẽ tìm endpoint khác (hoặc trả 404). Đây là điểm khác biệt quan trọng so với query string (có thể bỏ qua).

5 Route Parameter có giá trị mặc định

Bài 5 · Route parameters with default values

Đôi khi bạn muốn route parameter có một giá trị ngầm định khi người dùng không cung cấp segment đó trong URL. Đây gọi là default value cho route parameter.

Cú pháp

Thêm =giá_trị ngay sau tên parameter trong template:

Program.cscsharp
// {size=medium} — nếu không có segment, size = "medium"
// URL khớp: /products/small, /products/large, /products  (→ size = "medium")
app.MapGet("/products/{size=medium}", (string size) =>
{
    return Results.Ok($"Hiển thị sản phẩm kích cỡ: {size}");
});

// Ví dụ thực tế: phân trang với trang mặc định = 1
// URL: /employees/page/2  hoặc  /employees/page  (→ page = 1)
app.MapGet("/employees/page/{page=1}", (int page) =>
{
    int pageSize = 10;
    var employees = EmployeesRepository.GetEmployees()
        .Skip((page - 1) * pageSize)
        .Take(pageSize);
    return Results.Ok(employees);
});

ℹ️ Khi dùng default value, segment đó trở nên không bắt buộc — URL thiếu segment vẫn khớp và parameter nhận giá trị mặc định. Đây là khác biệt quan trọng so với parameter bắt buộc.

Nhiều parameter với default value

Program.cscsharp
// /report/2024/6  → year=2024, month=6
// /report/2024    → year=2024, month=1
// /report         → year=2024 (năm hiện tại), month=1
app.MapGet("/report/{year=2024}/{month=1}", (int year, int month) =>
{
    return Results.Ok($"Báo cáo tháng {month}/{year}");
});

⚠️ Các parameter có default value phải đặt sau parameter bắt buộc trong template. Đặt ngược lại sẽ gây lỗi khi khởi động ứng dụng.

6 Route Parameter tùy chọn

Bài 6 · Optional Route Parameters

Optional parameter tương tự default-value parameter ở chỗ segment có thể vắng mặt trong URL — nhưng khác ở điểm quan trọng: khi vắng mặt, parameter nhận giá trị null thay vì một giá trị mặc định cụ thể.

Cú pháp

Thêm dấu hỏi ? sau tên parameter:

Program.cscsharp
// {name?} — segment tùy chọn, có thể thiếu
// URL khớp: /greet/Alice  (name = "Alice")
// URL khớp: /greet        (name = null)
app.MapGet("/greet/{name?}", (string? name) =>
{
    if (name is null)
        return Results.Ok("Xin chào, khách!");
    return Results.Ok($"Xin chào, {name}!");
});

So sánh ba loại parameter

LoạiCú phápKhi thiếu segmentKiểu C#
Bắt buộc{id}Route không khớpint id
Có default{size=medium}Nhận giá trị mặc địnhstring size
Tùy chọn{name?}Nhận nullstring? name
Program.cs — ví dụ tìm kiếm nhân viêncsharp
// Tìm nhân viên theo tên (tùy chọn) và vị trí (tùy chọn)
// /employees/search            → trả toàn bộ
// /employees/search/John       → lọc theo tên
// /employees/search/John/Manager → lọc cả tên lẫn chức vụ
app.MapGet("/employees/search/{name?}/{position?}",
    (string? name, string? position) =>
    {
        var result = EmployeesRepository.GetEmployees().AsEnumerable();
        if (name is not null)
            result = result.Where(e => e.Name.Contains(name, StringComparison.OrdinalIgnoreCase));
        if (position is not null)
            result = result.Where(e => e.Position == position);
        return Results.Ok(result.ToList());
    });

💡 Optional parameter chỉ có thể đặt ở cuối route template. Không thể có parameter bắt buộc sau optional parameter vì sẽ tạo ra sự mơ hồ trong quá trình khớp route.

7 Route Constraints

Bài 7 · Parameter Constraints

Route constraint (ràng buộc route) cho phép bạn chỉ định điều kiện mà giá trị của route parameter phải thỏa mãn để route đó được chọn. Đây là cơ chế giúp phân biệt các route có cùng template nhưng kiểu dữ liệu khác nhau, đồng thời tự động từ chối các giá trị không hợp lệ.

Tại sao cần Route Constraints?

Giả sử bạn có hai endpoint:

  • /employees/{id} — lấy nhân viên theo ID (số nguyên)
  • /employees/{name} — lấy nhân viên theo tên (chuỗi)

Không có constraint, routing sẽ không biết chọn endpoint nào cho URL /employees/42 hay /employees/John — xảy ra lỗi ambiguous match. Route constraint giải quyết vấn đề này:

Program.cscsharp
app.UseRouting();

app.UseEndpoints(endpoints =>
{
    // Chỉ khớp khi {id} là số nguyên (int)
    // /employees/42   → endpoint này
    endpoints.MapGet("/employees/{id:int}", async (HttpContext context) =>
    {
        var id = int.Parse(context.Request.RouteValues["id"]!.ToString()!);
        var employee = EmployeesRepository.GetEmployeeById(id);
        if (employee is not null)
            await context.Response.WriteAsJsonAsync(employee);
        else
        {
            context.Response.StatusCode = 404;
            await context.Response.WriteAsync("Không tìm thấy.");
        }
    });

    // Chỉ khớp khi {name} là chuỗi (mặc định) — tất cả giá trị không phải int
    // /employees/John → endpoint này
    endpoints.MapGet("/employees/{name:alpha}", async (HttpContext context) =>
    {
        var name = context.Request.RouteValues["name"]!.ToString()!;
        var employees = EmployeesRepository.GetEmployees()
            .Where(e => e.Name.Contains(name, StringComparison.OrdinalIgnoreCase))
            .ToList();
        await context.Response.WriteAsJsonAsync(employees);
    });
});

Kết hợp nhiều constraint

Program.cscsharp
// {id} phải là int VÀ tối thiểu 1
app.MapGet("/employees/{id:int:min(1)}", (int id) =>
{
    var emp = EmployeesRepository.GetEmployeeById(id);
    return emp is not null ? Results.Ok(emp) : Results.NotFound();
});

⚠️ Route constraint chỉ dùng để chọn endpoint (routing), không phải để validate dữ liệu nghiệp vụ. Ví dụ :min(1) ngăn URL /employees/0 khớp route, nhưng không thay thế được validation logic trong handler (như kiểm tra employee tồn tại không).

8 Custom Route Constraints

Bài 8 · Custom Parameter Constraints

Khi các constraint dựng sẵn không đủ đáp ứng yêu cầu, bạn có thể tự viết custom route constraint bằng cách implement interface IRouteConstraint.

Tình huống thực tế

Giả sử ứng dụng quản lý nhân viên chỉ hỗ trợ hai chức vụ: engineer và manager. Bạn muốn route /employees/positions/{position} chỉ khớp khi {position} thuộc hai giá trị đó.

Bước 1 — Implement IRouteConstraint

Constraints/PositionConstraint.cscsharp
using Microsoft.AspNetCore.Routing;

namespace WebApp.Constraints;

public class PositionConstraint : IRouteConstraint
{
    // Danh sách chức vụ hợp lệ
    private static readonly string[] ValidPositions = { "engineer", "manager" };

    public bool Match(
        HttpContext? httpContext,
        IRouter? route,
        string routeKey,
        RouteValueDictionary values,
        RouteDirection routeDirection)
    {
        // Lấy giá trị của parameter từ route values
        if (!values.TryGetValue(routeKey, out var routeValue))
            return false;

        var position = routeValue?.ToString();

        // Khớp khi position nằm trong danh sách hợp lệ (không phân biệt hoa thường)
        return ValidPositions.Contains(
            position, StringComparer.OrdinalIgnoreCase);
    }
}

Bước 2 — Đăng ký constraint vào DI

Program.cscsharp
using WebApp.Constraints;

var builder = WebApplication.CreateBuilder(args);

// Đăng ký custom constraint với tên "position"
builder.Services.Configure<RouteOptions>(options =>
{
    options.ConstraintMap.Add("position", typeof(PositionConstraint));
});

var app = builder.Build();

app.UseRouting();

app.UseEndpoints(endpoints =>
{
    // Dùng constraint tùy chỉnh: {pos:position}
    // /employees/positions/engineer → khớp
    // /employees/positions/manager  → khớp
    // /employees/positions/intern   → KHÔNG khớp (404)
    endpoints.MapGet("/employees/positions/{pos:position}",
        async (HttpContext context) =>
        {
            var position = context.Request.RouteValues["pos"]!.ToString()!;
            var employees = EmployeesRepository.GetEmployees()
                .Where(e => e.Position.Equals(position, StringComparison.OrdinalIgnoreCase))
                .ToList();
            await context.Response.WriteAsJsonAsync(employees);
        });
});

app.Run();

💡 Tên đăng ký trong ConstraintMap ("position") chính là tên dùng trong route template (:position). Tên này phải duy nhất và không trùng với constraint dựng sẵn.

Khi nào nên dùng Custom Constraint?

  • Khi giá trị route cần khớp một tập hợp giá trị định sẵn (enum-like).
  • Khi cần logic kiểm tra phức tạp hơn regex (ví dụ: tra cứu database, kiểm tra định dạng mã tùy chỉnh).
  • Khi muốn tái sử dụng cùng logic constraint ở nhiều route khác nhau.

10 Đáp án bài tập CRUD

Bài 10 · Assignment 3 (Answer) — Implement CRUD operations with routing

Dưới đây là đáp án đầy đủ, trích từ source code thật của khóa học.

Model: Employee

Models/Employee.cscsharp
namespace WebApp.Models;

public class Employee
{

    public Employee(int id, string name, string position, double salary)

Repository in-memory

Models/EmployeesRepository.cscsharp
namespace WebApp.Models;

public static class EmployeesRepository
{
    private static List<Employee> employees = new()
    {
        new Employee(1, "John Doe",   "Engineer",   60000),
        new Employee(2, "Jane Smith", "Manager",    75000),
        new Employee(3, "Sam Brown",  "Technician", 50000)
    };

    public static List<Employee> GetEmployees() => employees;

    public static Employee? GetEmployeeById(int id) =>
        employees.FirstOrDefault(x => x.Id == id);

    public static void AddEmployee(Employee? employee)
    {
        if (employee is not null) employees.Add(employee);
    }

    public static bool UpdateEmployee(Employee? employee)
    {
        if (employee is null) return false;
        var emp = employees.FirstOrDefault(x => x.Id == employee.Id);
        if (emp is null) return false;
        emp.Name = employee.Name;
        emp.Position = employee.Position;
        emp.Salary = employee.Salary;
        return true;
    }

    public static bool DeleteEmployee(int id)
    {
        var employee = employees.FirstOrDefault(x => x.Id == id);
        if (employee is null) return false;
        employees.Remove(employee);
        return true;
    }
}

Program.cs — đầy đủ CRUD với routing

Program.cscsharp
using System.Text.Json;
using WebApp.Models;

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.UseRouting();

app.UseEndpoints(endpoints =>
{
    // GET /  →  trang chào
    endpoints.MapGet("/", async (HttpContext context) =>
    {
        await context.Response.WriteAsync("Welcome to the home page.");
    });

    // GET /employees  →  danh sách tất cả nhân viên
    endpoints.MapGet("/employees", async (HttpContext context) =>
    {
        var employees = EmployeesRepository.GetEmployees();
        context.Response.ContentType = "text/html";
        await context.Response.WriteAsync("<h2>Employees</h2><ul>");
        foreach (var employee in employees)
        {
            await context.Response.WriteAsync(
                $"<li><b>{employee.Name}</b>: {employee.Position}</li>");
        }
        await context.Response.WriteAsync("</ul>");
    });

    // GET /employees/{id:int}  →  thông tin một nhân viên
    endpoints.MapGet("/employees/{id:int}", async (HttpContext context) =>
    {
        var id = int.Parse(context.Request.RouteValues["id"]!.ToString()!);
        var employee = EmployeesRepository.GetEmployeeById(id);

        context.Response.ContentType = "text/html";
        if (employee is not null)
        {
            await context.Response.WriteAsync("<h2>Employee</h2>");
            await context.Response.WriteAsync($"Name: {employee.Name}<br/>");
            await context.Response.WriteAsync($"Position: {employee.Position}<br/>");
            await context.Response.WriteAsync($"Salary: {employee.Salary}<br/>");
        }
        else
        {
            context.Response.StatusCode = 404;
            await context.Response.WriteAsync("Employee not found.");
        }
    });

    // POST /employees  →  thêm nhân viên mới
    endpoints.MapPost("/employees", async (HttpContext context) =>
    {
        using var reader = new StreamReader(context.Request.Body);
        var body = await reader.ReadToEndAsync();
        try
        {
            var employee = JsonSerializer.Deserialize<Employee>(body);
            if (employee is null || employee.Id <= 0)
            {
                context.Response.StatusCode = 400;
                return;
            }
            EmployeesRepository.AddEmployee(employee);
            context.Response.StatusCode = 201;
            await context.Response.WriteAsync("Employee added successfully.");
        }
        catch (Exception ex)
        {
            context.Response.StatusCode = 400;
            await context.Response.WriteAsync(ex.Message);
        }
    });

    // PUT /employees  →  cập nhật nhân viên (JSON body chứa Id)
    endpoints.MapPut("/employees", async (HttpContext context) =>
    {
        using var reader = new StreamReader(context.Request.Body);
        var body = await reader.ReadToEndAsync();
        var employee = JsonSerializer.Deserialize<Employee>(body);

        if (EmployeesRepository.UpdateEmployee(employee))
        {
            context.Response.StatusCode = 204;
        }
        else
        {
            context.Response.StatusCode = 404;
            await context.Response.WriteAsync("Employee not found.");
        }
    });

    // DELETE /employees/{id:int}  →  xóa nhân viên (cần header Authorization)
    endpoints.MapDelete("/employees/{id}", async (HttpContext context) =>
    {
        var id = int.Parse(context.Request.RouteValues["id"]!.ToString()!);

        if (context.Request.Headers["Authorization"] == "frank")
        {
            if (EmployeesRepository.DeleteEmployee(id))
            {
                await context.Response.WriteAsync("Employee deleted successfully.");
            }
            else
            {
                context.Response.StatusCode = 404;
                await context.Response.WriteAsync("Employee not found.");
            }
        }
        else
        {
            context.Response.StatusCode = 401;
            await context.Response.WriteAsync("Unauthorized.");
        }
    });
});

app.Run();

# Lấy tất cả nhân viên

curl -X PUT http://localhost:5000/employees \
  -H "Content-Type: application/json" \

curl -X DELETE http://localhost:5000/employees/3 \
  -H "Authorization: frank"

🎯 Điểm mấu chốt của bài tập này: route parameter {id:int} giúp URL RESTful hơn (/employees/1 thay vì /employees?id=1), đồng thời constraint :int tự động từ chối các URL có ID không phải số nguyên — không cần validate thủ công trong handler.
