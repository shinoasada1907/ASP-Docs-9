CHƯƠNG 03

Middleware Pipeline

Middleware là trái tim của ASP.NET Core. Chương này giải thích mô hình pipeline, cách sử dụng Use / Run / Map / MapWhen / UseWhen để xây dựng và rẽ nhánh pipeline, cách đóng gói middleware thành class tái sử dụng, và những điểm cần lưu ý khi viết response.

1 Lý thuyết Middleware Pipeline

Bài 1 · Middleware Pipeline Theory

Khi một HTTP request đến ứng dụng ASP.NET Core, Kestrel (web server tích hợp) nhận request, chuyển đổi thành đối tượng HttpContext rồi chuyển qua middleware pipeline. Pipeline là một chuỗi các middleware component được thực thi theo thứ tự, sau đó response đi ngược trở lại qua chính chuỗi đó.

Tại sao cần middleware pipeline?

Nếu không có pipeline, toàn bộ logic của ứng dụng (routing, authentication, authorization, logging, model validation, xử lý ngoại lệ, sinh response…) phải đặt vào một component duy nhất — rất khó bảo trì và mở rộng. Middleware pipeline cho phép tách các mối quan tâm (separation of concerns): mỗi middleware chỉ làm một việc, có thể bật/tắt, sắp xếp lại theo ý muốn.

Cơ chế hoạt động

Pipeline hoạt động giống như một chuỗi lồng nhau (Russian doll): middleware số 1 bao quanh số 2, số 2 bao quanh số 3, và cứ thế tiếp tục.

  1. Request đến → middleware #1 xử lý phần trước (before).
  2. Middleware #1 gọi next(context) → chuyển qua middleware #2.
  3. Middleware #2 xử lý phần trước, gọi next(context) → chuyển qua middleware #3…
  4. Middleware cuối (terminal) sinh response, không gọi next.
  5. Luồng quay ngược: middleware #3 xử lý phần sau → trả về #2 → trả về #1 → response đến Kestrel → trả về browser.

🔑Điểm mấu chốt: Middleware được đăng ký theo thứ tự trong Program.cs — thứ tự đó chính là thứ tự thực thi. Middleware đứng đầu luôn được gọi đầu tiên, và cũng là nơi cuối cùng có cơ hội xử lý response trước khi trả về client.

Request Delegate

Mỗi middleware thực chất là một request delegate — một hàm nhận HttpContext và trả về Task. Trong ASP.NET Core, kiểu này được định nghĩa là:

Định nghĩacsharp
public delegate Task RequestDelegate(HttpContext context);

Khi một middleware muốn chuyển request cho middleware kế tiếp, nó gọi tham số next (cũng là một RequestDelegate) và truyền vào context.

2 app.Use — tạo middleware thông thường

Bài 2 · Use app.Use to create middleware

app.Use() là phương thức chính để thêm middleware vào pipeline. Middleware tạo bởi app.Use có thể:

  • Thực thi logic trước khi gọi next (xử lý request).
  • Gọi await next(context) để chuyển cho middleware tiếp theo.
  • Thực thi logic sau khi next hoàn thành (xử lý response trên đường về).
  • Short-circuit: không gọi next nếu muốn dừng pipeline sớm.
Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Middleware #1 — thông thường, gọi next
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #1: Before calling next\r\n");

    await next(context); // chuyển sang #2

    await context.Response.WriteAsync("Middleware #1: After calling next\r\n");
});

// Middleware #2 — short-circuit: không gọi next
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #2: Before calling next\r\n");

    // await next(context);  <-- comment out = dừng pipeline tại đây

    await context.Response.WriteAsync("Middleware #2: After calling next\r\n");
});

// Middleware #3 — sẽ không được thực thi nếu #2 short-circuit
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #3: Before calling next\r\n");

    await next(context);

    await context.Response.WriteAsync("Middleware #3: After calling next\r\n");
});

app.Run();

Output khi Middleware #2 short-circuit

Response (text/plain)text
Middleware #1: Before calling next
Middleware #2: Before calling next
Middleware #2: After calling next
Middleware #1: After calling next

ℹ️Phải dùng await khi gọi next(context). Nếu bỏ await, dòng code phía sau next(context) có thể chạy song song với middleware kế tiếp, dẫn đến lỗi race condition trên response stream.

Cú pháp ngắn gọn hơn (từ .NET 7+)

ASP.NET Core cho phép viết gọn hơn bằng cách dùng IMiddlewareFactory-friendly overload:

Program.cscsharp
// Overload ngắn: (context, next) không cần khai báo kiểu tường minh
app.Use(async (context, next) =>
{
    // xử lý trước
    await next(context);
    // xử lý sau
});

3 app.Run — tạo terminal middleware

Bài 3 · Use app.Run to create middleware

app.Run() luôn tạo ra một terminal middleware — middleware cuối cùng trong pipeline. Nó không nhận tham số next nên không thể gọi middleware kế tiếp.

⚠️Bất kỳ middleware nào đăng ký sau app.Run() sẽ không bao giờ được thực thi vì pipeline đã bị dừng lại.

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Middleware #1 — thông thường
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #1: Before calling next\r\n");

    await next(context);

    await context.Response.WriteAsync("Middleware #1: After calling next\r\n");
});

// Middleware #2 — terminal, không có tham số next
app.Run(async (context) =>
{
    await context.Response.WriteAsync("Middleware #2: Processed.\r\n");
});

// Middleware #3 — KHÔNG BAO GIỜ được thực thi
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #3: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #3: After calling next\r\n");
});

app.Run(); // khởi động web server

Output

Response (text/plain)text
Middleware #1: Before calling next
Middleware #2: Processed.
Middleware #1: After calling next

💡Lưu ý app.Run() cuối file (không có tham số) là lệnh khởi động server, khác với app.Run(delegate) dùng để thêm terminal middleware.

4 app.Map — rẽ nhánh pipeline

Bài 4 · Use app.Map to branch the pipeline

app.Map() tạo ra một nhánh pipeline riêng biệt dựa trên đường dẫn URL (path prefix). Khi request có path khớp với prefix được chỉ định, request sẽ đi vào nhánh đó và không bao giờ quay lại pipeline chính.

Cú pháp

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Middleware #1 — pipeline chính
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #1: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #1: After calling next\r\n");
});

// Nhánh riêng: chỉ áp dụng khi path bắt đầu bằng "/employees"
app.Map("/employees", (appBuilder) =>
{
    appBuilder.Use(async (HttpContext context, RequestDelegate next) =>
    {
        await context.Response.WriteAsync("Middleware #5: Before calling next\r\n");
        await next(context);
        await context.Response.WriteAsync("Middleware #5: After calling next\r\n");
    });

    appBuilder.Use(async (HttpContext context, RequestDelegate next) =>
    {
        await context.Response.WriteAsync("Middleware #6: Before calling next\r\n");
        await next(context);
        await context.Response.WriteAsync("Middleware #6: After calling next\r\n");
    });
});

// Middleware #2 — chỉ chạy khi path KHÔNG khớp "/employees"
app.Use(async (context, next) =>
{
    await context.Response.WriteAsync("Middleware #2: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #2: After calling next\r\n");
});

// Middleware #3 — chỉ chạy khi path KHÔNG khớp "/employees"
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #3: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #3: After calling next\r\n");
});

app.Run();

Minh họa luồng thực thi

Request pathThứ tự middleware thực thi
GET /#1 → #2 → #3 (pipeline chính)
GET /employees#1 → #5 → #6 (nhánh /employees, không về #2, #3)
GET /employees/123#1 → #5 → #6 (nhánh /employees, path prefix khớp)

ℹ️app.Map() khớp theo prefix, không phải exact match. /employees/123 vẫn vào nhánh /employees. Nhánh rẽ bởi Map không rejoining về pipeline chính — đây là đặc điểm phân biệt với UseWhen.

5 app.MapWhen — rẽ nhánh có điều kiện phức tạp

Bài 5 · Use app.MapWhen to branch the pipeline

app.Map() chỉ cho phép điều kiện là path prefix. Khi cần rẽ nhánh dựa trên điều kiện phức tạp hơn (query string, header, method HTTP, body…), dùng app.MapWhen().

Cú pháp

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Middleware #1
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #1: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #1: After calling next\r\n");
});

// Rẽ nhánh: path = /employees VÀ có query string "id"
app.MapWhen(
    (context) =>
    {
        return context.Request.Path.StartsWithSegments("/employees") &&
               context.Request.Query.ContainsKey("id");
    },
    (appBuilder) =>
    {
        appBuilder.Use(async (HttpContext context, RequestDelegate next) =>
        {
            await context.Response.WriteAsync("Middleware #5: Before calling next\r\n");
            await next(context);
            await context.Response.WriteAsync("Middleware #5: After calling next\r\n");
        });

        appBuilder.Use(async (HttpContext context, RequestDelegate next) =>
        {
            await context.Response.WriteAsync("Middleware #6: Before calling next\r\n");
            await next(context);
            await context.Response.WriteAsync("Middleware #6: After calling next\r\n");
        });
    });

// Middleware #2 — chỉ chạy khi điều kiện MapWhen KHÔNG thỏa
app.Use(async (context, next) =>
{
    await context.Response.WriteAsync("Middleware #2: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #2: After calling next\r\n");
});

// Middleware #3
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #3: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #3: After calling next\r\n");
});

app.Run();

Tham số của MapWhen

Tham sốKiểuMô tả
predicateFunc<HttpContext, bool>Hàm nhận HttpContext, trả về true/false. Toàn bộ context có thể được kiểm tra.
configurationAction<IApplicationBuilder>Hành động cấu hình nhánh pipeline mới.

💡Vì có thể truy cập toàn bộ HttpContext, MapWhen có thể đưa ra quyết định dựa trên bất kỳ thứ gì: header Authorization, request method, content-type, claim của user… Tuy nhiên tương tự Map, nhánh này không rejoining về pipeline chính.

6 app.UseWhen — nhánh có thể rejoining

Bài 6 · Use app.UseWhen to create rejoinable branch

app.UseWhen() hoạt động giống MapWhen về cú pháp điều kiện, nhưng có điểm khác biệt quan trọng: sau khi nhánh thực thi xong, request quay về và rejoining pipeline chính từ điểm ngay sau chỗ UseWhen được đặt.

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Middleware #1
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #1: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #1: After calling next\r\n");
});

// UseWhen: rẽ nhánh có điều kiện, SAU ĐÓ rejoining pipeline chính
app.UseWhen(
    (context) =>
    {
        return context.Request.Path.StartsWithSegments("/employees") &&
               context.Request.Query.ContainsKey("id");
    },
    (appBuilder) =>
    {
        appBuilder.Use(async (HttpContext context, RequestDelegate next) =>
        {
            await context.Response.WriteAsync("Middleware #5: Before calling next\r\n");
            await next(context);
            await context.Response.WriteAsync("Middleware #5: After calling next\r\n");
        });

        appBuilder.Use(async (HttpContext context, RequestDelegate next) =>
        {
            await context.Response.WriteAsync("Middleware #6: Before calling next\r\n");
            await next(context);
            await context.Response.WriteAsync("Middleware #6: After calling next\r\n");
        });
    });

// Middleware #2 — được thực thi dù điều kiện UseWhen có thỏa hay không
app.Use(async (context, next) =>
{
    await context.Response.WriteAsync("Middleware #2: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #2: After calling next\r\n");
});

// Middleware #3
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #3: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #3: After calling next\r\n");
});

app.Run();

So sánh Map / MapWhen / UseWhen

Phương thứcĐiều kiện rẽ nhánhRejoining pipeline chính?
app.Map(path)Path prefix (string cố định)Không
app.MapWhen(predicate)Bất kỳ điều kiện nào trên HttpContextKhông

🔑Dùng UseWhen khi bạn muốn chạy thêm một vài middleware có điều kiện (ví dụ: logging đặc biệt, kiểm tra header) nhưng sau đó vẫn để pipeline chính tiếp tục xử lý (routing, endpoint…). Dùng MapWhen khi nhánh đó hoàn toàn độc lập và không cần pipeline chính nữa.

7 Tránh lỗi khi ghi response

Bài 7 · Avoid problems when writing response

Khi middleware gọi context.Response.WriteAsync(), ASP.NET Core bắt đầu gửi response headers đến client. Sau thời điểm đó, việc thay đổi headers (status code, Content-Type, custom headers…) sẽ gây lỗi vì headers đã được flush xuống stream.

Lỗi phổ biến

Program.cs — SAI (header sau WriteAsync)csharp
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #1: Before calling next\r\n");

    // LỖI: headers đã bị flush, không thể đặt header nữa
    context.Response.Headers["My-Header"] = "My-Header-Content";
    context.Response.StatusCode = 201;

    await next(context);
});

Cách đúng: đặt headers TRƯỚC khi WriteAsync

Program.cs — ĐÚNG (header trước WriteAsync)csharp
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    // Đặt headers TRƯỚC khi ghi body
    context.Response.Headers["My-Header"] = "My-Header-Content";
    context.Response.StatusCode = 200;

    await context.Response.WriteAsync("Middleware #1: Before calling next\r\n");

    await next(context);
});

Kiểm tra xem response đã bắt đầu chưa

Nếu cần kiểm tra trước khi thay đổi header (ví dụ trong middleware xử lý lỗi), dùng thuộc tính context.Response.HasStarted:

Program.cscsharp
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await next(context);

    // Chỉ thêm header nếu response chưa bắt đầu gửi
    if (!context.Response.HasStarted)
    {
        context.Response.Headers["X-Custom-Header"] = "value";
    }
});

⚠️Lỗi "Headers are read-only, response has already started" là lỗi rất phổ biến trong middleware. Luôn nhớ quy tắc: set headers trước, write body sau. Khi middleware xử lý response trên đường về (sau await next(context)), headers đã bị flush nên chỉ có thể ghi thêm body hoặc dùng HasStarted để bảo vệ.

8 Các middleware có sẵn (Built-in)

Bài 8 · Built-in Middleware Components

ASP.NET Core đi kèm rất nhiều middleware có sẵn, được Microsoft xây dựng và kiểm tra kỹ lưỡng. Trong thực tế, bạn hiếm khi cần tự viết middleware từ đầu cho các tính năng phổ biến — hãy dùng built-in trước.

Các built-in middleware quan trọng

Extension methodChức năng
app.UseExceptionHandler()Bắt exception toàn cục, chuyển hướng sang trang lỗi hoặc endpoint xử lý lỗi.
app.UseHsts()Thêm header Strict-Transport-Security cho HTTPS.
app.UseHttpsRedirection()Chuyển hướng HTTP sang HTTPS tự động.
app.UseStaticFiles()Phục vụ file tĩnh từ thư mục wwwroot.
app.UseRouting()Kích hoạt routing — phân tích request path để khớp với endpoint.
app.UseAuthentication()Xác thực người dùng (đọc cookie, JWT…).
app.UseAuthorization()Kiểm tra quyền truy cập của người dùng đã xác thực.
app.UseCors()Cấu hình Cross-Origin Resource Sharing.
app.UseSession()Kích hoạt session.
app.UseResponseCaching()Cache response ở phía server.
app.UseResponseCompression()Nén response (gzip, brotli).
app.MapControllers()Kích hoạt endpoint routing cho MVC/Web API controllers.
app.MapRazorPages()Kích hoạt endpoint routing cho Razor Pages.

Thứ tự khuyến nghị cho ứng dụng MVC / Web API

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddControllersWithViews();
var app = builder.Build();

if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler("/Home/Error");
    app.UseHsts();
}

app.UseHttpsRedirection();
app.UseStaticFiles();

app.UseRouting();

app.UseAuthentication();
app.UseAuthorization();

// Custom middleware của bạn đặt sau Authorization, trước endpoint
// app.UseMiddleware<MyCustomMiddleware>();

app.MapControllerRoute(
    name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}");

app.Run();

🔑Thứ tự middleware rất quan trọng. Ví dụ: UseAuthentication phải đứng trước UseAuthorization; UseRouting phải đứng trước UseAuthorization và MapControllers. Sai thứ tự sẽ gây lỗi logic khó phát hiện.

Middleware là extension method

Mỗi built-in middleware thực chất là một extension method trên IApplicationBuilder. Bạn có thể xem source code của chúng trên GitHub của ASP.NET Core. Đây cũng là cách bạn nên đóng gói custom middleware của mình — viết extension method để code gọi trở nên gọn gàng.

9 Custom Middleware Class

Bài 9 · Custom Middleware Class

Khi pipeline có nhiều middleware inline trong Program.cs, file trở nên lộn xộn và khó bảo trì. Giải pháp là đưa implementation ra file riêng bằng cách tạo custom middleware class.

Hai cách tạo middleware class

CáchMô tảNhận ra qua
Convention-based Class tự do, có constructor nhận RequestDelegate và method InvokeAsync(HttpContext). ASP.NET Core nhận ra theo quy ước. Constructor + InvokeAsync
Factory-based (IMiddleware) Class implement interface IMiddleware, được resolve từ DI container mỗi request. Cần đăng ký service trước. : IMiddleware

💡Khóa học ưu tiên IMiddleware (factory-based) vì: intent rõ ràng (thấy interface là biết đây là middleware), buộc implement đúng method, tích hợp tốt với DI (có thể inject service vào constructor).

Tạo IMiddleware class

MiddleComponents/MyCustomMiddleware.cscsharp
namespace WebApp.MiddleComponents;

public class MyCustomMiddleware : IMiddleware
{
    // Có thể inject service qua constructor, ví dụ:
    // private readonly ILogger<MyCustomMiddleware> _logger;
    // public MyCustomMiddleware(ILogger<MyCustomMiddleware> logger) { _logger = logger; }

    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        await context.Response.WriteAsync("My custom middleware: Before calling next\r\n");

        await next(context); // chuyển sang middleware tiếp theo

        await context.Response.WriteAsync("My custom middleware: After calling next\r\n");
    }
}

Đăng ký service và sử dụng

Program.cscsharp
using WebApp.MiddleComponents;

var builder = WebApplication.CreateBuilder(args);

// Bắt buộc: đăng ký middleware class vào DI container
builder.Services.AddTransient<MyCustomMiddleware>();

var app = builder.Build();

// Middleware #1 (inline)

11 Bài tập 2 (đáp án) — Custom Exception Handling Middleware

Bài 11 · Assignment 2 (answer) — Custom exception handling middleware

Vì middleware pipeline là một chuỗi lồng nhau, middleware đầu tiên là hàm bao ngoài cùng — nó gọi tất cả các middleware phía sau như một call stack. Nếu đặt try/catch ở middleware đầu tiên, khối catch sẽ bắt được mọi exception ném ra từ bất kỳ middleware nào trong toàn bộ pipeline.

Giải thích logic

🔑Exception handling middleware phải đứng đầu pipeline vì nó bao quanh toàn bộ chuỗi gọi. Nếu đặt ở giữa hoặc cuối, nó chỉ bắt được exception của các middleware phía sau nó, còn exception từ middleware phía trước sẽ không bị bắt.

Triển khai

MiddleComponents/MyCustomExceptionHandler.cscsharp
namespace WebApp.MiddleComponents;

public class MyCustomExceptionHandler : IMiddleware
{
    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        try
        {
            context.Response.ContentType = "text/html";
            await next(context); // gọi toàn bộ phần còn lại của pipeline
        }
        catch (Exception ex)
        {
            // Bắt exception từ bất kỳ middleware nào phía sau
            await context.Response.WriteAsync("<h2>Error:</h2>");
            await context.Response.WriteAsync($"<p>{ex.Message}</p>");
        }
    }
}
Program.cscsharp
using WebApp.MiddleComponents;

var builder = WebApplication.CreateBuilder(args);

// Đăng ký cả hai middleware vào DI
builder.Services.AddTransient<MyCustomMiddleware>();
builder.Services.AddTransient<MyCustomExceptionHandler>();

var app = builder.Build();

// Exception handler PHẢI đứng ĐẦU TIÊN
app.UseMiddleware<MyCustomExceptionHandler>();

// Middleware #1
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #1: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #1: After calling next\r\n");
});

app.UseMiddleware<MyCustomMiddleware>();

// Middleware #2 — ném exception để test
app.Use(async (context, next) =>
{
    throw new ApplicationException("Exception for testing.");

    // Dòng dưới đây sẽ không chạy
    await context.Response.WriteAsync("Middleware #2: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #2: After calling next\r\n");
});

// Middleware #3
app.Use(async (HttpContext context, RequestDelegate next) =>
{
    await context.Response.WriteAsync("Middleware #3: Before calling next\r\n");
    await next(context);
    await context.Response.WriteAsync("Middleware #3: After calling next\r\n");
});

app.Run();

Kết quả khi có exception

Response (text/html)html
Middleware #1: Before calling next
My custom middleware: Before calling next
<h2>Error:</h2>
<p>Exception for testing.</p>

Nhận xét: Middleware #1 và MyCustomMiddleware đã chạy phần "Before" trước khi exception xảy ra. Khi exception được ném từ Middleware #2, nó bị bắt bởi MyCustomExceptionHandler (đứng đầu pipeline), không truyền lên tầng trên thành 500 Internal Server Error.

ℹ️Trong thực tế production, ASP.NET Core cung cấp app.UseExceptionHandler() với nhiều tính năng hơn (logging tự động, problem details, redirect…). Custom exception middleware như trên rất hữu ích để học cơ chế hoạt động hoặc khi cần hành vi tùy chỉnh đặc biệt.
