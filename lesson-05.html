CHƯƠNG 05

Minimal API — Model Binding & Validation

Tìm hiểu cách ASP.NET Core tự động trích xuất dữ liệu từ HTTP request vào tham số của endpoint handler (model binding), và cách xác thực dữ liệu đó trước khi đưa vào business logic (model validation).

1 Model Binding là gì

Bài 1 · What is Model Binding

Trước khi có model binding, để lấy dữ liệu từ một HTTP request, lập trình viên phải viết code thủ công: đọc route value từ HttpContext.Request.RouteValues, đọc query string từ HttpContext.Request.Query, đọc body bằng StreamReader rồi deserialize JSON... Mỗi endpoint lại phải lặp lại những đoạn code kỹ thuật này, khiến code phình to và khó tập trung vào business logic.

Model binding là tính năng của ASP.NET Core giải quyết vấn đề đó: framework tự động trích xuất dữ liệu từ HTTP request và đưa thẳng vào các tham số của endpoint handler function, giúp developer chỉ cần khai báo tham số, không cần viết code trích xuất thủ công.

Định nghĩa chính thức: Model binding là quá trình trích xuất dữ liệu từ HTTP request và ánh xạ (map) chúng vào các tham số của endpoint handler.

Các nguồn dữ liệu (Binding Sources)

ASP.NET Core Minimal API hỗ trợ bốn nguồn dữ liệu chính trong HTTP request:

NguồnVí dụAttribute
Route values/employees/{id}[FromRoute]
Query string?id=1&name=John[FromQuery]
HTTP headersAuthorization: Bearer ...[FromHeader]
Request bodyJSON payload[FromBody]

Model binding diễn ra khi nào?

Trong pipeline xử lý request, model binding xảy ra sau routing (xác định endpoint cần gọi) và trước khi endpoint handler thực sự được thực thi. Framework nhìn vào danh sách tham số của handler, tìm giá trị tương ứng trong HTTP request, rồi truyền vào.

Thuật ngữ thường dùng: ta nói "binding tham số vào nguồn dữ liệu". Về mặt dữ liệu, luồng chạy là từ HTTP request → tham số handler, nhưng về cách framework hoạt động: nó nhìn tham số trước, rồi đi tìm nguồn dữ liệu tương ứng trong request.

2 Bind vào Route Values

Bài 2 · Bind to Route Values

Route values là các đoạn giá trị được nhúng trực tiếp trong đường dẫn URL, ví dụ /employees/42 — ở đây 42 là route value của tham số id.

Implicit binding (ràng buộc ngầm định)

Nếu tên tham số trong handler trùng khớp với tên segment trong route template, ASP.NET Core tự động bind mà không cần attribute:

Program.cscsharp
app.MapGet("/employees/{id:int}", (int id) =>
{
    var employee = EmployeesRepository.GetEmployeeById(id);
    return employee;
});

Explicit binding với [FromRoute]

Dùng [FromRoute] để chỉ rõ nguồn dữ liệu, hoặc khi tên tham số handler khác tên route segment:

Program.cscsharp
// Tên tham số khác tên route segment → dùng Name để ánh xạ
app.MapGet("/employees/{id:int}", ([FromRoute(Name = "id")] int identityNumber) =>
{
    var employee = EmployeesRepository.GetEmployeeById(identityNumber);
    return employee;
});

Route parameter tùy chọn (optional)

Khi route segment là optional (có dấu ?), tham số handler cũng phải là nullable:

Program.cscsharp
// Route: /employees/{id?}
app.MapGet("/employees/{id:int?}", (int? id) =>
{
    if (id.HasValue)
        return EmployeesRepository.GetEmployeeById(id.Value);

    return null;
});

Nếu kiểu dữ liệu của tham số không khớp với giá trị trong route (ví dụ route nhận chuỗi "abc" nhưng tham số là int), framework trả về HTTP 400 Bad Request — khác với routing constraint không khớp (trả về 404).

3 Bind vào Query String

Bài 3 · Bind to Query String

Query string là phần sau dấu ? trong URL, ví dụ /employees?id=2&name=Jane. Binding từ query string hoạt động theo cùng quy tắc với route values.

Implicit binding

Nếu tên tham số handler trùng với key trong query string, không cần attribute:

Program.cscsharp
// GET /employees?id=2
app.MapGet("/employees", (int? id) =>
{
    if (id.HasValue)
        return EmployeesRepository.GetEmployeeById(id.Value);

    return null;
});

Explicit binding với [FromQuery]

Khi tên tham số handler khác key trong query string:

Program.cscsharp
// GET /employees?id=2 → bind vào tham số identityNumber
app.MapGet("/employees", ([FromQuery(Name = "id")] int? identityNumber) =>
{
    if (identityNumber.HasValue)
        return EmployeesRepository.GetEmployeeById(identityNumber.Value);

    return null;
});

Query string về bản chất là tùy chọn (không bắt buộc phải có trong URL). Vì vậy hãy luôn khai báo tham số query string là nullable (int?, string?) để tránh HTTP 400 khi client không gửi key đó.

Ưu tiên route values hơn query string

Nếu cùng một tên tham số tồn tại cả trong route values và query string, route values được ưu tiên:

Program.cscsharp
// GET /employees/1?id=3 → id = 1 (từ route, không phải query)
app.MapGet("/employees/{id:int}", (int id) =>
{
    return EmployeesRepository.GetEmployeeById(id);
});

4 Bind vào HTTP Headers

Bài 4 · Bind to HTTP Headers

Binding từ HTTP header bắt buộc phải dùng attribute [FromHeader] — không có implicit binding như route values hay query string. Điều này hợp lý vì header thường mang thông tin hệ thống (Authorization, Content-Type...), không phải data của người dùng.

Program.cscsharp
// Client gửi header: id: 2
app.MapGet("/employees", ([FromHeader] int id) =>
{
    return EmployeesRepository.GetEmployeeById(id);
});

Ánh xạ tên header khác tên tham số

Tên HTTP header thường dùng kebab-case (ví dụ employee-id) trong khi tên tham số C# dùng camelCase. Dùng Name để ánh xạ:

Program.cscsharp
// Client gửi header: identity: 3
app.MapGet("/employees", ([FromHeader(Name = "identity")] int id) =>
{
    return EmployeesRepository.GetEmployeeById(id);
});

Không thể test binding từ header bằng trình duyệt vì trình duyệt không cho phép tùy chỉnh request header tùy ý. Dùng công cụ như Postman hoặc curl để kiểm tra.

5 Nhóm tham số với AsParameters

Bài 5 · Use AsParameters to Group Parameters

Khi một endpoint cần nhận nhiều tham số từ các nguồn khác nhau (route, query string, header...), danh sách tham số của handler sẽ rất dài và khó đọc. Attribute [AsParameters] cho phép nhóm các tham số vào một class hoặc struct, giữ handler gọn gàng hơn.

Không dùng AsParameters — handler nhiều tham số

Program.cscsharp
// Khó đọc khi có nhiều nguồn khác nhau
app.MapGet("/employees/{id:int}",
    ([FromRoute] int id,
     [FromQuery] string name,
     [FromHeader] string position) =>
{
    var employee = EmployeesRepository.GetEmployeeById(id);
    employee.Name = name;
    employee.Position = position;
    return employee;
});

Dùng AsParameters — gom vào struct/class

Program.cscsharp
app.MapGet("/employees/{id:int}", ([AsParameters] GetEmployeeParams p) =>
{
    var employee = EmployeesRepository.GetEmployeeById(p.Id);
    employee.Name = p.Name;
    employee.Position = p.Position;
    return employee;
});

// Có thể dùng struct, class, hoặc record — đều hoạt động
struct GetEmployeeParams
{
    [FromRoute]
    public int Id { get; set; }

    [FromQuery]
    public string Name { get; set; }

    [FromHeader]
    public string Position { get; set; }
}

[AsParameters] "trải phẳng" (flatten) các thuộc tính của type thành các tham số riêng biệt về mặt binding. Viết [AsParameters] GetEmployeeParams p tương đương hoàn toàn với việc khai báo từng thuộc tính riêng lẻ. Bạn có thể dùng class, struct, hoặc record — tùy ý.

6 Bind mảng từ Query String hoặc Header

Bài 6 · Bind Arrays to Query Strings or Headers

ASP.NET Core hỗ trợ binding một mảng (array) từ query string hoặc header. Client gửi nhiều giá trị với cùng một key, framework tự động gom lại thành array.

Bind mảng từ Query String

Client gửi: GET /employees?id=1&id=2&id=3

Program.cscsharp
// Tên tham số trùng key → implicit binding
app.MapGet("/employees", (int[] id) =>
{
    var employees = EmployeesRepository.GetEmployees();
    return employees.Where(x => id.Contains(x.Id)).ToList();
});

// Tên tham số khác key → dùng [FromQuery(Name = ...)]
app.MapGet("/employees", ([FromQuery(Name = "id")] int[] ids) =>
{
    var employees = EmployeesRepository.GetEmployees();
    return employees.Where(x => ids.Contains(x.Id)).ToList();
});

Bind mảng từ Header

Client gửi nhiều header cùng tên: id: 1, id: 2, id: 3

Program.cscsharp
app.MapGet("/employees", ([FromHeader(Name = "id")] int[] ids) =>
{
    var employees = EmployeesRepository.GetEmployees();
    return employees.Where(x => ids.Contains(x.Id)).ToList();
});

Binding mảng không phù hợp với route values vì route template không thiết kế cho nhiều segment cùng loại. Hãy dùng query string hoặc header khi cần truyền danh sách ID.

7 Bind vào HTTP Body

Bài 7 · Bind to HTTP Body

Khi tham số của handler là một complex type (class/record, không phải primitive), ASP.NET Core Minimal API mặc định bind từ HTTP request body — giả định body chứa JSON và tự động deserialize.

Program.cscsharp
// Employee là complex type → tự động bind từ body (JSON)
app.MapPost("/employees", (Employee employee) =>
{
    if (employee is null || employee.Id <= 0)
        return "Employee is not provided or is not valid.";

    EmployeesRepository.AddEmployee(employee);
    return "Employee added successfully.";
});

Giới hạn của binding từ body trong Minimal API

Giới hạnChi tiết
Chỉ hỗ trợ JSON Minimal API chỉ đọc body dạng application/json. Không thể cấu hình để dùng XML hay format khác (khác với MVC/Razor Pages).
Chỉ một complex type duy nhất Không thể bind nhiều complex type khác nhau từ cùng một body.
Chỉ dùng cho POST/PUT/PATCH Không nên đặt body trong GET hay DELETE request — server sẽ không xử lý đúng.

Nếu body không phải JSON hợp lệ hoặc không có body, framework trả về HTTP 400 Bad Request với thông báo lỗi deserialization.

8 Custom Binding với BindAsync

Bài 8 · Custom Binding with BindAsync Method

Đôi khi bạn muốn một complex type được bind từ nguồn tùy chỉnh — ví dụ lấy Id từ query string và Name từ header thay vì từ body. Để làm điều này, định nghĩa một phương thức tĩnh BindAsync theo quy ước (convention-based, không cần implement interface) ngay trong class đó.

Quy ước BindAsync

Phương thức phải có signature chính xác sau:

Program.cscsharp
class Person
{
    public int Id { get; set; }
    public string? Name { get; set; }

    // Convention: static, trả về ValueTask<T?>, nhận HttpContext
    public static ValueTask<Person?> BindAsync(HttpContext context)
    {
        var idStr = context.Request.Query["id"];
        var nameStr = context.Request.Headers["name"];

        if (int.TryParse(idStr, out var id))
        {
            return new ValueTask<Person?>(new Person { Id = id, Name = nameStr });
        }

        return new ValueTask<Person?>(Task.FromResult<Person?>(null));
    }
}

// Sử dụng trong endpoint — không cần attribute gì thêm
app.MapGet("/people", (Person? p) =>
{
    return $"Id is {p?.Id}; Name is {p?.Name}";
});

Cách hoạt động

  1. Khi framework thấy tham số có kiểu định nghĩa BindAsync, nó ưu tiên gọi phương thức đó.
  2. Mọi logic binding khác (body, query string mặc định...) đều bị bỏ qua.
  3. Bên trong BindAsync, bạn có toàn quyền truy cập HttpContext — lấy từ bất kỳ phần nào của request.

ValueTask<T> là phiên bản hiệu năng cao hơn Task<T> (tránh cấp phát heap khi kết quả đã sẵn sàng đồng bộ). Dùng new ValueTask<T?>(value) để trả về kết quả đồng bộ, hoặc new ValueTask<T?>(someTask) để wrap một task bất đồng bộ.

9 Thứ tự ưu tiên nguồn Binding

Bài 9 · Binding Source Priorities

Khi một tham số có thể được bind từ nhiều nguồn khác nhau, ASP.NET Core tuân theo thứ tự ưu tiên sau (từ cao đến thấp):

Ưu tiênĐiều kiệnVí dụ
1 Explicit attribute được khai báo rõ ràng [FromBody], [FromQuery], [FromHeader], [FromRoute]
2 Complex type có BindAsync được định nghĩa Class có static ValueTask<T?> BindAsync(...)
3 Primitive type → bind vào route value (nếu route có segment cùng tên) (int id) khi route là /employees/{id}
4 Primitive type → bind vào query string (nếu key trùng tên) (int id) khi URL là ?id=2
5 Array/collection → bind từ query string hoặc header (int[] ids)
6 Complex type (không có BindAsync) → bind từ body (Employee employee)

Nếu bạn chỉ định explicit attribute (ưu tiên 1) nhưng nguồn đó không có dữ liệu, framework không fallback sang nguồn khác — kết quả sẽ là null hoặc HTTP 400. Hãy dùng ưu tiên này khi debug binding không hoạt động như kỳ vọng.

10 Model Validation

Bài 10 · Model Validation

Sau khi model binding hoàn thành, dữ liệu đã có mặt trong tham số handler — nhưng chưa chắc đã hợp lệ. Người dùng có thể gửi giá trị thiếu, sai định dạng, ngoài phạm vi cho phép, hoặc thậm chí dữ liệu độc hại. Model validation là bước xác thực tính hợp lệ của dữ liệu đã được bind.

Data Annotations — xác thực khai báo trên model

C# cung cấp namespace System.ComponentModel.DataAnnotations với nhiều attribute sẵn có để annotate trực tiếp lên thuộc tính của class:

Program.cscsharp
using System.ComponentModel.DataAnnotations;

public class Employee
{
    public int Id { get; set; }

    [Required]
    public string Name { get; set; }

    public string? Position { get; set; }

    [Required]
    [Range(50000, 200000)]
    public double Salary { get; set; }
}

Kích hoạt validation trong Minimal API

Khác với MVC/Razor Pages (validation tự động), Minimal API yêu cầu cài thêm package NuGet và gọi .WithParameterValidation():

Terminalbash
dotnet add package MinimalApis.Extensions
Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapPost("/employees", (Employee employee) =>
{
    EmployeesRepository.AddEmployee(employee);
    return "Employee is added successfully.";
}).WithParameterValidation(); // <-- bật validation

app.Run();

Khi validation thất bại, framework tự động trả về HTTP 400 Bad Request với danh sách lỗi chi tiết — không cần viết code kiểm tra thủ công trong handler.

Các Data Annotation attribute phổ biến

AttributeMục đích
[Required]Bắt buộc phải có giá trị (không được null/empty)
[Range(min, max)]Giá trị số phải trong khoảng [min, max]
[StringLength(max, MinimumLength = min)]Độ dài chuỗi trong khoảng [min, max]
[EmailAddress]Định dạng email hợp lệ
[Phone]Định dạng số điện thoại
[RegularExpression(pattern)]Khớp với regex pattern
[Compare(nameof(Prop))]Giá trị phải bằng thuộc tính khác (dùng cho confirm password)
[CreditCard]Số thẻ tín dụng hợp lệ
[Url]Định dạng URL hợp lệ

Data Annotations là tính năng thuần C# (System.ComponentModel.DataAnnotations), không phải của ASP.NET Core. Vì vậy chúng có thể dùng trong bất kỳ loại ứng dụng .NET nào, không chỉ web.

11 Custom Validation với ValidationAttribute

Bài 11 · Custom Model Validation with ValidationAttribute

Data Annotations tích hợp sẵn xử lý tốt các rule đơn giản trên từng thuộc tính riêng lẻ. Khi cần xác thực logic phức tạp hơn — ví dụ lương phải lớn hơn 100,000 NẾU chức vụ là Manager (cần đọc cả hai thuộc tính cùng lúc) — bạn cần tạo custom ValidationAttribute.

Tạo custom ValidationAttribute

EmployeeSalaryValidationAttribute.cscsharp
using System.ComponentModel.DataAnnotations;

public class EmployeeSalaryValidationAttribute : ValidationAttribute
{
    protected override ValidationResult? IsValid(
        object? value,
        ValidationContext validationContext)
    {
        // validationContext.ObjectInstance là object đang được validate
        var employee = validationContext.ObjectInstance as Employee;

        if (employee is not null
            && !string.IsNullOrEmpty(employee.Position)
            && employee.Position.Equals("Manager", StringComparison.OrdinalIgnoreCase)
            && employee.Salary < 100_000)
        {
            return new ValidationResult(
                "A manager's salary must be greater than or equal to 100,000.");
        }

        return ValidationResult.Success;
    }
}

Áp dụng lên model

Program.cscsharp
public class Employee
{
    public int Id { get; set; }

    [Required]
    public string Name { get; set; }

    [Required]
    public string Position { get; set; }

    [Required]
    [Range(50_000, 200_000)]
    [EmployeeSalaryValidation] // <-- custom attribute
    public double Salary { get; set; }
}

// Trong endpoint phải có .WithParameterValidation()
app.MapPost("/employees", (Employee employee) =>
{
    EmployeesRepository.AddEmployee(employee);
    return "Employee is added successfully.";
}).WithParameterValidation();
  1. Tạo class kế thừa ValidationAttribute (từ System.ComponentModel.DataAnnotations).
  2. Override phương thức IsValid(object? value, ValidationContext context).
  3. Dùng context.ObjectInstance để truy cập toàn bộ object — từ đó đọc nhiều thuộc tính cùng lúc.
  4. Trả về ValidationResult.Success nếu hợp lệ, hoặc new ValidationResult("error message") nếu không hợp lệ.
  5. Áp dụng attribute lên thuộc tính phù hợp trong model.

Custom ValidationAttribute hoạt động tốt cho logic phụ thuộc nhiều thuộc tính trong cùng một object. Nếu cần logic phức tạp hơn hoặc muốn tách biệt hoàn toàn khỏi model class, hãy cân nhắc thư viện FluentValidation.

12 Bài tập — Binding & Validation thông tin đăng ký

Bài 12 · Assignment 4 — Binding & Validating Registration Info

Bài tập này giúp bạn thực hành tổng hợp model binding và model validation trong một tình huống thực tế: xử lý thông tin đăng ký tài khoản.

Mô tả bài toán

Thông tin đăng ký gồm ba trường: Email, Password, ConfirmPassword.

Yêu cầu validation

  • Cả ba trường đều là bắt buộc.
  • Email phải có định dạng email hợp lệ.
  • Password phải có ít nhất 6 ký tự.
  • ConfirmPassword phải khớp với Password.

Yêu cầu endpoint

Xây dựng hai phiên bản endpoint /register:

  • Phiên bản 1 (GET): Thông tin đăng ký đến từ query string.
  • Phiên bản 2 (POST): Thông tin đăng ký đến từ request body.

13 Đáp án bài tập

Bài 13 · Assignment 4 (Answer) — Binding & Validating Registration Info

Model Registration với Data Annotations

Program.cscsharp
using System.ComponentModel.DataAnnotations;

public class Registration
{
    [Required]
    [EmailAddress(ErrorMessage = "Invalid email address.")]
    public string? Email { get; set; }

    [Required]
    [StringLength(100, MinimumLength = 6,
        ErrorMessage = "Password must be at least 6 characters.")]
    public string? Password { get; set; }

    [Required(ErrorMessage = "Confirm password is required.")]
    [Compare(nameof(Password), ErrorMessage = "Passwords do not match.")]
    public string? ConfirmPassword { get; set; }

    // BindAsync: cho phép lấy dữ liệu từ query string thay vì body
    public static ValueTask<Registration?> BindAsync(HttpContext context)
    {
        var email = context.Request.Query["email"].ToString();
        var pw    = context.Request.Query["password"].ToString();
        var cpw   = context.Request.Query["confirmpassword"].ToString();

        return new ValueTask<Registration?>(new Registration
        {
            Email           = email,
            Password        = pw,
            ConfirmPassword = cpw
        });
    }
}

Hai phiên bản endpoint

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Phiên bản 1: GET — dữ liệu từ query string (qua BindAsync)
// Ví dụ: GET /register?email=a@b.com&password=abc123&confirmpassword=abc123
app.MapGet("/register", (Registration? reg) =>
{
    return $"User {reg?.Email} registered successfully.";
}).WithParameterValidation();

// Phiên bản 2: POST — dữ liệu từ body (JSON), bỏ qua BindAsync bằng [FromBody]
app.MapPost("/register", ([FromBody] Registration reg) =>
{
    return $"User {reg?.Email} registered successfully.";
}).WithParameterValidation();

app.Run();

Phiên bản nào tốt hơn?

POST với body là lựa chọn đúng cho chức năng đăng ký. Khi gửi qua query string, mật khẩu xuất hiện trực tiếp trên URL — browser lưu vào lịch sử duyệt web, server log, và các proxy trung gian đều có thể thấy. Gửi qua request body (HTTPS) đảm bảo dữ liệu nhạy cảm như mật khẩu được mã hóa và không lưu trong lịch sử.
