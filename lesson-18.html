CHƯƠNG 18

Minimal API — Filter Pipeline

Endpoint filter trong Minimal API — tạo, gắn, chuỗi nhiều filter, tách logic validation ra khỏi handler để code sạch hơn.

1 Filter Pipeline là gì và tại sao dùng

Bài 1 · What is and Why use Filter Pipeline

Để hiểu filter pipeline, hãy ôn lại cách ASP.NET Core xử lý một HTTP request. Một request đi qua chuỗi middleware — mỗi middleware xử lý request rồi chuyển tiếp cho middleware kế tiếp, và response sẽ đi ngược lại theo chiều đó.

Trong chuỗi middleware đó, hai thành phần thiết yếu nhất là:

  • Routing middleware — phân tích URL và chọn endpoint phù hợp.
  • Endpoint middleware — thực thi handler của endpoint đã chọn.

Filter pipeline nằm ở đâu?

Filter pipeline nằm bên trong endpoint middleware. Khi routing đã chọn được endpoint (ví dụ MapPut("/employees/{id}")), thay vì gọi thẳng handler, ASP.NET Core sẽ chạy qua một chuỗi các endpoint filter trước, rồi mới gọi handler thật sự ở cuối cùng.

ℹ️ Filter pipeline giống như một "middleware pipeline thu nhỏ" gắn riêng cho từng endpoint, thay vì áp dụng toàn cục cho mọi request.

Mô hình hoạt động

Mỗi filter nhận hai tham số:

  • EndpointFilterInvocationContext context — chứa các argument đã được model-bind từ request (tham số của handler).
  • EndpointFilterDelegate next — delegate để gọi filter tiếp theo (hoặc handler nếu không còn filter nào).

Một filter có thể:

  1. Kiểm tra/thay đổi dữ liệu đầu vào trước khi gọi next(context).
  2. Gọi await next(context) để chuyển quyền cho filter tiếp theo / handler.
  3. Kiểm tra/thay đổi kết quả trả về sau khi next hoàn thành.
  4. Short-circuit — trả về response ngay mà không gọi next, ví dụ khi validation thất bại.

Tại sao dùng filter pipeline?

Mục đíchVí dụ
Tách logic validation ra khỏi handlerKiểm tra id hợp lệ, employee tồn tại...
Logging / tracing cho từng endpointGhi log trước & sau khi handler chạy
Xác thực nghiệp vụ (business rules)Kiểm tra quyền, trạng thái đối tượng...
Tái sử dụng logic xuyên suốt nhiều endpointFilter EnsureEmployeeExists dùng cho GET/PUT/DELETE

🎯Lợi ích chính: handler chỉ chứa logic nghiệp vụ thuần tuý; mọi cross-cutting concern (validation, logging...) được chuyển vào filter — code dễ đọc, dễ test và tái sử dụng.

2 Tạo và gắn filter vào endpoint

Bài 2 · Create and Attach a filter

Chúng ta sẽ tạo một filter kiểm tra rằng id trong URL path khớp với id trong body JSON khi gọi endpoint cập nhật employee (PUT /employees/{id}).

Bước 1 — Tạo class filter

Filter phải implement interface IEndpointFilter và override phương thức InvokeAsync.

Filters/EmployeeUpdateFilter.cscsharp
using WebApi.Models;

namespace WebApi.Filters
{
    public class EmployeeUpdateFilter : IEndpointFilter
    {
        public async ValueTask<object?> InvokeAsync(
            EndpointFilterInvocationContext context,
            EndpointFilterDelegate next)
        {
            // Lấy argument theo vị trí tham số của handler
            var id       = context.GetArgument<int>(0);      // {id} từ route
            var employee = context.GetArgument<Employee>(1); // body JSON

            if (id != employee.Id)
            {
                return Results.ValidationProblem(
                    new Dictionary<string, string[]>
                    {
                        { "id", new[] { "Employee id is not the same as id." } }
                    },
                    statusCode: 400);
            }

            // Validation qua, gọi filter/handler tiếp theo
            return await next(context);
        }
    }
}

ℹ️context.GetArgument<T>(index) lấy argument theo vị trí tương ứng với tham số của handler. Index bắt đầu từ 0, không tính các service được inject (DI parameters).

Bước 2 — Gắn filter vào endpoint bằng AddEndpointFilter<T>()

Endpoints/EmployeeEndpoints.cscsharp
using WebApi.Filters;
using WebApi.Models;

namespace WebApi.Endpoints
{
    public static class EmployeeEndpoints
    {
        public static void MapEmployeeEndpoints(this WebApplication app)
        {
            // ... các endpoint khác ...

            app.MapPut("/employees/{id:int}",
                (int id, Employee employee, IEmployeesRepository repo) =>
                {
                    return repo.UpdateEmployee(employee)
                        ? TypedResults.NoContent()
                        : Results.ValidationProblem(
                            new Dictionary<string, string[]>
                            {
                                { "id", new[] { "Employee doesn't exist." } }
                            },
                            statusCode: 404);
                })
                .AddEndpointFilter<EmployeeUpdateFilter>(); // <-- gắn filter
        }
    }
}

Khi nào filter được gọi?

  1. Client gửi PUT /employees/3 với body {"id": 5, ...}.
  2. Routing chọn endpoint MapPut.
  3. Model binding điền id = 3, employee.Id = 5 vào các tham số.
  4. EmployeeUpdateFilter.InvokeAsync được gọi: phát hiện 3 != 5, trả về 400 ngay.
  5. Handler không được thực thi.

💡Ngoài cách dùng class (AddEndpointFilter<T>()), có thể viết filter inline bằng lambda: .AddEndpointFilter(async (ctx, next) => { ... return await next(ctx); }). Dùng class khi filter phức tạp hoặc cần tái sử dụng; dùng lambda cho logic đơn giản, tạm thời.

3 Thứ tự thực thi, Model Binding & Validation

Bài 3 · Filter execution, Model Binding and Model Validation

Một câu hỏi quan trọng: filter chạy trước hay sau model binding và model validation?

Model Binding — luôn chạy trước filter

Vì bên trong InvokeAsync chúng ta có thể gọi context.GetArgument<Employee>(1) và nhận được đối tượng Employee đã được điền đầy đủ từ JSON body, điều đó chứng tỏ model binding đã hoàn thành trước khi bất kỳ filter nào chạy.

Model Validation — phụ thuộc vào thứ tự chuỗi

Trong Minimal API, model validation được thêm vào qua .WithParameterValidation() — về bản chất đây cũng là một endpoint filter. Thứ tự thực thi hoàn toàn phụ thuộc vào vị trí trong chuỗi .Add....

Endpoints/EmployeeEndpoints.cscsharp
// Trường hợp A: filter của chúng ta chạy TRƯỚC validation
app.MapPut("/employees/{id:int}", handler)
    .AddEndpointFilter<EmployeeUpdateFilter>()  // 1. chạy trước
    .WithParameterValidation();                  // 2. chạy sau

// Trường hợp B: validation chạy TRƯỚC filter của chúng ta
app.MapPut("/employees/{id:int}", handler)
    .WithParameterValidation()                   // 1. chạy trước (short-circuit nếu invalid)
    .AddEndpointFilter<EmployeeUpdateFilter>(); // 2. chỉ chạy nếu validation qua

⚠️Lưu ý quan trọng: Thứ tự gọi AddEndpointFilter và WithParameterValidation quyết định thứ tự thực thi. Nếu validation nằm trước, một request invalid sẽ bị short-circuit ngay mà filter của bạn không bao giờ được gọi. Thiết kế cẩn thận để tránh bỏ sót kiểm tra.

Tóm tắt thứ tự

BướcThành phầnGhi chú
1Model BindingLuôn xảy ra đầu tiên, trước mọi filter
2Filter / Validation (theo thứ tự Add...)Thứ tự do lập trình viên quyết định
3Endpoint handlerChỉ chạy nếu không có filter nào short-circuit

💡Thực hành tốt: đặt .WithParameterValidation() trước các filter nghiệp vụ. Data annotation validation (như [Required]) nên chạy sớm để loại bỏ request rõ ràng sai ngay từ đầu.

4 Nhiều filter tạo thành pipeline

Bài 4 · Add multiple filters to form a filter pipeline

Bạn có thể gắn nhiều filter vào cùng một endpoint bằng cách gọi AddEndpointFilter nhiều lần. Các filter hình thành một pipeline — giống như middleware pipeline nhưng ở phạm vi endpoint.

Thứ tự thực thi — mô hình "Russian Doll"

Giả sử bạn gắn ba filter A, B, C (theo thứ tự này). Thứ tự thực thi sẽ là:

Thứ tự thực thitext
→ Before A
  → Before B
    → Before C
      → Handler chạy
    ← After C
  ← After B
← After A

Filter được đăng ký trước (A) bao bọc bên ngoài, filter đăng ký sau (C) gần handler nhất — giống hệt cơ chế middleware.

Demo: gắn ba filter inline để quan sát pipeline

Endpoints/EmployeeEndpoints.cscsharp
app.MapGet("/employees", (IEmployeesRepository repo) =>
{
    Console.WriteLine("Execution of the endpoint handler.");
    return TypedResults.Ok(repo.GetEmployees());
})
.AddEndpointFilter(async (context, next) =>
{
    Console.WriteLine("Before filter A");
    var result = await next(context);
    Console.WriteLine("After filter A");
    return result;
})
.AddEndpointFilter(async (context, next) =>
{
    Console.WriteLine("Before filter B");
    var result = await next(context);
    Console.WriteLine("After filter B");
    return result;
})
.AddEndpointFilter(async (context, next) =>
{
    Console.WriteLine("Before filter C");
    var result = await next(context);
    Console.WriteLine("After filter C");
    return result;
});

Kết quả trên console khi gọi GET /employees:

Console outputtext
Before filter A
Before filter B
Before filter C
Execution of the endpoint handler.
After filter C
After filter B
After filter A

🎯Filter đăng ký trước bao bọc filter đăng ký sau. Đây là nguyên tắc LIFO (Last In, First Out) khi "tháo lớp" response — cùng pattern với middleware pipeline.

Short-circuit trong pipeline nhiều filter

Nếu một filter không gọi await next(context) mà trả về kết quả ngay, toàn bộ phần còn lại của pipeline (các filter sau và handler) sẽ không được thực thi. Chỉ phần "After" của các filter bên ngoài (đã chạy trước nó) mới được thực thi khi response đi ngược về.

Short-circuit ở filter Bcsharp
.AddEndpointFilter(async (context, next) =>
{
    Console.WriteLine("Before filter B");
    // Short-circuit: trả về lỗi ngay, không gọi next
    return Results.BadRequest("Filter B rejected the request.");
    // Filter C và handler sẽ không chạy
    // After A vẫn chạy khi response đi qua filter A
})

5 Bài tập 10 — Tách validation vào filter

Bài 5 · Assignment 10 — Create filters to extract all validation logics

Ở các bài trước, logic validation (kiểm tra employee tồn tại, kiểm tra id khớp...) được viết trực tiếp bên trong handler. Điều này làm handler dài và khó tái sử dụng. Bài tập này yêu cầu bạn tách toàn bộ validation ra thành các filter riêng.

Hiện trạng — validation lẫn lộn trong handler

EmployeeEndpoints.cs (trước khi refactor)csharp
app.MapGet("/employees/{id:int}", (int id, IEmployeesRepository repo) =>
{
    var employee = repo.GetEmployeeById(id);
    // Validation lẫn với logic nghiệp vụ
    return employee is not null
        ? TypedResults.Ok(employee)
        : Results.ValidationProblem(
            new Dictionary<string, string[]>
            {
                { "id", new[] { $"Employee with the id {id} doesn't exist." } }
            },
            statusCode: 404);
});

app.MapPut("/employees/{id:int}", (int id, Employee employee, IEmployeesRepository repo) =>
{
    // Validation 1: employee tồn tại không?
    // Validation 2: id path == id body?
    return repo.UpdateEmployee(employee)
        ? TypedResults.NoContent()
        : Results.ValidationProblem(
            new Dictionary<string, string[]>
            {
                { "id", new[] { "Employee doesn't exist." } }
            },
            statusCode: 404);
}).AddEndpointFilter<EmployeeUpdateFilter>();

Yêu cầu bài tập

  1. Tạo filter EnsureEmployeeExistsFilter — kiểm tra employee với id trong route có tồn tại không. Dùng lại cho GET /employees/{id}, PUT /employees/{id} và DELETE /employees/{id}.
  2. Tạo filter EmployeeCreateFilter — kiểm tra employee body hợp lệ khi POST /employees.
  3. Giữ nguyên EmployeeUpdateFilter — kiểm tra id path == id body.
  4. Thêm phương thức EmployeeExists(int id) vào IEmployeesRepository để filter dùng mà không phải gọi GetEmployeeById hai lần.
  5. Handler sau refactor chỉ chứa logic nghiệp vụ thuần tuý — không có validation.

⚠️Cân nhắc readability: khi validation được tách hết vào filter, handler trông rất "sạch", nhưng người đọc code sau này có thể không biết endpoint áp dụng những validation nào. Hãy đặt tên filter rõ ràng và cân nhắc thêm comment tóm tắt.

6 Bài tập 10 — Đáp án chi tiết

Bài 6 · Assignment 10 (Answer) — Create filters to extract all validation logics

Dưới đây là đáp án hoàn chỉnh, trích thẳng từ source code của giảng viên.

Bước 1 — Cập nhật interface & repository

Thêm phương thức EmployeeExists vào interface để filter có thể kiểm tra sự tồn tại mà không cần lấy toàn bộ đối tượng.

Models/IEmployeesRepository.cscsharp
namespace WebApi.Models
{
    public interface IEmployeesRepository
    {
        void AddEmployee(Employee? employee);
        bool DeleteEmployee(Employee? employee);
        bool EmployeeExists(int id);          // <-- thêm mới
        Employee? GetEmployeeById(int id);
        List<Employee> GetEmployees();
        bool UpdateEmployee(Employee? employee);
    }
}

Bước 2 — Filter kiểm tra employee tồn tại

Filter này nhận IEmployeesRepository qua constructor injection — DI container tự động cung cấp khi filter được resolve.

Filters/EnsureEmployeeExistsFilter.cscsharp
using WebApi.Models;

namespace WebApi.Filters
{
    public class EnsureEmployeeExistsFilter : IEndpointFilter
    {
        private readonly IEmployeesRepository _repository;

        public EnsureEmployeeExistsFilter(IEmployeesRepository repository)
        {
            _repository = repository;
        }

        public async ValueTask<object?> InvokeAsync(
            EndpointFilterInvocationContext context,
            EndpointFilterDelegate next)
        {
            var id = context.GetArgument<int>(0);

            if (!_repository.EmployeeExists(id))
            {
                return Results.ValidationProblem(
                    new Dictionary<string, string[]>
                    {
                        { "id", new[] { $"Employee with the id {id} doesn't exist." } }
                    },
                    statusCode: 404);
            }

            return await next(context);
        }
    }
}

Bước 3 — Filter kiểm tra dữ liệu tạo mới

Filters/EmployeeCreateFilter.cscsharp
using WebApi.Models;

namespace WebApi.Filters
{
    public class EmployeeCreateFilter : IEndpointFilter
    {
        public async ValueTask<object?> InvokeAsync(
            EndpointFilterInvocationContext context,
            EndpointFilterDelegate next)
        {
            var employee = context.GetArgument<Employee>(0);

            if (employee is null || employee.Id < 0)
            {
                return Results.ValidationProblem(
                    new Dictionary<string, string[]>
                    {
                        { "id", new[] { "Employee is not provided or is not valid." } }
                    },
                    statusCode: 400);
            }

            return await next(context);
        }
    }
}

Bước 4 — Endpoints sau khi refactor

Handler giờ chỉ chứa logic nghiệp vụ thuần tuý. Validation hoàn toàn được quản lý bởi các filter.

Endpoints/EmployeeEndpoints.cs (sau refactor)csharp
using WebApi.Filters;
using WebApi.Models;

namespace WebApi.Endpoints
{
    public static class EmployeeEndpoints
    {
        public static void MapEmployeeEndpoints(this WebApplication app)
        {
            app.MapGet("/employees", (IEmployeesRepository repo) =>
                TypedResults.Ok(repo.GetEmployees()));

            // Filter đảm bảo employee tồn tại trước khi handler chạy
            app.MapGet("/employees/{id:int}", (int id, IEmployeesRepository repo) =>
                TypedResults.Ok(repo.GetEmployeeById(id)))
                .AddEndpointFilter<EnsureEmployeeExistsFilter>();

            // Validation data annotation + filter kiểm tra nghiệp vụ
            app.MapPost("/employees", (Employee employee, IEmployeesRepository repo) =>
            {
                repo.AddEmployee(employee);
                return TypedResults.Created($"/employees/{employee.Id}", employee);
            })
            .WithParameterValidation()
            .AddEndpointFilter<EmployeeCreateFilter>();

            // Pipeline: validation => tồn tại? => id khớp? => handler
            app.MapPut("/employees/{id:int}",
                (int id, Employee employee, IEmployeesRepository repo) =>
                {
                    repo.UpdateEmployee(employee);
                    return TypedResults.NoContent();
                })
                .WithParameterValidation()
                .AddEndpointFilter<EnsureEmployeeExistsFilter>()
                .AddEndpointFilter<EmployeeUpdateFilter>();

            app.MapDelete("/employees/{id:int}",
                (int id, IEmployeesRepository repo) =>
                {
                    var employee = repo.GetEmployeeById(id);
                    repo.DeleteEmployee(employee);
                    return TypedResults.Ok(employee);
                })
                .AddEndpointFilter<EnsureEmployeeExistsFilter>();
        }
    }
}

So sánh trước và sau refactor

EndpointFilter pipelineHandler làm gì
GET /employees/{id} EnsureEmployeeExists Trả về employee
POST /employees ParameterValidation → EmployeeCreate Thêm employee, trả 201
PUT /employees/{id} ParameterValidation → EnsureEmployeeExists → EmployeeUpdate Cập nhật employee, trả 204
DELETE /employees/{id} EnsureEmployeeExists Xoá employee, trả 200

💡DI trong filter: khi dùng AddEndpointFilter<EnsureEmployeeExistsFilter>() (generic version), ASP.NET Core tự động resolve filter từ DI container, bao gồm cả các dependency trong constructor (IEmployeesRepository). Bạn không cần đăng ký filter vào DI thủ công.

🎯Kết quả cuối cùng: mỗi handler chỉ còn 2-3 dòng logic thuần tuý. Toàn bộ validation logic được tập trung, có thể test độc lập, và tái sử dụng xuyên suốt nhiều endpoint — đây là kiến trúc clean và maintainable cho Minimal API trong .NET 9.
