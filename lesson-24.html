Truy cập CSDL với Entity Framework Core

EF Core — DbContext, entities & relationships, kết nối SQL Server, migrations, repository pattern với EF Core, eager loading với Include.

1 Tại sao dùng EF Core

Bài 1 · Why use Entity Framework Core

Trước đây, project sử dụng in-memory repository để lưu trữ dữ liệu tạm thời. Mục tiêu của chương này là thay thế bằng SQL Server thật. Có hai lựa chọn phổ biến khi kết nối cơ sở dữ liệu từ .NET:

  • ADO.NET — thư viện tích hợp sẵn trong .NET, truy cập CSDL ở mức thấp bằng cách viết câu SQL thủ công.
  • Entity Framework Core — ORM (Object-Relational Mapper) xây dựng trên nền ADO.NET, tự động ánh xạ giữa class C# và bảng CSDL.

Vấn đề khi dùng ADO.NET thuần

Để thực hiện một thao tác đơn giản như lấy một bản ghi Department theo ID, lập trình viên phải tự xử lý nhiều bước:

  1. Tạo và mở SqlConnection (tài nguyên cần được giải phóng — dùng using).
  2. Viết câu lệnh SQL thô dưới dạng chuỗi, dễ gây lỗi chính tả.
  3. Tạo SqlCommand và thêm từng tham số một vào command.Parameters.
  4. Tạo SqlDataReader, duyệt kết quả và thủ công convert từng cột sang đúng kiểu dữ liệu.
  5. Đóng connection và reader, xử lý exception.

Với truy vấn có 10–20 tham số, khối code ADO.NET trở nên rất dài và dễ sinh lỗi. Đây chính là lý do ORM ra đời.

ORM giải quyết vấn đề gì

ORM — Object-Relational Mapping là kỹ thuật ánh xạ (map) các class trong ứng dụng tới các bảng trong CSDL quan hệ. Thay vì viết SQL thủ công, lập trình viên làm việc với các đối tượng C# bình thường; ORM tự sinh câu SQL tương ứng.

🎯Lý do dùng EF Core: Viết ít code hơn, ít lỗi hơn, dễ bảo trì hơn so với ADO.NET thuần. EF Core là ORM chính thức của Microsoft cho .NET, hỗ trợ nhiều CSDL: SQL Server, PostgreSQL, MySQL, SQLite, v.v.

2 EF Core hoạt động như thế nào

Bài 2 · How Entity Framework Core Works

EF Core dựa trên ba khái niệm trung tâm: Entities, DbContext và Migrations.

Entities (Thực thể)

Mỗi entity là một class C# thông thường tương ứng với một bảng trong CSDL. Mỗi property tương ứng với một cột. Quan hệ giữa các entity (1-nhiều, nhiều-nhiều) được biểu diễn bằng navigation property.

DbContext

DbContext là trung tâm của EF Core. Nó đóng vai trò như một "đại diện in-memory" của toàn bộ database:

  • Chứa các DbSet<T> — mỗi DbSet tương ứng một bảng, cung cấp LINQ để truy vấn dữ liệu.
  • Theo dõi trạng thái từng entity thông qua Change Tracker: Unchanged, Modified, Added, Deleted.
  • Khi gọi SaveChanges(), EF Core so sánh trạng thái hiện tại với trạng thái gốc và tự sinh câu INSERT / UPDATE / DELETE tương ứng.
Trạng thái Change TrackerÝ nghĩaSQL được sinh
UnchangedDữ liệu chưa thay đổiKhông có
ModifiedProperty đã được sửaUPDATE
AddedEntity mới thêm vào contextINSERT
DeletedEntity đánh dấu xóaDELETE

Migrations (Di chuyển schema)

Migration là cơ chế đồng bộ schema giữa model C# và cấu trúc CSDL thực tế. Mỗi lần thêm property vào entity hoặc tạo entity mới, developer chạy lệnh Add-Migration để EF Core sinh file migration mô tả thay đổi, rồi chạy Update-Database để áp dụng lên CSDL.

ℹ️Migration cho phép kiểm soát lịch sử thay đổi schema như source control — mỗi migration là một "commit" trên cấu trúc CSDL.

4 Cấu hình DbContext, Entities & Relationships

Bài 4 · Setup DbContext, Entities and Relationships

Bước đầu tiên khi sử dụng EF Core là tạo entity classes và DbContext — đây là "bản đồ in-memory" của CSDL.

Entity: Department

Models/Department.cscsharp
using System.ComponentModel.DataAnnotations;
using System.Text.Json.Serialization;

namespace WebApi.Models
{
    public class Department
    {
        public Department() { }

        public Department(int id, string name, string? description = "")
        {
            Id = id;
            Name = name;
            Description = description;
        }

        public int Id { get; set; }

        [Required]
        public string? Name { get; set; }

        [StringLength(500)]
        public string? Description { get; set; }

        [EmailAddress]
        public string? Email { get; set; }

        // Navigation property — không serialize để tránh vòng lặp
        [JsonIgnore]
        public List<Employee>? Employees { get; set; }
    }
}

DbContext: CompanyDbContext

Tạo thư mục Data/ và class CompanyDbContext kế thừa từ DbContext:

Data/CompanyDbContext.cscsharp
using Microsoft.EntityFrameworkCore;
using WebApi.Models;

namespace WebApi.Data
{
    public class CompanyDbContext : DbContext
    {
        public CompanyDbContext(DbContextOptions options) : base(options) { }

        // Mỗi DbSet<T> tương ứng một bảng trong CSDL
        public DbSet<Department>? Departments { get; set; }
        public DbSet<Employee>? Employees { get; set; }

        protected override void OnModelCreating(ModelBuilder modelBuilder)
        {
            base.OnModelCreating(modelBuilder);

            // Cấu hình quan hệ 1-nhiều: Department có nhiều Employee
            modelBuilder.Entity<Department>()
                .HasMany(d => d.Employees)
                .WithOne(e => e.Department)
                .HasForeignKey(e => e.DepartmentId);

            // Seed data — dữ liệu mẫu khởi tạo cùng migration
            modelBuilder.Entity<Department>().HasData(
                new Department(1, "Sales", "Sales Department"),
                new Department(2, "Engineering", "Engineering Department"),
                new Department(3, "QA", "Quality Assurance")
            );

            modelBuilder.Entity<Employee>().HasData(
                new Employee(1, "John Doe", "Engineer", 60000, 1),
                new Employee(2, "Jane Smith", "Manager", 75000, 1),
                new Employee(3, "Sam Brown", "Technician", 50000, 1),
                new Employee(4, "Alice Johnson", "Analyst", 55000, 2),
                new Employee(5, "Bob Lee", "Developer", 65000, 2),
                new Employee(6, "Carol Wang", "Designer", 70000, 2),
                new Employee(7, "David Kim", "Support", 48000, 3),
                new Employee(8, "Eve Rogers", "Consultant", 72000, 3),
                new Employee(9, "Franklin Zhang", "Architect", 80000, 3)
            );
        }
    }
}

Giải thích các thành phần quan trọng

  • DbContextOptions options trong constructor — cho phép DI container truyền cấu hình kết nối vào (connection string, provider). Luôn chuyển sang base(options).
  • DbSet<T> — biểu diễn một bảng, hỗ trợ LINQ đầy đủ: .Where(), .Find(), .ToList(), v.v.
  • OnModelCreating — nơi cấu hình Fluent API: quan hệ, ràng buộc, index, seed data. Đây là cách tường minh hơn so với Data Annotations.
  • HasMany / WithOne / HasForeignKey — khai báo quan hệ 1-nhiều giữa Department và Employee.
  • HasData() — seed data tự động được chèn vào CSDL khi chạy migration lần đầu.

ℹ️Fluent API vs Data Annotations: Cả hai đều cấu hình được schema. Fluent API (trong OnModelCreating) ưu tiên hơn vì không "ô nhiễm" class entity bằng attribute và hỗ trợ nhiều tùy chọn hơn.

5 Kết nối EF Core với SQL Server

Bài 5 · Configure EF Core to connect to SQL Server

Sau khi có CompanyDbContext, bước tiếp theo là cấu hình để EF Core biết kết nối tới đâu và đăng ký vào DI container.

1. Khai báo connection string trong appsettings

appsettings.Development.jsonjson
{
  "ConnectionStrings": {
    "CompanyManagement": "Data Source=(local);Initial Catalog=CompanyManagement;User ID=sa;Password=Password1;Trust Server Certificate=True"
  },
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning"
    }
  }
}

Giải thích các phần trong connection string:

Thành phầnÝ nghĩa
Data Source=(local)Tên server SQL. (local) = SQL Server cài local trên máy hiện tại
Initial Catalog=CompanyManagementTên database sẽ tạo/kết nối
User ID / PasswordXác thực SQL Authentication
Trust Server Certificate=TrueBỏ qua kiểm tra SSL certificate (dùng khi develop local)

2. Đăng ký DbContext vào DI container trong Program.cs

Program.cscsharp
using Microsoft.EntityFrameworkCore;
using WebApi.Data;
using WebApi.Models;

var builder = WebApplication.CreateBuilder(args);

// Đăng ký CompanyDbContext với SQL Server provider
builder.Services.AddDbContext<CompanyDbContext>(options =>
{
    options.UseSqlServer(
        builder.Configuration.GetConnectionString("CompanyManagement")
    );
});

// Thay thế in-memory repository bằng EF repository
builder.Services.AddTransient<IDepartmentsRepository, DepartmentsEFRepository>();
builder.Services.AddTransient<IEmployeesRepository, EmployeesEFRepository>();

var app = builder.Build();
// ... middleware pipeline
app.Run();
  • AddDbContext<T>(options => ...) đăng ký CompanyDbContext với lifetime Scoped (mặc định) — mỗi HTTP request có một instance DbContext riêng.
  • options.UseSqlServer(...) chỉ định provider và connection string. Đổi thành UseNpgsql(), UseSqlite()… để đổi CSDL.
  • builder.Configuration.GetConnectionString("CompanyManagement") đọc từ appsettings.json (section ConnectionStrings).

⚠️Không commit connection string (đặc biệt password) vào source control. Ở môi trường production, dùng Environment Variables, Azure Key Vault hoặc User Secrets để lưu trữ an toàn.

Migration là bước kết nối giữa model C# và cấu trúc CSDL vật lý. Lần đầu tiên, migration initial sẽ tạo toàn bộ database, bảng và seed data.

Các lệnh Migration trong Package Manager Console (PMC)

Package Manager Consolepowershell

# 2. Áp dụng migration lên SQL Server — tạo database nếu chưa có

💡Đảm bảo Default project trong PMC được chọn đúng project chứa DbContext (ở đây là WebApi) trước khi chạy lệnh.

File migration được sinh ra

Lệnh Add-Migration Init tạo thư mục Migrations/ với file mô tả thay đổi schema:

Migrations/20250204220844_Init.cscsharp
public partial class Init : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.CreateTable(
            name: "Departments",
            columns: table => new
            {
                Id = table.Column<int>(type: "int", nullable: false)
                    .Annotation("SqlServer:Identity", "1, 1"),
                Name = table.Column<string>(type: "nvarchar(max)", nullable: false),
                Description = table.Column<string>(type: "nvarchar(500)", maxLength: 500, nullable: true),
                Email = table.Column<string>(type: "nvarchar(max)", nullable: true)
            },
            constraints: table =>
            {
                table.PrimaryKey("PK_Departments", x => x.Id);
            });

        migrationBuilder.CreateTable(
            name: "Employees",
            columns: table => new
            {
                Id = table.Column<int>(type: "int", nullable: false)
                    .Annotation("SqlServer:Identity", "1, 1"),
                Name = table.Column<string>(type: "nvarchar(max)", nullable: false),
                Position = table.Column<string>(type: "nvarchar(max)", nullable: false),
                Salary = table.Column<double>(type: "float", nullable: true),
                DepartmentId = table.Column<int>(type: "int", nullable: false)
            },
            constraints: table =>
            {
                table.PrimaryKey("PK_Employees", x => x.Id);
                table.ForeignKey(
                    name: "FK_Employees_Departments_DepartmentId",
                    column: x => x.DepartmentId,
                    principalTable: "Departments",
                    principalColumn: "Id",
                    onDelete: ReferentialAction.Cascade);
            });

        // Seed data (InsertData...) được sinh tự động từ HasData()
        migrationBuilder.InsertData(
            table: "Departments",
            columns: new[] { "Id", "Description", "Email", "Name" },
            values: new object[,]
            {
                { 1, "Sales Department", null, "Sales" },
                { 2, "Engineering Department", null, "Engineering" },
                { 3, "Quality Assurance", null, "QA" }
            });
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropTable(name: "Employees");
        migrationBuilder.DropTable(name: "Departments");
    }
}
  • Phương thức Up() — chứa các thay đổi cần áp dụng khi migrate lên (tạo bảng, thêm cột, v.v.).
  • Phương thức Down() — rollback (xóa bảng), dùng khi muốn hoàn tác migration.
  • EF Core tự động tạo __EFMigrationsHistory trong database để theo dõi migration nào đã được apply.

7 Triển khai Departments EF Repository

Bài 7 · Implement Departments EF Repository

Thay vì viết EF Core trực tiếp vào endpoint, chúng ta tạo một EF Repository mới implement cùng interface IDepartmentsRepository. Nhờ Dependency Injection, chỉ cần đổi đăng ký trong Program.cs — endpoints không cần thay đổi gì.

Interface IDepartmentsRepository

Models/IDepartmentsRepository.cscsharp
namespace WebApi.Models
{
    public interface IDepartmentsRepository
    {
        void AddDepartment(Department? department);
        bool DeleteDepartment(Department? department);
        bool DepartmentExists(int departmentId);
        Department? GetDepartmentById(int id);
        List<Department> GetDepartments(string? filter = null);
        bool UpdateDepartment(Department? department);
    }
}

DepartmentsEFRepository — triển khai đầy đủ

Models/DepartmentsEFRepository.cscsharp
using Microsoft.EntityFrameworkCore;
using WebApi.Data;

namespace WebApi.Models
{
    public class DepartmentsEFRepository : IDepartmentsRepository
    {
        private readonly CompanyDbContext context;

        public DepartmentsEFRepository(CompanyDbContext context)
        {
            this.context = context;
        }

        public void AddDepartment(Department? department)
        {
            if (department == null) return;
            context.Add(department);       // EF theo dõi entity với trạng thái Added
            context.SaveChanges();         // sinh câu INSERT và thực thi
        }

        public bool DeleteDepartment(Department? department)
        {
            if (department == null) return false;

            var dep = context.Departments?.Find(department.Id);
            if (dep == null) return false;

            context.Departments?.Remove(department); // trạng thái Deleted
            context.SaveChanges();                   // sinh câu DELETE
            return true;
        }

        public bool DepartmentExists(int departmentId)
        {
            var exists = context.Departments?.Any(x => x.Id == departmentId);
            return exists.HasValue && exists.Value;
        }

        public Department? GetDepartmentById(int id)
        {
            return context.Departments?.Find(id); // tìm theo primary key, dùng cache nếu có
        }

        public List<Department> GetDepartments(string? filter = null)
        {
            if (string.IsNullOrWhiteSpace(filter))
                return context.Departments?.ToList() ?? new();

            // EF.Functions.Like = LIKE SQL — hỗ trợ pattern matching
            return context.Departments?
                .Where(x => EF.Functions.Like(x.Name, $"%{filter}%"))
                .ToList() ?? new();
        }

        public bool UpdateDepartment(Department? department)
        {
            if (department == null) return false;

            var dep = context.Departments?.Find(department.Id);
            if (dep == null) return false;

            // Cập nhật từng property — Change Tracker đánh dấu Modified
            dep.Name = department.Name;
            dep.Description = department.Description;
            dep.Email = department.Email;

            context.SaveChanges(); // sinh câu UPDATE chỉ với cột đã thay đổi
            return true;
        }
    }
}

Lưu ý quan trọng

  • context.Add(entity) — generic, EF tự xác định bảng từ kiểu. Cũng có thể dùng context.Departments.Add(department) — tường minh hơn, cùng kết quả.
  • context.Departments?.Find(id) — tìm theo primary key. Ưu điểm: nếu entity đã có trong Change Tracker (đã load trước đó trong cùng request), EF không gửi thêm query SQL.
  • EF.Functions.Like() — tương đương LIKE '%filter%' trong SQL. Không thể dùng string.Contains() với EF nếu muốn case-insensitive nhất quán trên mọi CSDL.
  • SaveChanges() — phải gọi sau mỗi thay đổi để thực sự ghi vào CSDL. Nếu quên, dữ liệu chỉ thay đổi trong bộ nhớ.

🎯Sức mạnh của Repository Pattern + DI: Endpoints không biết gì về EF Core hay SQL Server. Muốn đổi sang provider khác hoặc in-memory lại để test, chỉ cần thay một dòng đăng ký trong Program.cs.

9 Đáp án bài tập 13

Bài 9 · Assignment 13 (Answer) — Implement Employees EF Repository

Đây là cài đặt hoàn chỉnh của EmployeesEFRepository:

Models/EmployeesEFRepository.cscsharp
using Microsoft.EntityFrameworkCore;
using WebApi.Data;

namespace WebApi.Models
{
    public class EmployeesEFRepository : IEmployeesRepository
    {
        private readonly CompanyDbContext context;

        public EmployeesEFRepository(CompanyDbContext context)
        {
            this.context = context;
        }

        public void AddEmployee(Employee? employee)
        {
            if (employee == null) return;

            // Cách 1: dùng context.Employees?.Add() — tường minh
            context.Employees?.Add(employee);
            context.SaveChanges();
        }

        public bool DeleteEmployee(Employee? employee)
        {
            if (employee == null) return false;

            var emp = context.Employees?.Find(employee.Id);
            if (emp == null) return false;

            context.Employees?.Remove(employee);
            context.SaveChanges();
            return true;
        }

        public bool EmployeeExists(int employeeId)
        {
            var exists = context.Employees?.Any(x => x.Id == employeeId);
            return exists.HasValue && exists.Value;
        }

        public Employee? GetEmployeeById(int id)
        {
            // Dùng Include để tải eager Department — xem Bài 10
            return context.Employees?
                .Include(x => x.Department)
                .First(x => x.Id == id);
        }

        public List<Employee> GetEmployees(string? filter = null, int? departmentId = null)
        {
            // Lọc theo departmentId có ưu tiên cao hơn
            if (departmentId.HasValue)
            {
                return context.Employees?
                    .Where(x => x.DepartmentId == departmentId.Value)
                    .ToList() ?? new();
            }
            else if (!string.IsNullOrWhiteSpace(filter))
            {
                return context.Employees?
                    .Where(x => EF.Functions.Like(x.Name, $"%{filter}%"))
                    .ToList() ?? new();
            }

            return context.Employees?.ToList() ?? new();
        }

        public bool UpdateEmployee(Employee? employee)
        {
            if (employee == null) return false;

            var emp = context.Employees?.Find(employee.Id);
            if (emp == null) return false;

            emp.Name = employee.Name;
            emp.Position = employee.Position;
            emp.Salary = employee.Salary;
            emp.DepartmentId = employee.DepartmentId;

            context.SaveChanges();
            return true;
        }
    }
}

Điểm khác biệt với DepartmentsEFRepository

  • GetEmployees hỗ trợ hai tham số lọc: departmentId (lọc theo phòng ban) và filter (lọc theo tên). Nếu có departmentId thì ưu tiên dùng trước.
  • GetEmployeeById dùng .Include(x => x.Department) để tải eager related entity — chi tiết ở Bài 10.
  • Sử dụng context.Employees?.Add(employee) thay vì context.Add() — cả hai đều đúng, Add() tường minh hơn khi làm việc với nhiều DbSet.

10 Include related entities — Eager Loading

Bài 10 · Include related entities

Khi truy vấn một entity có navigation property, EF Core theo mặc định không tự động tải dữ liệu liên quan (lazy loading mặc định bị tắt). Để tải dữ liệu quan hệ cùng lúc với entity chính, dùng eager loading qua phương thức .Include().

Vấn đề không có Include

Không có Include — Department = nullcsharp
// Truy vấn Employee theo ID — Department navigation property sẽ là null
public Employee? GetEmployeeById(int id)
{
    return context.Employees?.Find(id);
    // employee.Department == null !
}

Giải pháp: Eager Loading với .Include()

Models/EmployeesEFRepository.cscsharp
public Employee? GetEmployeeById(int id)
{
    // Include() yêu cầu EF Core JOIN với bảng Departments
    // và populate Department navigation property
    return context.Employees?
        .Include(x => x.Department)   // eager load related Department
        .FirstOrDefault(x => x.Id == id);
}

EF Core sẽ sinh câu SQL có INNER JOIN hoặc LEFT JOIN tùy cấu hình:

SQL được sinh bởi EF Coresql
SELECT e.Id, e.Name, e.Position, e.Salary, e.DepartmentId,
       d.Id, d.Name, d.Description, d.Email
FROM Employees AS e
LEFT JOIN Departments AS d ON e.DepartmentId = d.Id
WHERE e.Id = @id

ThenInclude — Include lồng nhiều cấp

Nếu entity liên quan cũng có navigation property, dùng .ThenInclude() tiếp nối:

Ví dụ Include nhiều cấpcsharp
// Giả sử Department có navigation property Manager (Employee)
context.Employees
    .Include(e => e.Department)
        .ThenInclude(d => d.Manager)
    .ToList();

Xử lý vòng lặp tuần hoàn (Circular Reference)

Khi serialize Employee có Department, và Department lại có List<Employee>, JSON serializer sẽ gặp vòng lặp vô tận. Giải pháp: dùng [JsonIgnore] trên property gây vòng lặp.

Models/Department.cs — ngăn circular referencecsharp
public class Department
{
    // ...các property khác...

    [JsonIgnore]  // Không serialize collection Employees khi trả về API
    public List<Employee>? Employees { get; set; }
}

⚠️N+1 Query Problem: Nếu dùng lazy loading và load danh sách 100 Employee, mỗi lần access employee.Department sẽ tạo thêm 1 query SQL — tổng cộng 101 queries. Dùng .Include() (eager loading) để tải tất cả trong 1 query JOIN duy nhất.
