Exception Handling

Xử lý ngoại lệ — logging trong middleware xử lý lỗi, và Exception Handling Middleware tích hợp sẵn trong ASP.NET Core .NET 9.

1 Logging trong middleware xử lý ngoại lệ

Bài 1 · Logging in Exception Handling Middleware

Trong phần trước chúng ta đã xây dựng một custom exception handling middleware. Bài này bổ sung một điểm quan trọng còn thiếu: ghi log (logging) bên trong middleware đó, tận dụng hệ thống logging có sẵn của ASP.NET Core thay vì tự cài đặt riêng.

Tại sao phải ghi log trong exception handler?

Khi một ngoại lệ xảy ra trong pipeline, middleware bắt được nó và trả về response phù hợp cho client. Nhưng nếu chỉ dừng ở đó, đội vận hành / nhà phát triển sẽ không biết lỗi đã xảy ra ở đâu, khi nào, với thông tin gì. Ghi log ở mức Critical cho phép hệ thống logging (dù là built-in console, Serilog, hay bất kỳ provider nào) ghi lại toàn bộ chi tiết ngoại lệ để phân tích sau.

Inject ILogger vào custom middleware

Middleware ASP.NET Core là một class thường, do vậy ta inject ILogger<T> qua constructor như bất kỳ dịch vụ nào khác. Category name tự động là tên class, thuận tiện cho việc lọc log.

Middleware/MyExceptionHandlerMiddleware.cscsharp
using Microsoft.AspNetCore.Http;
using Microsoft.Extensions.Logging;

namespace MyApp.Middleware;

public class MyExceptionHandlerMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<MyExceptionHandlerMiddleware> _logger;

    public MyExceptionHandlerMiddleware(
        RequestDelegate next,
        ILogger<MyExceptionHandlerMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        try
        {
            await _next(context);   // gọi middleware tiếp theo trong pipeline
        }
        catch (Exception ex)
        {
            // Ghi log với mức Critical — mức nghiêm trọng nhất
            _logger.LogCritical(ex, "An exception occurred: {Message}", ex.Message);

            context.Response.StatusCode = StatusCodes.Status500InternalServerError;
            await context.Response.WriteAsync("An unexpected error occurred. Please try again later.");
        }
    }
}

Tại sao dùng mức Critical? Thang mức log của .NET gồm: Trace < Debug < Information < Warning < Error < Critical. Ngoại lệ chưa được xử lý ảnh hưởng trực tiếp đến toàn bộ request — đây là sự cố nghiêm trọng nhất, nên dùng Critical. Tuy nhiên trong một số dự án, Error cũng được chấp nhận; hãy tuân theo quy ước của nhóm.

Đăng ký middleware vào pipeline

Middleware xử lý ngoại lệ phải được đặt đầu tiên trong pipeline để bắt mọi ngoại lệ từ tất cả middleware phía sau.

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Phải đứng ĐẦU TIÊN để bao phủ toàn bộ pipeline
app.UseMiddleware<MyExceptionHandlerMiddleware>();

app.MapGet("/", () => "Hello World!");

app.MapGet("/error", () =>
{
    throw new ApplicationException("This is a test exception");
});

app.Run();

Kết quả log thực tế

Khi exception xảy ra, output trong console (hoặc file log) sẽ tương tự:

Console outputtext
crit: MyApp.Middleware.MyExceptionHandlerMiddleware[0]
      An exception occurred: This is a test exception
      System.ApplicationException: This is a test exception
         at Program.<>c.<<-main>b__0_1>d.MoveNext() in Program.cs:line 17
         at MyApp.Middleware.MyExceptionHandlerMiddleware.InvokeAsync(...) in ...line 19

Từ stack trace ta thấy rõ: dòng gây lỗi (throw) và dòng mà middleware gọi await _next(context) — rất hữu ích khi debug.

Không cần biết logger nào đang chạy. Bằng cách inject ILogger<T>, middleware hoàn toàn không phụ thuộc vào provider cụ thể. Dù dự án dùng console logger, Serilog, NLog hay Application Insights, middleware vẫn hoạt động đúng mà không cần sửa code.

2 Exception Handling Middleware tích hợp sẵn

Bài 2 · Use the built-in Exception Handling Middleware

ASP.NET Core cung cấp sẵn nhiều cơ chế xử lý ngoại lệ mà không cần tự viết middleware từ đầu. Bài này trình bày tất cả các lựa chọn: Developer Exception Page, UseExceptionHandler (handler path và lambda), IExceptionHandler (.NET 8+), Problem Details, và UseStatusCodePages.

1 — Developer Exception Page (môi trường Development)

UseDeveloperExceptionPage() hiển thị trang chi tiết lỗi chỉ dành cho nhà phát triển: stack trace, request headers, cookies, query string… Không bao giờ bật ở Production.

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    // Trang lỗi chi tiết — CHỈ dùng ở Development
    app.UseDeveloperExceptionPage();
}
else
{
    // Trang lỗi thân thiện cho người dùng cuối
    app.UseExceptionHandler("/error");
}

app.MapGet("/", () => "Hello World!");
app.MapGet("/boom", () => { throw new InvalidOperationException("Simulated error!"); });

app.Run();

Từ .NET 6+, WebApplication tự động gọi UseDeveloperExceptionPage() trong Development nên bạn thường không cần gọi tường minh. Tuy nhiên, viết rõ ra giúp code dễ đọc và dễ kiểm soát hơn.

2 — UseExceptionHandler với handler path

Khi có lỗi, ASP.NET Core tự động chuyển hướng (re-execute) request sang đường dẫn được chỉ định. Phù hợp khi dùng Razor Pages hoặc MVC Views để hiển thị trang lỗi.

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddRazorPages();

var app = builder.Build();

app.UseExceptionHandler("/Error");   // re-execute sang /Error
app.MapRazorPages();

app.Run();
Pages/Error.cshtml.cscsharp
using Microsoft.AspNetCore.Diagnostics;
using Microsoft.AspNetCore.Mvc.RazorPages;
using Microsoft.Extensions.Logging;

public class ErrorModel : PageModel
{
    private readonly ILogger<ErrorModel> _logger;

    public ErrorModel(ILogger<ErrorModel> logger)
    {
        _logger = logger;
    }

    public string? ErrorMessage { get; private set; }

    public void OnGet()
    {
        // Lấy thông tin ngoại lệ từ IExceptionHandlerFeature
        var exceptionFeature = HttpContext.Features.Get<IExceptionHandlerFeature>();
        if (exceptionFeature?.Error is { } ex)
        {
            _logger.LogCritical(ex, "Unhandled exception: {Message}", ex.Message);
            ErrorMessage = ex.Message;
        }
    }
}
Pages/Error.cshtmlhtml
@page
@model ErrorModel

<h2>Hệ thống gặp sự cố</h2>
<p>@Model.ErrorMessage</p>

IExceptionHandlerFeature được middleware ghi vào HttpContext.Features trước khi re-execute. Dùng HttpContext.Features.Get<IExceptionHandlerFeature>() để lấy đối tượng ngoại lệ gốc và log/hiển thị thông tin cần thiết.

3 — UseExceptionHandler với lambda (không cần endpoint riêng)

Cách này phù hợp với API — trả về JSON thay vì HTML, không cần tạo Razor Page.

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.UseExceptionHandler(errApp =>
{
    errApp.Run(async context =>
    {
        var exceptionFeature = context.Features.Get<IExceptionHandlerFeature>();
        var logger = context.RequestServices
            .GetRequiredService<ILogger<Program>>();

        if (exceptionFeature is not null)
        {
            logger.LogCritical(exceptionFeature.Error,
                "Unhandled exception at {Path}", exceptionFeature.Path);
        }

        context.Response.StatusCode = StatusCodes.Status500InternalServerError;
        context.Response.ContentType = "application/json";

        await context.Response.WriteAsJsonAsync(new
        {
            status = 500,
            message = "An internal server error occurred."
        });
    });
});

app.MapGet("/boom", () => { throw new InvalidOperationException("Lambda demo"); });

app.Run();

4 — IExceptionHandler interface (.NET 8+)

Từ .NET 8, ASP.NET Core giới thiệu interface IExceptionHandler cho phép viết handler theo kiểu class rõ ràng hơn, dễ test hơn, và hỗ trợ chuỗi handler (nhiều handler xếp nối tiếp).

Handlers/GlobalExceptionHandler.cscsharp
using Microsoft.AspNetCore.Diagnostics;
using Microsoft.AspNetCore.Http;
using Microsoft.AspNetCore.Mvc;
using Microsoft.Extensions.Logging;

namespace MyApp.Handlers;

public sealed class GlobalExceptionHandler : IExceptionHandler
{
    private readonly ILogger<GlobalExceptionHandler> _logger;

    public GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger)
    {
        _logger = logger;
    }

    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext,
        Exception exception,
        CancellationToken cancellationToken)
    {
        _logger.LogCritical(exception,
            "Exception occurred: {Message}", exception.Message);

        var problemDetails = new ProblemDetails
        {
            Status = StatusCodes.Status500InternalServerError,
            Title  = "Server Error",
            Detail = exception.Message
        };

        httpContext.Response.StatusCode = problemDetails.Status.Value;
        await httpContext.Response.WriteAsJsonAsync(problemDetails, cancellationToken);

        // Trả về true = đã xử lý, không cần handler tiếp theo
        return true;
    }
}
Program.cscsharp
var builder = WebApplication.CreateBuilder(args);

// Đăng ký IExceptionHandler implementation
builder.Services.AddExceptionHandler<GlobalExceptionHandler>();
// Cần AddProblemDetails nếu muốn dùng RFC 7807 Problem Details
builder.Services.AddProblemDetails();

var app = builder.Build();

// Kích hoạt middleware sử dụng IExceptionHandler đã đăng ký
app.UseExceptionHandler();

app.MapGet("/boom", () => { throw new ArgumentException("Bad argument!"); });

app.Run();

Chuỗi handler: Có thể đăng ký nhiều AddExceptionHandler<T>(). ASP.NET Core gọi chúng theo thứ tự đăng ký; handler nào trả về true thì dừng chuỗi. Hữu ích khi muốn xử lý riêng từng loại ngoại lệ (ví dụ: ValidationException → 422, NotFoundException → 404, còn lại → 500).

5 — Problem Details (RFC 7807)

AddProblemDetails() tích hợp chuẩn RFC 7807 vào ASP.NET Core. Khi bật, framework tự sinh JSON response theo định dạng chuẩn cho mọi HTTP error response.

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddProblemDetails(options =>
{
    // Tuỳ chỉnh thêm thông tin vào Problem Details (ví dụ: traceId)
    options.CustomizeProblemDetails = ctx =>
    {
        ctx.ProblemDetails.Extensions["traceId"] =
            ctx.HttpContext.TraceIdentifier;
    };
});

var app = builder.Build();
app.UseExceptionHandler();
app.UseStatusCodePages();   // xử lý 404, 405... không có body

app.MapGet("/not-found", () => Results.NotFound());

app.Run();

Response JSON khi có lỗi sẽ trông như sau:

HTTP 500 response bodyjson
{
  "type":    "https://tools.ietf.org/html/rfc7807",
  "title":   "Server Error",
  "status":  500,
  "detail":  "Bad argument!",
  "traceId": "00-4bf5122f344e2107-00f067aa0ba902b7-00"
}

6 — UseStatusCodePages

UseStatusCodePages() xử lý các response có mã lỗi HTTP (4xx, 5xx) nhưng không có body — điển hình là 404 do không có route khớp. Middleware này tự thêm body mặc định hoặc redirect sang trang tùy chỉnh.

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddProblemDetails();

var app = builder.Build();

app.UseExceptionHandler();

// Thêm body cho 4xx/5xx không có body (vd: 404 route không tìm thấy)
app.UseStatusCodePages();

// Hoặc redirect sang trang cụ thể:
// app.UseStatusCodePagesWithRedirects("/errors/{0}");

// Hoặc re-execute sang endpoint khác:
// app.UseStatusCodePagesWithReExecute("/errors/{0}");

app.MapGet("/", () => "Hello!");

app.Run();
Middleware / API Mục đích Khi nào dùng
UseDeveloperExceptionPage() Trang lỗi chi tiết cho developer Chỉ môi trường Development
UseExceptionHandler("/path") Re-execute sang endpoint/page lỗi MVC/Razor Pages — Production
UseExceptionHandler(lambda) Xử lý lỗi inline, trả JSON Minimal API / Web API
IExceptionHandler + AddExceptionHandler<T> Handler theo class, hỗ trợ chuỗi .NET 8+, khi cần phân loại exception
AddProblemDetails() Chuẩn hóa response lỗi RFC 7807 API trả JSON — nên bật mặc định
UseStatusCodePages() Thêm body cho 4xx/5xx trống Khi route không khớp (404) không có body

Tổng hợp: cấu hình khuyến nghị cho ứng dụng .NET 9

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddExceptionHandler<GlobalExceptionHandler>();
builder.Services.AddProblemDetails();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    // Developer Exception Page — bật sẵn trong Development nhưng viết rõ cho dễ đọc
    app.UseDeveloperExceptionPage();
}
else
{
    // Production: dùng IExceptionHandler đã đăng ký
    app.UseExceptionHandler();
    app.UseStatusCodePages();
}

app.MapGet("/", () => Results.Ok("OK"));
app.MapGet("/error", () => { throw new Exception("Demo error"); });

app.Run();

Lưu ý thứ tự middleware: UseExceptionHandler() / UseDeveloperExceptionPage() phải được gọi trước tất cả middleware khác trong pipeline để đảm bảo bắt được mọi ngoại lệ, kể cả lỗi từ middleware routing hay authentication.
