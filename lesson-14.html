CHƯƠNG 14

Razor Pages

Mô hình Razor Pages trong ASP.NET Core — route template dựa trên tên file, page handler, model binding, validation, producing results, layout/sections, partial view và view component.

1 Tại sao dùng Razor Pages

Bài 1 · Why Razor Pages

ASP.NET Core cung cấp nhiều mô hình để xây dựng ứng dụng web: Minimal API, MVC, và Razor Pages. Trong khi MVC hướng tới action method (endpoint xử lý yêu cầu), Razor Pages hướng tới trang (page) — phản ánh đúng cách người dùng tương tác với ứng dụng web: chuyển từ trang này sang trang khác.

Vấn đề với MVC khi xây dựng ứng dụng web truyền thống

Với MVC, để tìm View của một controller bạn phải biết quy ước: Views/{ControllerName}/{ActionName}.cshtml. Controller, Action, và View nằm ở ba vị trí khác nhau — thiết kế tốt về mặt kỹ thuật nhưng đôi khi gây khó khăn khi nhìn vào yêu cầu nghiệp vụ, vốn thường được mô tả theo từng trang (page).

Razor Pages là gì

Razor Pages nhóm giao diện (.cshtml) và logic xử lý (.cshtml.cs — PageModel) vào cùng một cặp file, một cặp tương ứng một trang web. Điều này làm cho cấu trúc project phản ánh trực tiếp yêu cầu nghiệp vụ: mỗi trang là một đơn vị độc lập, dễ tìm, dễ bảo trì.

ℹ️ Microsoft khuyến nghị Razor Pages là công nghệ mặc định để xây dựng các ứng dụng web server-side rendering trong ASP.NET Core. MVC vẫn được hỗ trợ đầy đủ nhưng phù hợp hơn với các project lớn, phức tạp cần tách biệt rõ ràng.

Tiêu chíMVCRazor Pages
Đơn vị tổ chứcController / ActionPage / PageModel
Vị trí logicControllers/Cùng file .cshtml.cs
Phù hợp vớiProject lớn, API, tái sử dụng caoCRUD đơn giản, page-focused
Quy ước routeTheo controller/actionTheo đường dẫn file

2 Tạo ứng dụng Razor Pages đầu tiên

Bài 2 · Create Our First Razor Pages App

Razor Pages không yêu cầu template riêng — bạn có thể bắt đầu từ template Empty và thêm hỗ trợ Razor Pages vào Program.cs.

Cấu hình Program.cs

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);

// Đăng ký các dịch vụ cần thiết cho Razor Pages
builder.Services.AddRazorPages();

var app = builder.Build();

app.UseRouting();

app.UseEndpoints(endpoints =>
{
    // Đăng ký tất cả Razor Pages làm endpoint
    endpoints.MapRazorPages();
});

app.Run();

💡 Từ .NET 6 trở đi bạn có thể dùng cú pháp ngắn hơn: app.MapRazorPages() trực tiếp trên app mà không cần UseEndpoints. Cả hai cách đều hoạt động tương đương.

Cấu trúc thư mục

Theo quy ước, toàn bộ Razor Pages được đặt trong thư mục Pages/ ở gốc project. Mỗi trang gồm hai file:

  • Pages/Index.cshtml — template Razor, chứa HTML và biểu thức @
  • Pages/Index.cshtml.cs — PageModel, chứa logic C#

File Razor Page tối giản

Pages/Index.cshtmlrazor
@page
@model WebApp.Pages.IndexModel

<h3>Hello world!</h3>
Pages/Index.cshtml.cscsharp
using Microsoft.AspNetCore.Mvc.RazorPages;

namespace WebApp.Pages;

public class IndexModel : PageModel
{
    public void OnGet()
    {
        // Xử lý GET request
    }
}

🔑 Directive @page là yếu tố quan trọng nhất — nó đánh dấu file .cshtml này là một Razor Page (không phải Razor View thông thường trong MVC). Thiếu @page, file sẽ không được xử lý như một page endpoint.

3 Route template cơ bản

Bài 3 · Route Matching Basics — Route Templates

Razor Pages tự động sinh route template dựa trên đường dẫn file tương đối so với thư mục Pages/. Đây là quy ước mạnh mẽ giúp không cần cấu hình route thủ công cho phần lớn các trang.

Quy tắc ánh xạ file → route

Đường dẫn fileRoute URL
Pages/Index.cshtml/ hoặc /Index
Pages/Employees/Index.cshtml/Employees hoặc /Employees/Index
Pages/Employees/Add.cshtml/Employees/Add
Pages/Employees/Edit.cshtml/Employees/Edit

ℹ️ Ngoại lệ đặc biệt: Trang có tên Index được ánh xạ thêm route ngắn — không cần ghi /Index ở cuối URL. Ví dụ: /Employees và /Employees/Index đều trỏ đến cùng trang Pages/Employees/Index.cshtml.

Cơ chế hoạt động

Quá trình xử lý một request Razor Pages tuân theo đúng pipeline mà bạn đã biết:

  1. Endpoint middleware đăng ký tất cả Razor Pages vào routing table khi ứng dụng khởi động.
  2. Routing middleware nhận request, so sánh URL với route templates đã đăng ký.
  3. Razor Page khớp được chọn, endpoint middleware thực thi page handler tương ứng.
  4. View engine kết hợp PageModel với template .cshtml để sinh HTML trả về client.

4 Cách Razor Pages hoạt động

Bài 4 · How Razor Pages Work

Dù Razor Pages có giao diện "hướng trang", bên dưới nó vẫn hoạt động theo cùng cơ chế endpoint handler của ASP.NET Core. Sự khác biệt chính nằm ở cách view engine biết page nào cần render.

Page Handler — endpoint thực sự

Trong MVC, endpoint handler là action method. Trong Razor Pages, endpoint handler là page handler — các method trong PageModel tuân theo quy ước đặt tên On{HttpVerb}:

Pages/Employees/Add.cshtml.cscsharp
using Microsoft.AspNetCore.Mvc.RazorPages;

namespace WebApp.Pages.Employees;

public class AddModel : PageModel
{
    // Xử lý GET /Employees/Add
    public void OnGet()
    {
    }

    // Xử lý POST /Employees/Add
    public void OnPost()
    {
    }
}

Tại sao page handler có thể trả về void

Trong MVC, action method phải trả về IActionResult vì framework cần biết view nào cần render. Với Razor Pages, view (file .cshtml) luôn được ghép cặp cố định với PageModel, nên framework đã biết phải render gì mà không cần action method chỉ định. Do đó, page handler có thể trả về void — tương đương với trả về Page().

🔑 Dù page handler trả về void, cuối cùng vẫn là view engine (không phải page handler) tạo ra HTTP response. View engine nhận kết quả từ handler, kết hợp PageModel với template Razor để sinh HTML.

5 Tùy chỉnh Route Template

Bài 5 · Customize Route Template

Mặc dù quy ước dựa trên đường dẫn file rất tiện lợi, đôi khi bạn muốn URL khác với tên file. Razor Pages cho phép tùy chỉnh route template thông qua directive @page.

Cách 1 — Thêm segment vào route mặc định

Thêm segment sau @page sẽ nối thêm vào cuối route mặc định:

Pages/Employees/Add.cshtmlrazor
@page "confirm"
@model WebApp.Pages.Employees.AddModel

<h3>Add Employee</h3>

Bây giờ trang này chỉ khớp với /Employees/Add/confirm, không còn khớp với /Employees/Add.

Cách 2 — Thay thế hoàn toàn route template

Bắt đầu bằng / để thay thế hoàn toàn route template mặc định:

Pages/Employees/Add.cshtmlrazor
@page "/add-employee"
@model WebApp.Pages.Employees.AddModel

<h3>Add Employee</h3>

Với cấu hình này, trang được truy cập tại /add-employee thay vì /Employees/Add.

💡 Trong thực tế, hầu hết dự án giữ route mặc định (theo tên file) để dễ dự đoán. Tùy chỉnh route chỉ nên dùng khi có yêu cầu URL cụ thể từ phía sản phẩm hoặc khi cần tích hợp với hệ thống cũ.

6 Route Parameters

Bài 6 · Route Parameters in Razor Pages

Route parameter trong Razor Pages được khai báo trong directive @page, sử dụng cú pháp {param} giống MVC. Có hai cách nhận giá trị route parameter trong page handler.

Cách 1 — Tham số của page handler

Pages/Employees/Edit.cshtmlrazor
@page "{id:int?}"
@model WebApp.Pages.EmployeesModel

<h3>Employee detail</h3>
Employee Id: @Model.Id
Pages/Employees/Edit.cshtml.cs — Cách 1csharp
public class EditModel : PageModel
{
    // Nhận id trực tiếp qua tham số method
    public void OnGet(int? id)
    {
        // id có giá trị từ route /Employees/Edit/42
    }
}

Cách 2 — Property với [BindProperty(SupportsGet = true)]

Pages/Employees/Edit.cshtml.cs — Cách 2 (từ ZIP thực tế)csharp
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.RazorPages;

namespace WebApp.Pages;

public class EmployeesModel : PageModel
{
    // SupportsGet = true cho phép bind từ GET request (route parameter)
    [BindProperty(SupportsGet = true)]
    public int? Id { get; set; }

    public IActionResult OnGet()
    {
        // Id đã được bind từ route /Employees/Edit/{id}
        return Page();
    }
}

ℹ️ Mặc định [BindProperty] chỉ bind cho POST. Với GET request (route parameter, query string), phải thêm SupportsGet = true. Ưu điểm của property so với tham số method là property có thể truy cập từ bất kỳ đâu trong PageModel và trong template Razor qua @Model.Id.

Route constraints

Mọi route constraint bạn đã học đều áp dụng được trong @page:

Cú phápÝ nghĩa
@page "{id:int}"Bắt buộc, kiểu int
@page "{id:int?}"Tùy chọn, kiểu int
@page "{slug:alpha}"Bắt buộc, chỉ chữ cái

7 Page Handler Selection

Bài 7 · Page Handler Selection

Một PageModel có thể chứa nhiều page handler. Framework chọn handler dựa trên tổ hợp HTTP verb + tên handler theo quy ước đặt tên: On{Verb}[{HandlerName}].

Quy ước đặt tên page handler

Tên methodKích hoạt khi
OnGet()GET request đến trang này
OnPost()POST request đến trang này
OnPostSave()POST với ?handler=save
OnPostDelete()POST với ?handler=delete
OnGetFirst()GET với ?handler=first

Ví dụ thực tế — hai form POST trên cùng trang

Pages/Employees/Edit.cshtml (trích từ ZIP)razor
@page "{id:int?}"
@model WebApp.Pages.EmployeesModel

<!-- Form Save: POST /Employees/Edit/9?handler=save -->
<form method="post" action="/employees/edit/@Model.Id?handler=save">
    @Html.AntiForgeryToken()
    <input type="hidden" name="inputmodel.id" value="@Model.Id" />
    Name: <input type="text" name="inputmodel.name" value="@Model.InputModel?.Name" />
    <button type="submit">Save</button>
</form>

<!-- Form Delete: POST /Employees/Edit/9?handler=delete -->
<form method="post" action="/employees/edit/@Model.Id?handler=delete">
    @Html.AntiForgeryToken()
    <button type="submit">Delete</button>
</form>
Pages/Employees/Edit.cshtml.cs (trích từ ZIP)csharp
public class EmployeesModel : PageModel
{
    [BindProperty(SupportsGet = true)]
    public int? Id { get; set; }

    [BindProperty]
    public InputModel? InputModel { get; set; }

    public IActionResult OnGet() => Page();

    // Xử lý form Save
    public IActionResult OnPostSave()
    {
        if (!ModelState.IsValid)
        {
            InputModel!.ErrorMessages = GetErrors();
            return Page();
        }
        // lưu dữ liệu ...
        return RedirectToPage("Index");
    }

    // Xử lý form Delete
    public IActionResult OnPostDelete()
    {
        // xóa dữ liệu ...
        return RedirectToPage("Index");
    }

    private List<string> GetErrors()
    {
        var errors = new List<string>();
        foreach (var value in ModelState.Values)
            foreach (var error in value.Errors)
                errors.Add(error.ErrorMessage);
        return errors;
    }
}

⚠️ Razor Pages bật anti-forgery token validation mặc định cho mọi POST request. Phải thêm @Html.AntiForgeryToken() vào mỗi form, hoặc dùng Tag Helper <form> (tự động thêm token). Nếu thiếu, server sẽ trả về lỗi 400 Bad Request.

8 Implicit Page Handlers

Bài 8 · Implicit Page Handlers

Khi một request đến trang Razor Pages nhưng không tìm thấy page handler phù hợp (đúng HTTP verb và handler name), ASP.NET Core không báo lỗi mà tự động sinh ra một implicit page handler.

Hành vi của implicit page handler

Implicit handler ngầm định:

  • Được tạo động bởi framework khi không có handler tường minh nào khớp.
  • Không thực thi bất kỳ logic nào.
  • Render ngay template Razor của trang đó (tương đương return Page()).

Ví dụ

Pages/Employees/Add.cshtml.cscsharp
public class AddModel : PageModel
{
    // Chỉ có OnPostSave, không có OnPostSaveOne
    public IActionResult OnPostSave()
    {
        // ...
        return RedirectToPage("Index");
    }
}

Nếu form submit đến ?handler=saveone (không khớp OnPostSaveOne), framework sẽ sinh implicit handler và vẫn render trang bình thường mà không báo lỗi. Đây là tính năng thiết kế để tránh crash khi handler tên không khớp.

💡 Implicit handler hữu ích khi bạn muốn trang render bình thường ngay cả khi POST không có handler tường minh (ví dụ form tìm kiếm chỉ cần reload trang). Tuy nhiên, hãy cẩn thận để không vô tình bỏ sót logic do tên handler đánh máy sai.

9 Model Binding trong Razor Pages

Bài 9 · Model Binding in Razor Pages

Model binding trong Razor Pages về bản chất giống hệt MVC — framework trích xuất dữ liệu từ HTTP request (form, route, query string, body) và gán vào kiểu C#. Điểm khác biệt chủ yếu là cách khai báo binding target.

Binding qua tham số page handler

Giống action method trong MVC, page handler có thể nhận dữ liệu qua tham số:

Pages/Employees/Edit.cshtml.cscsharp
public class EditModel : PageModel
{
    public IActionResult OnPostSave(int id, string name)
    {
        // id, name được bind từ form hoặc route
        return RedirectToPage("Index");
    }
}

Binding qua property (cách ưa dùng)

Cách phổ biến hơn trong Razor Pages là dùng property với [BindProperty]. Property này khả dụng toàn bộ PageModel và trong template Razor, không giới hạn trong một handler:

Pages/Employees/Edit.cshtml.cs (từ ZIP)csharp
using System.ComponentModel.DataAnnotations;
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.RazorPages;

namespace WebApp.Pages;

public class EmployeesModel : PageModel
{
    // Bind từ cả GET (route) lẫn POST
    [BindProperty(SupportsGet = true)]
    public int? Id { get; set; }

    // Bind từ POST (form body)
    [BindProperty]
    public InputModel? InputModel { get; set; }

    public IActionResult OnGet() => Page();

    public IActionResult OnPostSave()
    {
        // InputModel.Id, InputModel.Name đã được bind tự động
        if (!ModelState.IsValid)
            return Page();

        return RedirectToPage("Index");
    }
}

public class InputModel
{
    public int? Id { get; set; }

    [Required]
    public string? Name { get; set; }

    public List<string>? ErrorMessages { get; set; }
}

Binding sources — ưu tiên như MVC

Thứ tự ưu tiên nguồn binding và các attribute như [FromForm], [FromQuery], [FromRoute], [FromBody] hoạt động y hệt MVC. Bạn có thể áp dụng trực tiếp lên property.

💡 Attribute [BindProperties] (số nhiều) đặt ở cấp class sẽ áp dụng binding cho tất cả public property trong PageModel cùng lúc. Tuy nhiên, nên dùng [BindProperty] riêng lẻ trên từng property để tránh bind nhầm những property không mong muốn.

10 Model Validation trong Razor Pages

Bài 10 · Model Validations for Razor Pages

Workflow validation trong Razor Pages giống hệt MVC: data annotation → ModelState → page handler kiểm tra ModelState.IsValid.

Pipeline xử lý request (có validation)

  1. Routing middleware chọn Razor Page khớp với URL.
  2. Model binding trích xuất dữ liệu từ request vào binding model.
  3. Model validation chạy data annotation validators và ghi kết quả vào ModelState.
  4. Page handler thực thi — lập trình viên phải chủ động kiểm tra ModelState.IsValid.
  5. View engine render HTML kết quả.

Khai báo validation với Data Annotations

InputModel (trong Edit.cshtml.cs từ ZIP)csharp
using System.ComponentModel.DataAnnotations;

public class InputModel
{
    public int? Id { get; set; }

    [Required(ErrorMessage = "Tên nhân viên là bắt buộc.")]
    [StringLength(100, MinimumLength = 2)]
    public string? Name { get; set; }

    public List<string>? ErrorMessages { get; set; }
}

Xử lý ModelState trong page handler

Pages/Employees/Edit.cshtml.cscsharp
public IActionResult OnPostSave()
{
    if (!ModelState.IsValid)
    {
        // Thu thập lỗi từ ModelState để hiển thị trên trang
        InputModel!.ErrorMessages = ModelState.Values
            .SelectMany(v => v.Errors)
            .Select(e => e.ErrorMessage)
            .ToList();

        return Page(); // render lại trang với thông báo lỗi
    }

    // Logic lưu dữ liệu
    return RedirectToPage("Index");
}

Hiển thị lỗi trên template Razor

Pages/Employees/Edit.cshtml (trích từ ZIP)razor
@if (Model is not null &&
     Model.InputModel is not null &&
     Model.InputModel.ErrorMessages is not null)
{
    <ul style="color:red">
        @foreach (var errorMessage in Model.InputModel.ErrorMessages)
        {
            <li>@errorMessage</li>
        }
    </ul>
}

ℹ️ Khác với Minimal API (lỗi validation ném exception ngay), MVC và Razor Pages không tự động báo lỗi — framework chỉ ghi vào ModelState. Lập trình viên phải nhớ kiểm tra ModelState.IsValid trước khi tiến hành logic nghiệp vụ.

11 Producing Results

Bài 11 · Producing Results with Razor Pages

Page handler có thể trả về các loại kết quả khác nhau bằng cách đổi kiểu trả về thành IActionResult — giống action method trong MVC. Razor Pages không có IPageResult riêng mà dùng chung hệ thống IActionResult.

Các loại kết quả phổ biến

Cú phápKết quảKhi nào dùng
void / không returnRender trang hiện tạiHandler chỉ đọc dữ liệu, không điều hướng
return Page()Render trang hiện tạiXử lý lỗi validation, muốn ở lại cùng trang
return RedirectToPage("Index")Redirect 302 sang trang khácSau khi lưu/xóa thành công (PRG pattern)
return RedirectToPage("Edit", new { id = 5 })Redirect sang trang có route valueRedirect sang trang có tham số
return Content("...")Plain textÍt dùng trong Razor Pages

Ví dụ — luồng xử lý form đầy đủ

Pages/Employees/Edit.cshtml.cscsharp
public class EmployeesModel : PageModel
{
    [BindProperty(SupportsGet = true)]
    public int? Id { get; set; }

    [BindProperty]
    public InputModel? InputModel { get; set; }

    public IActionResult OnGet() => Page();

    public IActionResult OnPostSave()
    {
        if (!ModelState.IsValid)
        {
            InputModel!.ErrorMessages = GetErrors();
            return Page(); // ở lại trang, hiển thị lỗi
        }

        // lưu dữ liệu ...
        return RedirectToPage("Index"); // về danh sách sau khi thành công
    }

    public IActionResult OnPostDelete()
    {
        // xóa dữ liệu ...
        return RedirectToPage("Index");
    }

    private List<string> GetErrors() =>
        ModelState.Values
            .SelectMany(v => v.Errors)
            .Select(e => e.ErrorMessage)
            .ToList();
}

🔑 Page() và void đều cho kết quả như nhau — render template .cshtml liên kết với PageModel. Dùng IActionResult + Page() khi bạn cần phân nhánh: đôi khi render trang, đôi khi redirect.

12 Layout, Sections, _ViewStart, _ViewImports

Bài 12 · Layout, Sections, _ViewStart, _ViewImport

Razor Pages và MVC chia sẻ chung engine Razor, do đó Layout, Sections, _ViewStart, _ViewImports hoạt động y hệt. Điểm khác biệt duy nhất là quy ước đặt vị trí file trong thư mục Pages/ thay vì Views/.

Layout

File layout đặt trong Pages/Shared/_Layout.cshtml theo quy ước:

Pages/Shared/_Layout.cshtml (từ ZIP)razor
<html>
    <head></head>
    <body>
        <div>Header Bar</div>
        <div>
            @RenderSection("messagebar", required: false)
        </div>

        <br /><br />
        <div>
            @RenderBody()
        </div>
    </body>
</html>

_ViewStart — gán layout cho tất cả trang

13 Partial View & View Component

Bài 13 · PartialView and ViewComponent

Componentization trong Razor Pages dùng đúng hai kỹ thuật đã học ở MVC: Partial View và View Component. Không có công nghệ mới — chỉ có một vài thay đổi nhỏ về quy ước thư mục.

Partial View

Đặt partial view trong thư mục của page (hoặc Pages/Shared/). Tên file bắt đầu bằng dấu gạch dưới _ theo quy ước:

Pages/Employees/_MyPartialView.cshtml (từ ZIP)razor
<i>This is my message: @Model.Id</i>
Sử dụng partial view trong Razor Pagerazor
@page "{id:int?}"
@model WebApp.Pages.EmployeesModel

<h3>Employee detail</h3>

@* Truyền Model làm dữ liệu cho partial view *@
@await Html.PartialAsync("_MyPartialView", Model)

View Component

Quy ước thư mục: Pages/Shared/Components/{ComponentName}/Default.cshtml (thay vì Views/Shared/Components/... trong MVC). Mọi thứ còn lại giống hệt.

Pages/Shared/Components/MyViewComponent/MyViewComponent.cs (từ ZIP)csharp
using Microsoft.AspNetCore.Mvc;

namespace WebApp.Pages.Shared.Components.MyViewComponent;

[ViewComponent(Name = "MyViewComponent")]
public class MyViewComponent : ViewComponent
{
    public IViewComponentResult Invoke(string message)
    {
        ViewData["message"] = $"My message: {message}";
        return View(); // render Default.cshtml
    }
}
Pages/Shared/Components/MyViewComponent/Default.cshtml (từ ZIP)razor
<div>
    <p>Message from View Component:</p>
    @ViewData["message"]
</div>
Sử dụng View Component trong Razor Pagerazor
@page
@model WebApp.Pages.Employees.IndexModel

<h3>Employees</h3>

@* Gọi View Component và truyền tham số *@
@await Component.InvokeAsync("MyViewComponent", new { message = "Index message" })

💡 Nếu tên class kết thúc bằng ViewComponent (ví dụ MyViewComponent), framework tự suy ra tên component là My. Để tránh nhầm lẫn, dùng attribute [ViewComponent(Name = "MyViewComponent")] để đặt tên rõ ràng như trong ví dụ trên.

14 Razor Pages vs MVC

Bài 14 · Razor Pages vs MVC

Sau khi học xong Razor Pages, hãy so sánh hai mô hình một cách khách quan để biết khi nào nên chọn cái nào.

Luồng xử lý request: MVC

  1. Browser gửi HTTP request.
  2. Framework khởi tạo Controller và gọi Action method.
  3. Action method điều phối, gọi services, nạp dữ liệu vào Model (object bên ngoài controller).
  4. Action method trả về ViewResult, truyền Model vào View.
  5. View engine render HTML và trả về response.

Luồng xử lý request: Razor Pages

  1. Browser gửi HTTP request.
  2. Framework khởi tạo PageModel và gọi Page handler.
  3. Page handler điều phối, gọi services, nạp dữ liệu vào property của chính PageModel.
  4. Framework render Razor Page đã ghép cặp với PageModel.
  5. View engine render HTML và trả về response.

Sự khác biệt cốt lõi

Trong MVC, Controller chỉ làm nhiệm vụ điều phối (coordinator), còn Model là object độc lập chứa dữ liệu. Hai trách nhiệm này được tách biệt rõ ràng.

Trong Razor Pages, PageModel đảm nhiệm cả hai vai trò: vừa là controller (qua page handler), vừa là model (qua các property). Điều này làm mờ đi sự phân tách.

Tiêu chíMVCRazor Pages
Phân tách trách nhiệmRõ ràng (Controller / Model / View)PageModel kiêm Controller + Model
Định hướngKỹ thuật (action-oriented)Trang (page-oriented)
Phù hợp yêu cầu nghiệp vụCần ánh xạ thủ công requirement → controllerRequirement theo trang → tìm ngay file page
Bảo trì dài hạnCấu trúc rõ ràng, dễ mở rộngDễ tìm file theo tính năng, dễ bàn giao
Phù hợp projectLớn, phức tạp, cần tái sử dụng caoNhỏ-vừa, CRUD, tập trung vào trang

🎯Kết luận: Không có mô hình nào "tốt hơn" tuyệt đối. MVC phù hợp khi cần separation of concerns chặt chẽ và khả năng mở rộng. Razor Pages phù hợp khi yêu cầu nghiệp vụ được đặc tả theo từng trang và cần phát triển nhanh. Trong thực tế, nhiều project lớn kết hợp cả hai trong cùng một ứng dụng ASP.NET Core.
