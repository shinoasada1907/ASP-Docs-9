CHƯƠNG 09

MVC Controllers — Model Binding & Validation

Tìm hiểu cách ASP.NET Core MVC tự động ánh xạ dữ liệu từ HTTP request (form fields, query string, route, body) vào tham số action method, thứ tự ưu tiên các nguồn binding, cách xử lý khi nguồn dữ liệu bị thiếu, cách thêm Input Formatter XML, và cách dùng ModelState để kiểm tra tính hợp lệ của dữ liệu trong controller.

1 Binding vào Form Fields

Bài 1 · Binding to form fields

Hầu hết những gì đã học về model binding trong Minimal API vẫn áp dụng được trong MVC Controllers. Điểm khác biệt quan trọng nhất là controller có thể nhận dữ liệu từ form fields — điều mà Minimal API không hỗ trợ (vì form fields chỉ xuất hiện trong ứng dụng có giao diện người dùng, không phải API thuần).

Binding thông thường — route & query string

Với tham số kiểu nguyên thủy (ví dụ int id), ASP.NET Core MVC hoạt động giống hệt Minimal API: framework tự tìm giá trị trong route parameter hoặc query string và ánh xạ vào tham số action method.

Controllers/DepartmentsController.cscsharp
using Microsoft.AspNetCore.Mvc;

public class DepartmentsController : Controller
{
    // GET /departments/details/5   → id = 5  (route)
    // GET /departments/details?id=5 → id = 5  (query string)
    public string Details(int? id)
    {
        return $"Department info: {id}";
    }
}

Binding từ form fields (x-www-form-urlencoded)

Khi client gửi một POST request với Content-Type: application/x-www-form-urlencoded (hoặc multipart/form-data), ASP.NET Core MVC tự động đọc các field trong body và ánh xạ chúng vào tham số có tên khớp. Không cần thêm attribute nào.

Controllers/DepartmentsController.cscsharp
[HttpPost]
public string Create(string name, string description)
{
    // name và description được binding tự động từ form fields
    return $"Đã tạo: {name} — {description}";
}

Ví dụ request gửi từ Postman hoặc một HTML form:

HTTP Request (form)http
POST /departments/create HTTP/1.1
Content-Type: application/x-www-form-urlencoded

name=Software&description=Best+department

ℹ️ Nếu action method không được đánh dấu [HttpPost]/[HttpGet]… thì mặc định nó chấp nhận mọi HTTP method. Trong demo và kiểm thử với Postman, đây là điều thuận tiện để thử cả GET lẫn POST mà không cần thay đổi controller.

2 Binding Complex Types

Bài 2 · Complex types

Trong Minimal API, để nhận một object phức tạp (complex type) từ HTTP request body, bắt buộc phải dùng [FromBody]. Tuy nhiên, trong MVC Controllers, tình huống linh hoạt hơn: nếu không có attribute nào được chỉ định, framework sẽ tự động tìm giá trị trong form fields hoặc query string để điền vào từng property của object.

Model Department

Đây là class model lấy trực tiếp từ project trong zip:

Models/Department.cscsharp
using System.ComponentModel.DataAnnotations;

namespace WebApp.Models
{
    public class Department
    {
        public int Id { get; set; }

        [Required]
        public string? Name { get; set; }

        [StringLength(500)]
        public string? Description { get; set; }
    }
}

Controller nhận complex type không cần attribute

Controllers/DepartmentsController.cscsharp
using Microsoft.AspNetCore.Mvc;
using WebApp.Models;

public class DepartmentsController : Controller
{
    // Không có [FromBody], [FromForm] — framework tự binding từ form/query
    [HttpPost]
    public object Create(Department department)
    {
        return department; // tự động serialize thành JSON
    }
}

Gửi request dạng form-data từ Postman:

Postman — Body: form-datatext
Id          = 1
Name        = Software Development
Description = Best department ever

Hoặc dùng query string:

HTTP Requesthttp
POST /departments/create?Id=1&Name=Software&Description=Good+dept HTTP/1.1

Cả hai cách đều trả về object Department đã được điền đầy đủ và serialize thành JSON:

Response JSONjson
{
  "id": 1,
  "name": "Software Development",
  "description": "Best department ever"
}

🔑Khác biệt then chốt so với Minimal API: Trong Minimal API, complex type mặc định đến từ body (phải là JSON). Trong MVC Controller, complex type mặc định đến từ form fields hoặc query string. Muốn đọc từ body JSON phải dùng [FromBody] tường minh.

Program.cs — đăng ký MVC với routing

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();

var app = builder.Build();

app.UseRouting();

app.UseEndpoints(endpoints =>
{
    endpoints.MapControllerRoute(
        name: "default",
        pattern: "{controller=Home}/{action=Index}/{id?}"
    );
});

app.Run();

3 Binding Source Priorities

Bài 3 · Binding source priorities

Khi một tham số action method có thể đến từ nhiều nguồn cùng lúc (ví dụ cả form lẫn route đều có id), ASP.NET Core MVC áp dụng thứ tự ưu tiên (priority) để quyết định lấy từ đâu. Thứ tự này khác đôi chút so với Minimal API.

Thứ tự ưu tiên trong MVC Controllers

Ưu tiênNguồn bindingGhi chú
1 Explicit binding attribute [FromQuery], [FromRoute], [FromForm], [FromBody], [FromHeader] — luôn được ưu tiên tuyệt đối.
2 Binding async (custom model binder) Custom IModelBinder đăng ký trong DI.
3 Form fields & query string (primitive hoặc complex type) Cao hơn route — đây là điểm khác biệt chính so với Minimal API.
4 Route parameters (primitive type) Ví dụ {id} trong URL template.
5 Query string (complex type) Ánh xạ từng property của complex type qua query string.
6 Headers Ít dùng, cần attribute [FromHeader] tường minh.

⚠️Lưu ý quan trọng: Binding từ body (JSON/XML) không có trong danh sách mặc định của MVC Controllers — muốn đọc body phải dùng [FromBody] tường minh (thuộc nhóm ưu tiên 1 — explicit).

Ví dụ minh họa thứ tự ưu tiên

Kịch bản: cùng tên id xuất hiện ở cả form field và route parameter — xem form field thắng:

Controllers/DepartmentsController.cscsharp
[HttpPost]
[Route("departments/details/{id}")]
public string Details(int? id)
{
    return $"Department info: {id}";
}

Gửi POST request với route /departments/details/100 đồng thời có form field id=900:

HTTP Requesthttp
POST /departments/details/100 HTTP/1.1
Content-Type: application/x-www-form-urlencoded

id=900

Kết quả: Department info: 900 — form field (ưu tiên 3) thắng route parameter (ưu tiên 4).

💡Để tránh nhầm lẫn, hãy luôn dùng attribute tường minh khi cần binding từ một nguồn cụ thể: [FromRoute], [FromForm], [FromQuery].

4 Missing Binding Source

Bài 4 · Missing binding source

Điều gì xảy ra khi action method có tham số nhưng HTTP request không cung cấp dữ liệu tương ứng? Hành vi này trong MVC Controllers khác với Minimal API và cần hiểu rõ để tránh bug tiềm ẩn.

Trường hợp 1 — Complex type không có dữ liệu

Khi không có bất kỳ dữ liệu nào trong request khớp với complex type, MVC Controller sẽ tạo một instance mới với toàn bộ property ở giá trị mặc định — không ném lỗi, không trả về 400.

Controllers/DepartmentsController.cscsharp
[HttpPost]
public object Create(Department department)
{
    // Nếu request không có dữ liệu:
    // department.Id   = 0   (default int)
    // department.Name = null
    // department.Description = null
    // Không có lỗi nào được throw!
    return department;
}

⚠️Khác biệt với Minimal API: Trong Minimal API, nếu [FromBody] không nhận được JSON hợp lệ, framework tự động trả về HTTP 400 Bad Request. Trong MVC Controller, không có hành vi bảo vệ tự động này — bạn phải tự kiểm tra ModelState.

Trường hợp 2 — Primitive type không có dữ liệu

Với tham số nullable (int? id), giá trị sẽ là null. Với tham số không nullable (int id), giá trị mặc định là 0.

Controllers/DepartmentsController.cscsharp
// GET /departments/details  (không có id)
public string Details(int? id)
{
    // id = null — không có lỗi
    return $"Department info: {id}";  // "Department info: "
}

// GET /departments/info  (không có id)
public string Info(int id)
{
    // id = 0 — không có lỗi
    return $"Department info: {id}";  // "Department info: 0"
}

Trường hợp 3 — Route parameter bắt buộc bị thiếu

Nếu route template yêu cầu {id} (không optional) nhưng URL không cung cấp, routing sẽ không match endpoint đó — request có thể trả về 404 mà không bao giờ vào action method.

Controllers/DepartmentsController.cscsharp
[HttpGet]
[Route("departments/details/{id}")]   // id bắt buộc
public string Details(int id)
{
    return $"Department: {id}";
}

// GET /departments/details     → 404 (route không match)
// GET /departments/details/5   → "Department: 5"

// Nếu dùng {id?} thì:
[HttpGet]
[Route("departments/details/{id?}")]  // id optional
public string DetailsOptional(int? id)
{
    return $"Department: {id}";
}
// GET /departments/details     → "Department: "  (id = null)

5 Input Formatter

Bài 5 · Input Formatter

Khi sử dụng [FromBody] để đọc dữ liệu từ body của HTTP request, ASP.NET Core cần biết cách giải mã (deserialize) nội dung đó thành object C#. Thành phần đảm nhiệm việc này được gọi là Input Formatter.

Input Formatter là gì?

Input Formatter là thành phần đọc raw bytes từ HTTP request body, xác định định dạng dữ liệu (dựa vào header Content-Type), và chuyển đổi thành object .NET phù hợp.

Content-TypeInput FormatterMặc định?
application/jsonSystem.Text.JsonCó — được thêm tự động khi gọi AddControllers()
application/xml / text/xmlXmlSerializer / DataContractKhông — phải đăng ký thêm
application/x-www-form-urlencodedFormCollectionDùng cho form, không dùng cho [FromBody]

Vấn đề khi dùng XML mà chưa đăng ký formatter

Mặc định chỉ có JSON formatter. Nếu client gửi XML body mà không đăng ký XML formatter, server trả về HTTP 415 Unsupported Media Type.

Controllers/DepartmentsController.cscsharp
[HttpPost]
public object Create([FromBody] Department department)
{
    return department;
}
HTTP Request — XML bodyxml
POST /departments/create HTTP/1.1
Content-Type: application/xml

<Department>
  <Id>12</Id>
  <Name>Software</Name>
  <Description>Good department</Description>
</Department>

Kết quả khi chưa có XML formatter: 415 Unsupported Media Type.

Thêm XML Input Formatter

Để hỗ trợ cả JSON lẫn XML, thêm XML Serializer Formatter vào chuỗi cấu hình AddControllers():

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers()
    .AddXmlSerializerFormatters(); // Thêm dòng này để hỗ trợ XML

var app = builder.Build();

app.UseRouting();
app.UseEndpoints(endpoints =>
{
    endpoints.MapControllerRoute(
        name: "default",
        pattern: "{controller=Home}/{action=Index}/{id?}"
    );
});

app.Run();

Sau khi thêm, server hỗ trợ cả hai định dạng:

  • Content-Type: application/json → dùng JSON formatter (System.Text.Json)
  • Content-Type: application/xml → dùng XML formatter (XmlSerializer)

💡Ngoài AddXmlSerializerFormatters(), còn có AddXmlDataContractSerializerFormatters() dùng DataContractSerializer — phù hợp hơn khi model có [DataContract]/[DataMember] attributes. Thông thường, AddXmlSerializerFormatters() là lựa chọn phổ biến hơn.

6 ModelState & Validation

Bài 6 · Model State

Trong MVC Controllers, các Data Annotation như [Required], [StringLength], [Range]… được kiểm tra tự động trong quá trình model binding. Tuy nhiên — khác với Minimal API — kết quả kiểm tra không tự động trả về lỗi HTTP 400. Kết quả được lưu vào đối tượng ModelState và bạn phải tự kiểm tra trong action method.

ModelState là gì?

ModelState (kiểu ModelStateDictionary) là dictionary lưu trữ:

  • Các lỗi binding (ví dụ: truyền chuỗi "abc" vào tham số int)
  • Các lỗi validation từ Data Annotation (ví dụ: bỏ trống trường [Required])
  • Giá trị gốc (attempted value) mà client đã gửi

Vấn đề khi không kiểm tra ModelState

Xét action method sau với model Department có [Required] Name:

Controllers/DepartmentsController.cscsharp
// Sai — không kiểm tra ModelState
[HttpPost]
public object Create([FromBody] Department department)
{
    // Nếu client không gửi Name, department.Name = null
    // Nhưng action vẫn thực thi bình thường — lỗi bị bỏ qua!
    return department;
}

Gửi JSON thiếu Name:

HTTP Request — thiếu Namejson
POST /departments/create HTTP/1.1
Content-Type: application/json

{ "id": 1, "description": "No name department" }

Kết quả: action vẫn chạy, trả về object với name: null — không có lỗi nào báo về client.

Giải pháp — kiểm tra ModelState.IsValid

Controllers/DepartmentsController.cscsharp
using Microsoft.AspNetCore.Mvc;
using WebApp.Models;

public class DepartmentsController : Controller
{
    [HttpPost]
    public IActionResult Create([FromBody] Department department)
    {
        if (!ModelState.IsValid)
        {
            // Trả về 400 kèm danh sách lỗi validation
            return BadRequest(ModelState);
        }

        // Dữ liệu hợp lệ — xử lý tiếp
        return Ok(department);
    }

    // Ví dụ với tham số primitive bị sai kiểu
    [HttpGet]
    [Route("departments/details/{id}")]
    public IActionResult Details(int id)
    {
        if (!ModelState.IsValid)
        {
            return BadRequest(ModelState);
        }
        return Ok($"Department: {id}");
    }
}

Response khi Name bị thiếu:

Response — 400 Bad Requestjson
{
  "Name": [
    "The Name field is required."
  ]
}

[ApiController] — kiểm tra ModelState tự động

Nếu thêm attribute [ApiController] vào controller, ASP.NET Core sẽ tự động kiểm tra ModelState.IsValid và trả về 400 trước khi vào action method — giống như Minimal API. Không cần viết thủ công nữa.

Controllers/DepartmentsController.cscsharp
[ApiController]
[Route("api/[controller]")]
public class DepartmentsController : ControllerBase
{
    [HttpPost]
    public IActionResult Create([FromBody] Department department)
    {
        // [ApiController] đã tự kiểm tra ModelState rồi
        // Nếu vào được đây thì department chắc chắn hợp lệ
        return Ok(department);
    }
}

🔑Tóm tắt so sánh:

  • Minimal API: Validation lỗi → tự động 400 (không cần làm gì thêm).
  • MVC Controller (không có [ApiController]): Validation lỗi → ghi vào ModelState, bạn phải tự kiểm tra ModelState.IsValid.
  • MVC Controller với [ApiController]: Validation lỗi → tự động 400 giống Minimal API.

Các Data Annotation phổ biến cho validation

AttributeÝ nghĩa
[Required]Trường bắt buộc, không được null/rỗng
[StringLength(max)]Giới hạn độ dài chuỗi
[Range(min, max)]Giới hạn giá trị số trong khoảng
[EmailAddress]Kiểm tra định dạng email
[MinLength(n)]Độ dài tối thiểu
[RegularExpression(pattern)]Kiểm tra theo regex
[Compare("OtherProp")]Hai trường phải bằng nhau (ví dụ: xác nhận mật khẩu)
