MVC & Razor Pages — Filter Pipeline

Tìm hiểu toàn diện về filter pipeline trong MVC và Razor Pages: các loại filter — authorization, resource, exception, action, result và page filter — cùng cách khai báo scope và kiểm soát thứ tự thực thi.

1 Filter Pipeline là gì và tại sao cần dùng

Bài 1 · What and Why Filter Pipeline for MVC and Razor Pages

Ở chương trước chúng ta đã tìm hiểu filter pipeline cho Minimal API — đó là phần mở rộng của middleware pipeline chạy bên trong endpoint middleware, trước khi handler thực sự được thực thi. Filter pipeline cho MVC và Razor Pages có tư tưởng tương tự nhưng phong phú hơn nhiều bởi MVC là công nghệ lâu đời, có hệ sinh thái rất rộng.

Vị trí trong pipeline tổng thể

Khi một request đến, nó đi qua chuỗi middleware (Routing → Endpoint). Bên trong Endpoint middleware, với MVC thì filter pipeline chạy xung quanh action method; với Razor Pages thì chạy xung quanh page handler. Sau khi action/handler kết thúc, filter pipeline tiếp tục phần "sau" theo chiều ngược lại trước khi response được trả về.

Năm loại filter trong MVC

Loại filterInterfaceMô tả ngắn
AuthorizationIAuthorizationFilterChạy đầu tiên, kiểm tra xem request có được phép không.
ResourceIResourceFilterChạy sau authorization, trước model binding — dùng để short-circuit sớm.
ExceptionIExceptionFilterBắt ngoại lệ xảy ra trong action/page handler.
ActionIActionFilterChạy ngay trước và sau khi action method thực thi.
ResultIResultFilterChạy trước và sau khi IActionResult được thực thi (render).

Razor Pages không có Action Filter; thay vào đó có Page Filter (IPageFilter) hoạt động tương tự nhưng được thiết kế riêng cho page model.

Thứ tự thực thi tổng quát

ℹ️ Request đi vào: Authorization → Resource → (Model Binding) → Exception → Action (Before) → action method → Action (After) → Result (Before) → render → Result (After)
Response đi ra: theo chiều ngược lại của các bước "After".

🎯Tại sao cần filter pipeline? Filter cho phép tái sử dụng logic xuyên suốt nhiều action/page (authorization, validation, logging, caching, xử lý lỗi) mà không phải chép code vào từng method. Đây là cách áp dụng nguyên tắc DRY (Don't Repeat Yourself) trong tầng presentation.

2 Tạo và áp dụng filter

Bài 2 · Create and Apply a Filter

Mỗi filter trong MVC là một class implement một trong các interface filter. Để dùng filter như một attribute, class đó còn kế thừa thêm Attribute. Bài này dùng IResourceFilter làm ví dụ vì nó "trung tính" — không có mục đích chuyên biệt như authorization hay exception.

Bước 1: Implement interface

Tạo thư mục Filters/ trong project và thêm class sau:

Filters/WriteToConsoleResourceFilter.cscsharp
using Microsoft.AspNetCore.Mvc.Filters;

namespace WebApp.Filters
{
    public class WriteToConsoleResourceFilter : Attribute, IResourceFilter
    {
        public void OnResourceExecuting(ResourceExecutingContext context)
        {
            Console.WriteLine($"Executing {context.ActionDescriptor.DisplayName}");
        }

        public void OnResourceExecuted(ResourceExecutedContext context)
        {
            Console.WriteLine($"Executed {context.ActionDescriptor.DisplayName}");
        }
    }
}

💡Sync vs Async: IResourceFilter có hai phương thức đồng bộ. Nếu cần gọi API async bên trong filter, hãy implement IAsyncResourceFilter thay thế với phương thức OnResourceExecutionAsync(ResourceExecutingContext ctx, ResourceExecutionDelegate next).

Bước 2: Áp dụng lên action

Vì class kế thừa Attribute, ta dùng nó như một attribute C# bình thường:

Controllers/DepartmentsController.cscsharp
using WebApp.Filters;

public class DepartmentsController : Controller
{
    [HttpGet]
    [WriteToConsoleResourceFilter]   // áp dụng lên action
    public IActionResult Index()
    {
        return View();
    }
}

Các cách implement khác

CáchGhi chú
Implement interface trực tiếp (IResourceFilter)Sync — phù hợp khi không có I/O async
Implement async interface (IAsyncResourceFilter)Async — phù hợp khi cần gọi DB, HTTP…
Kế thừa base class (ResourceFilterAttribute)Đã sẵn thuộc tính Attribute, override các phương thức cần

3 Phạm vi áp dụng filter (Filter Scopes)

Bài 3 · Filter Scopes

Một filter có thể được áp dụng ở nhiều "cấp độ" (scope) khác nhau. Scope quyết định filter sẽ áp dụng cho bao nhiêu action/page.

Ba scope cho MVC

  1. Action level — đặt attribute trên action method: chỉ áp dụng cho action đó.
  2. Controller level — đặt attribute trên class controller: áp dụng cho mọi action trong controller.
  3. Global — đăng ký trong Program.cs: áp dụng cho toàn bộ action trong ứng dụng.

Đăng ký filter toàn cục

Program.cscsharp
using WebApp.Filters;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllersWithViews();
builder.Services.AddRazorPages().AddMvcOptions(options =>
{
    options.Filters.Add<WriteToConsoleResourceFilter>();
});

builder.Services.AddSingleton<IDepartmentsRepository, DepartmentsRepository>();
builder.Services.AddSingleton<IEmployeesRepository, EmployeesRepository>();

var app = builder.Build();

app.UseRouting();
app.UseEndpoints(endpoints =>
{
    endpoints.MapControllerRoute(
        name: "default",
        pattern: "{controller=Home}/{action=Index}/{id?}");
    endpoints.MapRazorPages();
});

app.Run();

Hai scope cho Razor Pages

Razor Pages không có Action-level scope (bạn không thể đặt filter trực tiếp lên một handler method riêng lẻ — Visual Studio sẽ gạch chân cảnh báo). Chỉ có hai scope:

  • Page level — đặt attribute trên class PageModel.
  • Global — đăng ký trong AddMvcOptions tương tự MVC.
Pages/Employees/Edit.cshtml.cscsharp
// Filter áp dụng ở page level
[WriteToConsoleResourceFilter]
public class EditModel : PageModel
{
    public void OnGet(int id) { /* ... */ }
    public IActionResult OnPost() { /* ... */ }
}

⚠️ Đặt attribute của MVC filter (IActionFilter) lên Razor Page handler sẽ không có tác dụng — bạn cần dùng IPageFilter hoặc filter có thể áp dụng cả hai (như IResourceFilter, IAuthorizationFilter).

4 Thứ tự thực thi (Order of Execution)

Bài 4 · Order of Execution

Khi cùng một loại filter được áp dụng ở nhiều scope (global, controller, action), thứ tự thực thi mặc định như sau:

PhaThứ tự (mặc định)
Trước action (Executing)Global → Controller → Action
Sau action (Executed)Action → Controller → Global

Đây là mô hình "bao bọc" (onion): filter ngoài cùng (Global) bọc controller, controller bọc action. Phần Executing đi từ ngoài vào trong; phần Executed đi từ trong ra ngoài.

Ghi đè thứ tự bằng IOrderedFilter

Implement thêm IOrderedFilter và thuộc tính Order (số nhỏ hơn = chạy trước) để điều chỉnh thứ tự:

Filters/WriteToConsoleResourceFilter.cscsharp
using Microsoft.AspNetCore.Mvc.Filters;

public class WriteToConsoleResourceFilter : Attribute, IResourceFilter, IOrderedFilter
{
    public string? Description { get; set; }

    // IOrderedFilter — số nhỏ hơn chạy trước
    public int Order { get; set; }

    public WriteToConsoleResourceFilter()
    {
        this.Description = "Global"; // giá trị mặc định
    }

    public void OnResourceExecuting(ResourceExecutingContext context)
    {
        Console.WriteLine($"Executing {Description} - {context.ActionDescriptor.DisplayName}");
    }

    public void OnResourceExecuted(ResourceExecutedContext context)
    {
        Console.WriteLine($"Executed {Description} - {context.ActionDescriptor.DisplayName}");
    }
}

Áp dụng cùng filter ở nhiều cấp với Order khác nhau

Controllers/DepartmentsController.cscsharp
[WriteToConsoleResourceFilter(Description = "Departments Controller")]
public class DepartmentsController : Controller
{
    [HttpGet]
    [WriteToConsoleResourceFilter(Description = "Index Method", Order = -1)]
    public IActionResult Index()
    {
        return View();
    }
}

Với cấu hình trên, filter có Order = -1 (action level) sẽ chạy trước filter của controller và global trong pha Executing, dù thông thường action chạy sau cùng.

💡Khi không cần thứ tự đặc biệt, hãy để mặc định (không implement IOrderedFilter). Chỉ dùng Order khi có yêu cầu cụ thể về ưu tiên.

5 Authorization Filter

Bài 5 · Authorization Filter

Authorization filter là loại filter chạy đầu tiên trong pipeline — trước cả model binding. Nếu kiểm tra thất bại, nó short-circuit toàn bộ pipeline: response được trả về ngay mà không thực thi action hay các filter khác.

Interface

Implement IAuthorizationFilter với một phương thức duy nhất OnAuthorization(AuthorizationFilterContext context). Để short-circuit, gán context.Result.

Filters/SpecialAuthorizationFilter.cscsharp
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.Filters;

namespace WebApp.Filters
{
    public class SpecialAuthorizationFilter : Attribute, IAuthorizationFilter
    {
        public void OnAuthorization(AuthorizationFilterContext context)
        {
            var user = context.HttpContext.User;

            if (user is null || user.Identity is null || !user.Identity.IsAuthenticated)
            {
                context.Result = new UnauthorizedResult();
                return;
            }

            if (!user.HasClaim(c => c.Type == "CustomClaim" && c.Value == "CustomValue"))
            {
                context.Result = new ForbidResult();
            }
        }
    }
}

Khi nào dùng Authorization Filter?

Trong thực tế, ASP.NET Core có hệ thống Identity và policy-based authorization rất hoàn chỉnh. Bạn nên ưu tiên tạo policy bằng [Authorize(Policy = "...")] thay vì tự viết authorization filter.

⚠️Khuyến nghị: Chỉ dùng IAuthorizationFilter khi có yêu cầu đặc biệt mà policy-based authorization không đáp ứng được. Với đa số trường hợp thông thường, hệ thống Authorization middleware + policy là lựa chọn tốt hơn.

Short-circuit trong Authorization Filter

Không giống các filter khác, authorization filter không gọi next(). Thay vào đó, framework tự quyết định tiếp tục hay không dựa trên việc context.Result có được gán hay không:

  • context.Result không được gán → request tiếp tục qua pipeline.
  • context.Result được gán → pipeline bị short-circuit, response trả về ngay.

6 Resource Filter

Bài 6 · Resource Filter

Resource filter chạy ngay sau authorization filter — vẫn rất sớm trong pipeline, trước model binding. Điều này có hai lợi ích chính:

  • Short-circuit sớm: nếu điều kiện không thỏa, từ chối request mà không tốn chi phí model binding, action execution.
  • Caching đầu ra (output caching thủ công): trong OnResourceExecuting kiểm tra cache; nếu có thì gán context.Result ngay; trong OnResourceExecuted lưu kết quả vào cache.

Ví dụ thực tế: EndpointExpiresFilter

Filter này cho phép đặt ngày hết hạn cho một endpoint. Khi quá hạn, trả về 400 Bad Request mà không cần thực thi action:

Filters/EndpointExpiresFilter.cscsharp
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.Filters;

namespace WebApp.Filters
{
    public class EndpointExpiresFilter : Attribute, IResourceFilter
    {
        public string? ExpiryDate { get; set; }

        public void OnResourceExecuting(ResourceExecutingContext context)
        {
            if (DateTime.TryParse(ExpiryDate, out DateTime expiryDate))
            {
                if (DateTime.Now > expiryDate)
                {
                    context.Result = new BadRequestResult(); // short-circuit
                }
            }
        }

        public void OnResourceExecuted(ResourceExecutedContext context)
        {
            // chạy sau khi action hoàn thành — có thể dùng để cache kết quả
        }
    }
}

Áp dụng lên action

Controllers/DepartmentsController.cscsharp
[HttpGet]
[EndpointExpiresFilter(ExpiryDate = "2028-01-18")]
public IActionResult Details(int id)
{
    var department = departmentsRepository.GetDepartmentById(id);
    return View(department);
}

Filter tích hợp sẵn: ConsumesAttribute

Một ví dụ resource filter có sẵn trong ASP.NET Core là [Consumes("application/json")]. Nó implement IResourceFilter và short-circuit nếu Content-Type của request không khớp với loại MIME được khai báo.

ℹ️IAsyncResourceFilter có một phương thức duy nhất: OnResourceExecutionAsync(ResourceExecutingContext ctx, ResourceExecutionDelegate next). Gọi await next() để tiếp tục pipeline; không gọi thì pipeline bị short-circuit.

7 Exception Filter

Bài 7 · Exception Filter

Exception filter bắt các ngoại lệ chưa được xử lý xảy ra trong quá trình model binding, action execution hoặc result execution. Nó chạy bên trong filter pipeline của MVC/Razor Pages.

So sánh với Exception Handling Middleware

Tiêu chíException FilterException Middleware
Phạm viChỉ MVC / Razor PagesToàn bộ request pipeline
Thông tin MVCBiết action/page nào gây lỗiKhông có
Áp dụng chọn lọcCó (action, controller, global)Không (toàn bộ)
Phù hợp khi nàoCần xử lý khác nhau giữa API endpoint và web UI; cần truy cập MVC contextXử lý lỗi chung cho mọi request

Implement bằng ExceptionFilterAttribute

Cách đơn giản nhất là kế thừa ExceptionFilterAttribute và override OnException:

Filters/HandleExceptionsFilter.cscsharp
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.Filters;

namespace WebApp.Filters
{
    public class HandleExceptionsFilter : ExceptionFilterAttribute
    {
        public override void OnException(ExceptionContext context)
        {
            base.OnException(context);

            var error = new ProblemDetails
            {
                Title = "An error occurred",
                Detail = context.Exception.ToString(),
                Status = 500
            };

            context.Result = new ObjectResult(error)
            {
                StatusCode = 500
            };

            // đánh dấu đã xử lý — tránh lan lên middleware bên ngoài
            context.ExceptionHandled = true;
        }
    }
}

Áp dụng lên action API

Controllers/DepartmentsController.cscsharp
[HttpGet]
[HandleExceptionsFilter]
public IActionResult GetDepartments()
{
    // nếu có ngoại lệ, HandleExceptionsFilter sẽ bắt và trả về ProblemDetails
    var departments = departmentsRepository.GetDepartments();
    return Json(departments);
}

💡Tình huống điển hình: Ứng dụng vừa phục vụ HTML (MVC views) vừa có Web API endpoint. Exception filter cho phép API endpoint trả về JSON ProblemDetails trong khi view endpoint vẫn redirect đến trang lỗi HTML — tất cả chọn lọc theo phạm vi filter.

ℹ️Nếu exception không được đánh dấu ExceptionHandled = true, nó sẽ tiếp tục lan ra ngoài và middleware xử lý lỗi (nếu có) sẽ bắt nó.

8 Action Filter

Bài 8 · Action Filter

Action filter chạy hai lần — trước và sau khi action method được thực thi. Vị trí này cho phép nó:

  • Truy cập các argument đã được model binding tạo ra (ActionExecutingContext.ActionArguments).
  • Truy cập kết quả của action (ActionExecutedContext.Result) và thay thế/sửa đổi trước khi result filter chạy.
  • Short-circuit bằng cách gán context.Result trong OnActionExecuting.

Ví dụ 1: EnsureDepartmentExistsFilter

Filter kiểm tra department có tồn tại không trước khi action chạy. Dùng ActionFilterAttribute làm base class cho tiện:

Filters/EnsureDepartmentExistsFilter.cscsharp
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.Filters;
using WebApp.Helpers;
using WebApp.Models;

namespace WebApp.Filters
{
    public class EnsureDepartmentExistsFilter : ActionFilterAttribute
    {
        public override void OnActionExecuting(ActionExecutingContext context)
        {
            base.OnActionExecuting(context);

            var departmentId = (int)context.ActionArguments["id"];

            var repo = context.HttpContext.RequestServices
                              .GetService<IDepartmentsRepository>();

            if (!repo.DepartmentExists(departmentId))
            {
                context.ModelState.AddModelError("id", "Department not found.");

                var result = new ViewResult { ViewName = "Error" };
                var controller = context.Controller as Controller;
                result.ViewData = controller.ViewData;
                result.ViewData.Model = ModelStateHelper.GetErrors(context.ModelState);

                context.Result = result; // short-circuit
            }
        }
    }
}

Ví dụ 2: EnsureValidModelStateFilter

Filter trích xuất logic kiểm tra ModelState ra khỏi action để có thể tái sử dụng:

Filters/EnsureValidModelStateFilter.cscsharp
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.Filters;
using WebApp.Helpers;

namespace WebApp.Filters
{
    public class EnsureValidModelStateFilter : ActionFilterAttribute
    {
        public override void OnActionExecuting(ActionExecutingContext context)
        {
            base.OnActionExecuting(context);

            if (!context.ModelState.IsValid)
            {
                var result = new ViewResult { ViewName = "Error" };
                var controller = context.Controller as Controller;
                result.ViewData = controller.ViewData;
                result.ViewData.Model = ModelStateHelper.GetErrors(context.ModelState);

                context.Result = result; // short-circuit nếu model không hợp lệ
            }
        }
    }
}

Áp dụng các filter lên controller

Controllers/DepartmentsController.cs (trích)csharp
[HttpGet]
[EndpointExpiresFilter(ExpiryDate = "2028-01-18")]
[EnsureDepartmentExistsFilter]
public IActionResult Details(int id)
{
    var department = departmentsRepository.GetDepartmentById(id);
    return View(department);
}

[HttpPost]
[EnsureValidModelStateFilter]
public IActionResult Edit(Department department)
{
    departmentsRepository.UpdateDepartment(department);
    return RedirectToAction(nameof(Index));
}

[HttpPost]
[EnsureValidModelStateFilter]
public IActionResult Create(Department department)
{
    departmentsRepository.AddDepartment(department);
    return RedirectToAction(nameof(Index));
}

🎯Use case phổ biến nhất: Tách logic validation (kiểm tra tồn tại resource, kiểm tra ModelState) ra khỏi action method giúp action method chỉ chứa business logic thuần túy — code sạch hơn, dễ test hơn.

ℹ️Để inject service vào filter, dùng context.HttpContext.RequestServices.GetService<T>() hoặc đăng ký filter qua DI (services.AddScoped<MyFilter>() và dùng [TypeFilter(typeof(MyFilter))] / [ServiceFilter(typeof(MyFilter))]).

9 Result Filter

Bài 9 · Result Filter

Result filter chạy trước và sau khi IActionResult được thực thi (tức là trước và sau khi Razor view được render, hoặc JSON được serialize và ghi vào response). Đây là loại filter cuối cùng trong pipeline trước khi response đến tay client.

Mục đích chính là sửa đổi response ngay trước khi nó được ghi ra — ví dụ: thêm HTTP header, thay đổi Content-Type, ghi log kết quả cuối cùng.

Interface

Filters/AddHeaderFilter.cscsharp
using Microsoft.AspNetCore.Mvc.Filters;

namespace WebApp.Filters
{
    public class AddHeaderFilter : Attribute, IResultFilter
    {
        public string? Name { get; set; }
        public string? Value { get; set; }

        public void OnResultExecuting(ResultExecutingContext context)
        {
            // chạy TRƯỚC khi result được render
            if (context.HttpContext.Response != null
                && this.Name is not null
                && this.Value is not null
                && !context.HttpContext.Response.Headers.ContainsKey(this.Name))
            {
                context.HttpContext.Response.Headers.Add(Name, Value);
            }
        }

        public void OnResultExecuted(ResultExecutedContext context)
        {
            // chạy SAU khi result được render
            // response đã ghi — không thể thay đổi body nữa
        }
    }
}

Áp dụng

Controllers/DepartmentsController.cs (trích)csharp
[HttpGet]
[AddHeaderFilter(Name = "X-Custom-Header", Value = "MyValue")]
public IActionResult Index()
{
    return View();
}

Filter tích hợp sẵn: ProducesAttribute

[Produces("application/json")] là một result filter tích hợp sẵn. Nó implement IResultFilter và trong OnResultExecuting sẽ thiết lập Content-Type của response theo loại MIME được khai báo, đồng thời lọc bỏ các formatter không phù hợp.

Phương thứcThời điểmDùng để
OnResultExecutingTrước khi render resultThêm header, thay đổi result, short-circuit
OnResultExecutedSau khi render xongLog, dọn dẹp — không thể thay đổi body đã ghi

💡Nếu muốn short-circuit trong result filter (không render kết quả), gán context.Cancel = true trong OnResultExecuting. Tuy nhiên dùng cẩn thận — client có thể nhận response rỗng.

10 Page Filter

Bài 10 · Page Filter

Page filter là phiên bản của action filter dành riêng cho Razor Pages. Nó implement IPageFilter và được gọi ba lần — nhiều hơn action filter một lần:

  1. OnPageHandlerSelected — sau khi page handler được chọn (ví dụ: OnGet, OnPost), trước model binding. Ít dùng trong thực tế.
  2. OnPageHandlerExecuting — sau model binding, trước khi handler method thực thi. Đây là nơi phổ biến nhất để đặt validation logic.
  3. OnPageHandlerExecuted — sau khi handler method thực thi xong.

Ví dụ 1: EnsureValidModelStatePageFilter

Filters/EnsureValidModelStatePageFilter.cscsharp
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.Filters;
using WebApp.Helpers;

namespace WebApp.Filters
{
    public class EnsureValidModelStatePageFilter : Attribute, IPageFilter
    {
        public void OnPageHandlerSelected(PageHandlerSelectedContext context)
        {
            // thường để trống
        }

        public void OnPageHandlerExecuting(PageHandlerExecutingContext context)
        {
            if (!context.ModelState.IsValid)
            {
                var errors = ModelStateHelper.GetErrors(context.ModelState);
                context.Result = new RedirectToPageResult("/Error", new { errors });
            }
        }

        public void OnPageHandlerExecuted(PageHandlerExecutedContext context)
        {
            // thường để trống
        }
    }
}

Ví dụ 2: EnsureEmployeeExistsPageFilter

Filter kiểm tra employee tồn tại. Lưu ý cách bỏ qua handler OnGet — ta không muốn kiểm tra khi lần đầu load trang:

Filters/EnsureEmployeeExistsPageFilter.cscsharp
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.Filters;
using WebApp.Models;
using WebApp.Pages.Employees;

namespace WebApp.Filters
{
    public class EnsureEmployeeExistsPageFilter : Attribute, IPageFilter
    {
        public void OnPageHandlerSelected(PageHandlerSelectedContext context) { }

        public void OnPageHandlerExecuting(PageHandlerExecutingContext context)
        {
            // bỏ qua khi là GET (chỉ load form, chưa có dữ liệu submit)
            if (context.HandlerMethod is not null
                && context.HandlerMethod.MethodInfo.Name == "OnGet") return;

            var repo = context.HttpContext.RequestServices
                              .GetService<IEmployeesRepository>();

            object? employeeId;
            if (context.HandlerArguments.ContainsKey("id"))
                employeeId = context.HandlerArguments["id"];
            else
                employeeId = ((EditModel)context.HandlerInstance).EmployeeViewModel.Employee.Id;

            if (!repo.EmployeeExists((int)employeeId))
            {
                context.ModelState.AddModelError("id", "Employee not found.");
                var errors = ModelStateHelper.GetErrors(context.ModelState);
                context.Result = new RedirectToPageResult("/Error", new { errors });
            }
        }

        public void OnPageHandlerExecuted(PageHandlerExecutedContext context) { }
    }
}

Áp dụng lên PageModel

Pages/Employees/Edit.cshtml.cscsharp
using Microsoft.AspNetCore.Mvc.RazorPages;
using WebApp.Filters;

namespace WebApp.Pages.Employees
{
    [EnsureValidModelStatePageFilter]
    [EnsureEmployeeExistsPageFilter]
    public class EditModel : PageModel
    {
        [BindProperty]
        public EmployeeViewModel? EmployeeViewModel { get; set; }

        public void OnGet(int id)
        {
            // load dữ liệu — EnsureEmployeeExistsPageFilter bỏ qua OnGet
            this.EmployeeViewModel = new EmployeeViewModel
            {
                Employee = employeesRepository.GetEmployeeById(id),
                Departments = departmentsRepository.GetDepartments()
            };
        }

        public IActionResult OnPost()
        {
            // EnsureValidModelStatePageFilter kiểm tra ModelState trước
            // EnsureEmployeeExistsPageFilter kiểm tra employee tồn tại trước
            employeesRepository.UpdateEmployee(EmployeeViewModel.Employee);
            return RedirectToPage("Index");
        }

        public IActionResult OnPostDeleteEmplopyee(int id)
        {
            var employee = employeesRepository.GetEmployeeById(id);
            employeesRepository.DeleteEmployee(employee);
            return RedirectToPage("Index");
        }
    }
}

So sánh Page Filter vs Action Filter

Tiêu chíAction Filter (IActionFilter)Page Filter (IPageFilter)
Dùng choMVC Controller actionRazor Pages page handler
Số lần gọi2 (Executing / Executed)3 (Selected / Executing / Executed)
Truy cập handler argumentsActionExecutingContext.ActionArgumentsPageHandlerExecutingContext.HandlerArguments
Có thể áp dụng chéo không?Không áp dụng được cho Razor PagesKhông áp dụng được cho MVC Controller

🎯Tóm tắt chương: Filter pipeline MVC/Razor Pages cung cấp 5 loại hook (Authorization → Resource → Exception → Action/Page → Result) để inject logic tái sử dụng ở đúng thời điểm cần thiết. Hiểu scope và thứ tự thực thi là chìa khóa để thiết kế filter hiệu quả.
