CHƯƠNG 02

HTTP Protocol & HTTP Context

Tìm hiểu sâu về giao thức HTTP — cú pháp request/response, các HTTP method (GET, POST, PUT, DELETE), query string, headers, status codes — và cách ASP.NET Core ánh xạ tất cả vào đối tượng HttpContext.

1 Cú pháp HTTP Request

Bài 1 · HTTP Request Syntax

Mọi giao tiếp giữa client và server trong ứng dụng web đều bắt đầu bằng một HTTP Request. Request này được gửi từ client (trình duyệt, mobile app, Postman…) đến server và mang theo đầy đủ thông tin cần thiết để server biết phải làm gì.

Mục đích của HTTP Request

HTTP Request có nhiệm vụ cung cấp yêu cầu (requirements) từ client đến server — xác định rõ client muốn làm gì, trên tài nguyên nào, và cung cấp dữ liệu gì kèm theo nếu cần.

Cấu trúc một HTTP Request

Một HTTP Request dạng văn bản thuần túy có ba phần:

HTTP Request (raw text)http
GET /employees HTTP/1.1
Host: localhost:5000
Accept: application/json
Authorization: Bearer <token>

(body — thường rỗng với GET)
PhầnVị tríÝ nghĩa
Request LineDòng đầu tiênGồm: METHOD URL HTTP-VERSION — ví dụ GET /employees HTTP/1.1.
HeadersTừ dòng 2 trở điCác cặp key-value cung cấp thông tin bổ sung (host, content-type, authorization…).
BodySau dòng trắngDữ liệu gửi kèm (dùng với POST, PUT). Với GET/DELETE thường để trống.

Thành phần Request Line

  • Method (động từ): loại hành động client muốn thực hiện — GET, POST, PUT, DELETE…
  • URL (path): vị trí tài nguyên trên server — ví dụ /employees, /employees?id=1.
  • HTTP Version: phiên bản giao thức — thường là HTTP/1.1 hoặc HTTP/2.

🎯Kết hợp Method + URL = định danh endpoint. Đây là cơ sở của routing: server căn cứ vào method và path để biết endpoint nào cần xử lý request.

2 HTTP Request trong HttpContext

Bài 2 · HTTP Request in HttpContext

Khi một HTTP request đến, Kestrel (web server tích hợp của ASP.NET Core) phân tích chuỗi văn bản HTTP thô và chuyển đổi nó thành đối tượng HttpContext. Đây là bước đầu tiên trong pipeline xử lý request.

HttpContext là gì?

HttpContext là đối tượng trung tâm bao gồm toàn bộ thông tin về một chu kỳ request/response:

  • context.Request — đối tượng HttpRequest chứa mọi thông tin từ phía client.
  • context.Response — đối tượng HttpResponse để xây dựng phản hồi gửi về client.
  • context.User — thông tin xác thực người dùng.
  • context.Items — dictionary chia sẻ dữ liệu giữa các middleware.

Truy cập thông tin Request qua HttpContext

Trong ví dụ dưới đây, app.Run() đăng ký một terminal middleware nhận HttpContext và đọc các thuộc tính của request:

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.Run(async (HttpContext context) =>
{
    // Đọc HTTP method (GET, POST, ...)
    await context.Response.WriteAsync($"Method: {context.Request.Method}\r\n");

    // Đọc đường dẫn URL
    await context.Response.WriteAsync($"Path: {context.Request.Path}\r\n");

    // Duyệt tất cả headers
    await context.Response.WriteAsync("\r\nHeaders:\r\n");
    foreach (var key in context.Request.Headers.Keys)
    {
        await context.Response.WriteAsync($"{key}: {context.Request.Headers[key]}\r\n");
    }
});

app.Run();
Thuộc tínhKiểu dữ liệuMô tả
Request.MethodstringHTTP method: "GET", "POST", "PUT", "DELETE"…
Request.PathPathStringĐường dẫn URL, ví dụ /employees.
Request.QueryStringQueryStringChuỗi query nguyên văn, ví dụ ?id=1.
Request.QueryIQueryCollectionDictionary các tham số query string.
Request.HeadersIHeaderDictionaryTập hợp các HTTP header.
Request.BodyStreamNội dung body của request.

ℹ️app.Run(delegate) đăng ký một terminal middleware — mọi request đều đi qua đây và không được chuyển tiếp tiếp. Đây là cách tiếp cận thủ công nhất để hiểu cách HttpContext hoạt động trước khi học routing cao cấp hơn.

3 HTTP Method GET

Bài 3 · HTTP Method GET

GET là HTTP method phổ biến nhất. Mục đích của nó là lấy (retrieve) thông tin từ server — không tạo mới, không sửa, không xóa dữ liệu.

Đặc điểm của GET

  • Không có body (hoặc body bị bỏ qua).
  • Tham số được truyền qua URL (path hoặc query string).
  • An toàn và idempotent — gọi nhiều lần cho cùng một kết quả.
  • Có thể được cache bởi trình duyệt và proxy.

Routing thủ công với method + path

Trong giai đoạn này của khóa học, routing được thực hiện thủ công bằng cách kiểm tra context.Request.Method và context.Request.Path:

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.Run(async (HttpContext context) =>
{
    if (context.Request.Method == "GET")
    {
        if (context.Request.Path.StartsWithSegments("/"))
        {
            await context.Response.WriteAsync("Trang chủ — GET /\r\n");
        }
        else if (context.Request.Path.StartsWithSegments("/employees"))
        {
            // Trả về danh sách tất cả nhân viên
            var employees = EmployeesRepository.GetEmployees();
            foreach (var emp in employees)
            {
                await context.Response.WriteAsync($"{emp.Name}: {emp.Position}\r\n");
            }
        }
    }
});

app.Run();

// --- Repository ---
static class EmployeesRepository
{
    private static List<Employee> employees = new()
    {
        new Employee(1, "John Doe",   "Engineer",   60000),
        new Employee(2, "Jane Smith", "Manager",    75000),
        new Employee(3, "Sam Brown",  "Technician", 50000)
    };

    public static List<Employee> GetEmployees() => employees;
}

public class Employee
{
    public int    Id       { get; set; }
    public string Name     { get; set; }
    public string Position { get; set; }
    public double Salary   { get; set; }

    public Employee(int id, string name, string position, double salary)
    {
        Id = id; Name = name; Position = position; Salary = salary;
    }
}

💡Path.StartsWithSegments("/employees") so sánh chính xác theo segment — /employees khớp, nhưng /employees-old thì không. Đây là cách an toàn hơn so với Path.Value.StartsWith("/employees").

4 HTTP Method POST

Bài 4 · HTTP Method POST

POST được dùng để tạo mới tài nguyên trên server. Dữ liệu cần tạo mới được client gửi kèm trong body của request, thường ở định dạng JSON.

Đặc điểm của POST

  • Có body chứa dữ liệu tạo mới (JSON, form-data, XML…).
  • Không idempotent — gọi nhiều lần sẽ tạo nhiều bản ghi.
  • Phản hồi thành công thường trả về status code 201 Created.

Đọc body JSON từ Request

context.Request.Body là một Stream, cần đọc bằng StreamReader rồi deserialize:

Program.cs — xử lý POST /employeescsharp
else if (context.Request.Method == "POST")
{
    if (context.Request.Path.StartsWithSegments("/employees"))
    {
        // Đọc body dưới dạng chuỗi
        using var reader = new StreamReader(context.Request.Body);
        var body = await reader.ReadToEndAsync();

        // Deserialize JSON thành đối tượng Employee
        var employee = JsonSerializer.Deserialize<Employee>(body);

        EmployeesRepository.AddEmployee(employee);

        context.Response.StatusCode = 201; // Created
        await context.Response.WriteAsync("Employee added successfully.");
    }
}

Gọi POST bằng Postman / curl

Terminal (curl)bash
curl -X POST http://localhost:5000/employees \
     -H "Content-Type: application/json" \
     -d '{"id":4,"name":"Alice","position":"Designer","salary":65000}'

⚠️JsonSerializer.Deserialize<T> mặc định phân biệt chữ hoa/thường với tên property. Nếu JSON gửi lên là "name" nhưng property C# là Name, cần thêm option JsonSerializerOptions { PropertyNameCaseInsensitive = true }.

5 HTTP Method PUT

Bài 5 · HTTP Method PUT

PUT được dùng để cập nhật toàn bộ tài nguyên đã tồn tại trên server. Giống POST, dữ liệu cập nhật được gửi trong body của request.

GET vs POST vs PUT — so sánh

MethodMục đíchBody?Idempotent?
GETLấy dữ liệuKhôngCó
POSTTạo mới tài nguyênCóKhông
PUTCập nhật toàn bộ tài nguyênCóCó

Xử lý PUT — cập nhật nhân viên

Program.cs — xử lý PUT /employeescsharp
else if (context.Request.Method == "PUT")
{
    if (context.Request.Path.StartsWithSegments("/employees"))
    {
        using var reader = new StreamReader(context.Request.Body);
        var body = await reader.ReadToEndAsync();
        var employee = JsonSerializer.Deserialize<Employee>(body);

        var result = EmployeesRepository.UpdateEmployee(employee);
        if (result)
        {
            await context.Response.WriteAsync("Employee updated successfully.");
        }
        else
        {
            await context.Response.WriteAsync("Employee not found.");
        }
    }
}

// --- Thêm phương thức UpdateEmployee vào repository ---
public static bool UpdateEmployee(Employee? employee)
{
    if (employee is not null)
    {
        var emp = employees.FirstOrDefault(x => x.Id == employee.Id);
        if (emp is not null)
        {
            emp.Name     = employee.Name;
            emp.Position = employee.Position;
            emp.Salary   = employee.Salary;
            return true;
        }
    }
    return false;
}

ℹ️Trong thực tế, cũng có PATCH để cập nhật một phần tài nguyên (chỉ cập nhật những trường được gửi). PUT thay thế toàn bộ, còn PATCH chỉ thay thế một phần.

6 Query String

Bài 6 · Query String

Query string là phần của URL nằm sau dấu ?, chứa các cặp key-value để truyền tham số bổ sung đến server mà không cần thay đổi path.

Cú pháp query string

URL ví dụhttp
GET /employees?id=1 HTTP/1.1
GET /employees?search=john&page=2&pageSize=10 HTTP/1.1
  • Bắt đầu bằng dấu ? sau phần path.
  • Mỗi tham số là cặp key=value.
  • Nhiều tham số cách nhau bằng dấu &.

Đọc query string trong ASP.NET Core

Program.cs — đọc query stringcsharp
// Kiểm tra xem query string có chứa key "id" không
if (context.Request.Query.ContainsKey("id"))
{
    var idStr = context.Request.Query["id"];          // trả về StringValues
    if (int.TryParse(idStr, out int employeeId))
    {
        await context.Response.WriteAsync($"Employee ID requested: {employeeId}");
    }
}

// Lấy tham số search (nếu không tồn tại, trả về StringValues.Empty)
var search = context.Request.Query["search"].ToString();

Ứng dụng thực tế

Query string thường được dùng để:

  • Truyền ID để lấy/xóa tài nguyên cụ thể: GET /employees?id=1, DELETE /employees?id=1.
  • Tìm kiếm và lọc: GET /employees?search=john&department=IT.
  • Phân trang: GET /employees?page=2&pageSize=10.
  • Sắp xếp: GET /employees?sortBy=salary&order=desc.

💡context.Request.Query["id"] trả về kiểu StringValues (có thể chứa nhiều giá trị cho cùng key). Luôn dùng int.TryParse() hoặc ToString() trước khi sử dụng để tránh lỗi.

7 HTTP Method DELETE

Bài 7 · HTTP Method DELETE

DELETE được dùng để xóa tài nguyên trên server. Vì cần biết xóa tài nguyên nào, thông tin định danh (thường là ID) được truyền qua query string hoặc URL path.

Tại sao cần query string với DELETE?

Nếu chỉ dùng DELETE /employees, server không biết xóa nhân viên nào. Cần truyền ID:

  • DELETE /employees?id=3 — qua query string (cách dùng trong bài này).
  • DELETE /employees/3 — qua URL path (route parameter, dùng khi học Minimal API/MVC).

Xử lý DELETE với query string

Program.cs — xử lý DELETE /employees?id=Ncsharp
else if (context.Request.Method == "DELETE")
{
    if (context.Request.Path.StartsWithSegments("/employees"))
    {
        if (context.Request.Query.ContainsKey("id"))
        {
            var id = context.Request.Query["id"];
            if (int.TryParse(id, out int employeeId))
            {
                var result = EmployeesRepository.DeleteEmployee(employeeId);

                if (result)
                {
                    await context.Response.WriteAsync("Employee deleted successfully.");
                }
                else
                {
                    await context.Response.WriteAsync("Employee not found.");
                }
            }
        }
    }
}

// --- Thêm DeleteEmployee vào repository ---
public static bool DeleteEmployee(int id)
{
    var employee = employees.FirstOrDefault(x => x.Id == id);
    if (employee is not null)
    {
        employees.Remove(employee);
        return true;
    }
    return false;
}

⚠️Thao tác DELETE là idempotent theo chuẩn HTTP — gọi lần đầu xóa, gọi lần sau trả về 404 (nhưng server state không thay đổi thêm). Tuy nhiên không nên dùng GET để xóa dữ liệu vì GET không được phép có side effect.

8 HTTP Request Headers

Bài 8 · HTTP Request Headers

HTTP Request Headers cung cấp thông tin bổ sung từ client đến server, giúp server xử lý request đúng cách. Headers được gửi dưới dạng các cặp key-value nằm sau request line.

Tại sao cần headers?

  • URL có giới hạn độ dài — không thể nhét tất cả thông tin vào URL.
  • Thông tin nhạy cảm — không nên đặt token xác thực lên URL (sẽ bị ghi vào log, lịch sử trình duyệt…).
  • Metadata kỹ thuật — content-type, encoding, ngôn ngữ mong muốn… không phải là dữ liệu nghiệp vụ.

Các header phổ biến

HeaderVí dụ giá trịÝ nghĩa
Hostlocalhost:5000Domain đích của request.
Content-Typeapplication/jsonĐịnh dạng của body request.
Acceptapplication/jsonĐịnh dạng response mà client chấp nhận.
AuthorizationBearer <token>Thông tin xác thực (token, API key…).
User-AgentMozilla/5.0 …Thông tin về trình duyệt/client.
CookiesessionId=abc123Cookie gửi kèm.

Đọc và sử dụng headers trong ASP.NET Core

Program.cs — kiểm tra header Authorizationcsharp
else if (context.Request.Method == "DELETE")
{
    if (context.Request.Path.StartsWithSegments("/employees"))
    {
        if (context.Request.Query.ContainsKey("id"))
        {
            var id = context.Request.Query["id"];
            if (int.TryParse(id, out int employeeId))
            {
                // Kiểm tra header Authorization
                if (context.Request.Headers["Authorization"] == "frank")
                {
                    var result = EmployeesRepository.DeleteEmployee(employeeId);
                    if (result)
                        await context.Response.WriteAsync("Employee deleted successfully.");
                    else
                        await context.Response.WriteAsync("Employee not found.");
                }
                else
                {
                    await context.Response.WriteAsync("You are not authorized to delete.");
                }
            }
        }
    }
}

ℹ️Đây là ví dụ minh họa đơn giản. Trong production, authorization phải dùng JWT / Cookie Authentication của ASP.NET Core — không so sánh header thủ công như vậy.

9 Tái cấu trúc routing code

Bài 9 · Refactor our routing code

Trong các bài trước, code routing kiểm tra method trước, path sau. Nhưng routing chuẩn trong thực tế hoạt động theo thứ tự path trước, method sau — tức là xác định tài nguyên trước, rồi mới xác định hành động cần thực hiện trên tài nguyên đó.

Tại sao phải đổi thứ tự?

Hãy tưởng tượng ứng dụng có nhiều endpoint:

  • GET /employees — lấy danh sách nhân viên
  • POST /employees — thêm nhân viên
  • GET /positions — lấy danh sách chức vụ

Routing đúng: xác định request đi đến /employees hay /positions trước, rồi xét method để chọn handler cụ thể.

Code sau khi refactor

Program.cs — routing theo path rồi methodcsharp
app.Run(async (HttpContext context) =>
{
    if (context.Request.Path.StartsWithSegments("/"))
    {
        // Trang chủ — hiển thị thông tin request
        await context.Response.WriteAsync($"Method: {context.Request.Method}\r\n");
        await context.Response.WriteAsync($"Path:   {context.Request.Path}\r\n");
    }
    else if (context.Request.Path.StartsWithSegments("/employees"))
    {
        // --- Branch theo method ---
        if (context.Request.Method == "GET")
        {
            var employees = EmployeesRepository.GetEmployees();
            foreach (var emp in employees)
                await context.Response.WriteAsync($"{emp.Name}: {emp.Position}\r\n");
        }
        else if (context.Request.Method == "POST")
        {
            using var reader = new StreamReader(context.Request.Body);
            var body = await reader.ReadToEndAsync();
            var employee = JsonSerializer.Deserialize<Employee>(body);
            EmployeesRepository.AddEmployee(employee);
        }
        else if (context.Request.Method == "PUT")
        {
            using var reader = new StreamReader(context.Request.Body);
            var body = await reader.ReadToEndAsync();
            var employee = JsonSerializer.Deserialize<Employee>(body);
            var result = EmployeesRepository.UpdateEmployee(employee);
            await context.Response.WriteAsync(result ? "Updated." : "Not found.");
        }
        else if (context.Request.Method == "DELETE")
        {
            if (context.Request.Query.ContainsKey("id") &&
                int.TryParse(context.Request.Query["id"], out int id))
            {
                if (context.Request.Headers["Authorization"] == "frank")
                {
                    var result = EmployeesRepository.DeleteEmployee(id);
                    await context.Response.WriteAsync(result ? "Deleted." : "Not found.");
                }
                else
                {
                    await context.Response.WriteAsync("Unauthorized.");
                }
            }
        }
    }
});

🎯Đây chính là mô hình tư duy mà Minimal API và MVC routing sử dụng: MapGet("/employees", handler), MapPost("/employees", handler)… — ASP.NET Core routing framework đã tự động hóa chính xác logic path-first, method-second này.

10 Cú pháp HTTP Response

Bài 10 · HTTP Response Syntax

Sau khi xử lý request, server gửi lại một HTTP Response cho client. Response có mục đích ngược lại với request: cung cấp thông tin từ server đến client để client (trình duyệt) có thể hiển thị hay xử lý dữ liệu nhận được.

Cấu trúc HTTP Response

HTTP Response (raw text)http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 42

<ul><li>John Doe: Engineer</li></ul>
PhầnVị tríÝ nghĩa
Status LineDòng đầu tiênGồm: HTTP-VERSION STATUS-CODE REASON-PHRASE, ví dụ HTTP/1.1 200 OK.
Response HeadersTừ dòng 2Metadata về response: content-type, content-length, cache-control, set-cookie…
BodySau dòng trắngNội dung thực tế: HTML, JSON, hình ảnh, file… Có thể rỗng (ví dụ 204 No Content).

Status Line — ba thành phần

  • HTTP Version: HTTP/1.1 hoặc HTTP/2.
  • Status Code: số nguyên 3 chữ số — ví dụ 200, 404, 500.
  • Reason Phrase: mô tả ngắn bằng tiếng Anh — OK, Not Found, Internal Server Error.

ℹ️Trình duyệt đọc Content-Type header để biết cách render body. Nếu là text/html, trình duyệt render HTML. Nếu là application/json, trình duyệt hiển thị JSON thô.

11 HTTP Response trong HttpContext

Bài 11 · HTTP Response in HttpContext

Đối tượng context.Response (kiểu HttpResponse) là nơi ta xây dựng HTTP response sẽ gửi về client. ASP.NET Core tự động chuyển đổi các thiết lập trên đối tượng này thành chuỗi HTTP response hợp lệ.

Các thuộc tính quan trọng của HttpResponse

Thuộc tính / Phương thứcKiểuÝ nghĩa
Response.StatusCodeintHTTP status code trả về, ví dụ 200, 201, 404.
Response.HeadersIHeaderDictionaryResponse headers — thiết lập Content-Type, Cache-Control…
Response.ContentTypestring?Shortcut cho header Content-Type.
Response.BodyStreamStream để ghi nội dung body.
Response.WriteAsync()TaskGhi chuỗi vào body response.
Response.Redirect()voidChuyển hướng client đến URL khác (trả về 302).

Thiết lập Content-Type và ghi body HTML

Program.cs — response HTML với status codecsharp
if (context.Request.Method == "GET")
{
    var employees = EmployeesRepository.GetEmployees();

    // Thiết lập Content-Type để trình duyệt render HTML
    context.Response.Headers["Content-Type"] = "text/html";
    // hoặc: context.Response.ContentType = "text/html";

    await context.Response.WriteAsync("<ul>");
    foreach (var emp in employees)
    {
        await context.Response.WriteAsync(

12 HTTP Response Status Codes

Bài 12 · HTTP Response Status Codes

Status code là con số 3 chữ số trong response, cho client biết kết quả xử lý request. Có 5 nhóm status code:

NhómPhạm viÝ nghĩa
1xx100–199Informational — thông tin, hiếm dùng trực tiếp.
2xx200–299Success — request thành công.
3xx300–399Redirection — client cần thực hiện thêm hành động.
4xx400–499Client Error — lỗi từ phía client.
5xx500–599Server Error — lỗi từ phía server.

Các status code thường dùng

CodeTênKhi nào dùng
200OKRequest thành công, có body trả về (GET, PUT thành công…).
201CreatedTạo mới tài nguyên thành công (POST).
204No ContentThành công nhưng không có body trả về (PUT/DELETE thành công).
302FoundRedirect tạm thời.
400Bad RequestDữ liệu client gửi không hợp lệ.
401UnauthorizedChưa xác thực hoặc xác thực sai.
403ForbiddenĐã xác thực nhưng không có quyền.
404Not FoundTài nguyên hoặc endpoint không tồn tại.
500Internal Server ErrorLỗi không xác định phía server.

13 Bài tập 1 — Hiển thị nhân viên cụ thể

Bài 13 · Assignment 1 — Display a particular employee

Bài tập này yêu cầu bạn mở rộng ứng dụng hiện tại để xử lý thêm một trường hợp mới trong GET /employees: ngoài việc liệt kê tất cả nhân viên, còn hỗ trợ lấy thông tin một nhân viên cụ thể theo ID truyền qua query string.

Yêu cầu

  • GET /employees → trả về danh sách tất cả nhân viên (đã có).
  • GET /employees?id=1 → trả về thông tin chi tiết của nhân viên có ID = 1.
  • Nếu không tìm thấy nhân viên → trả về 404 Not Found.

Gợi ý

  1. Bên trong branch GET /employees, kiểm tra xem query string có key "id" không.
  2. Nếu có → lấy employee theo ID và hiển thị chi tiết.
  3. Nếu không có → hiển thị danh sách như cũ.
  4. Thêm phương thức GetEmployeeById(int id) vào repository.

💡Hãy tự viết code trước khi xem đáp án ở bài 14. Đây là cơ hội tốt để luyện tập đọc/ghi HttpContext và xử lý query string.

Thêm GetEmployeeById vào repository

Program.cs — thêm phương thức GetEmployeeByIdcsharp
static class EmployeesRepository
{
    private static List<Employee> employees = new()
    {
        new Employee(1, "John Doe",   "Engineer",   60000),
        new Employee(2, "Jane Smith", "Manager",    75000),
        new Employee(3, "Sam Brown",  "Technician", 50000)
    };

    public static List<Employee> GetEmployees() => employees;

    // Phương thức mới: lấy nhân viên theo ID
    public static Employee? GetEmployeeById(int id)
        => employees.FirstOrDefault(x => x.Id == id);

    public static void AddEmployee(Employee? employee)
    {
        if (employee is not null) employees.Add(employee);
    }

    public static bool UpdateEmployee(Employee? employee)
    {
        if (employee is not null)
        {
            var emp = employees.FirstOrDefault(x => x.Id == employee.Id);
            if (emp is not null)
            {
                emp.Name = employee.Name; emp.Position = employee.Position;
                emp.Salary = employee.Salary;
                return true;
            }
        }
        return false;
    }

    public static bool DeleteEmployee(int id)
    {
        var emp = employees.FirstOrDefault(x => x.Id == id);
        if (emp is not null) { employees.Remove(emp); return true; }
        return false;
    }
}

Cập nhật handler GET /employees

Program.cs — GET phân nhánh theo query string idcsharp
if (context.Request.Method == "GET")
{
    context.Response.ContentType = "text/html";

    if (context.Request.Query.ContainsKey("id"))
    {
        // --- Lấy nhân viên cụ thể theo ID ---
        var idStr = context.Request.Query["id"];
        if (int.TryParse(idStr, out int employeeId))
        {
            var employee = EmployeesRepository.GetEmployeeById(employeeId);
            if (employee is not null)
            {
                await context.Response.WriteAsync($"Name: {employee.Name}<br/>");
                await context.Response.WriteAsync($"Position: {employee.Position}<br/>");
                await context.Response.WriteAsync($"Salary: {employee.Salary}<br/>");
            }
            else
            {
                context.Response.StatusCode = 404;
                await context.Response.WriteAsync("Employee not found.");
            }
        }
    }
    else
    {
        // --- Lấy tất cả nhân viên ---
        var employees = EmployeesRepository.GetEmployees();
        await context.Response.WriteAsync("<ul>");
        foreach (var emp in employees)
        {
            await context.Response.WriteAsync(
                $"<li><b>{emp.Name}</b>: {emp.Position}</li>");
        }
        await context.Response.WriteAsync("</ul>");
        context.Response.StatusCode = 200;
    }
}
