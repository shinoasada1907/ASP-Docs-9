MVC Controllers — Routing

Tìm hiểu MVC Controller là gì, khi nào nên dùng, và hai chiến lược định tuyến: attribute routing (khai báo trực tiếp trên action) và conventional routing (dùng template chung cho toàn ứng dụng).

1 Controllers là gì & tại sao dùng

Bài 1 · What and Why Controllers

Trước đây chúng ta đã học cách tạo endpoint bằng Minimal API — dùng các lambda expression ngắn gọn như app.MapGet(). Bắt đầu từ chương này, ta học thêm một cách khác: MVC Controllers.

Controller là gì?

Một controller là một class C# đóng vai trò như một container nhóm các endpoint có liên quan lại với nhau. Thay vì viết từng lambda rời rạc, ta tổ chức chúng thành các action method (phương thức hành động) bên trong một class có cấu trúc rõ ràng.

Khái niệmMinimal APIMVC Controller
Endpoint handlerLambda / delegateAction method (public method trong controller)
Tổ chứcRải rác trong Program.csNhóm theo class, theo chủ đề
RoutingMapGet/MapPost…Attribute routing hoặc Conventional routing

Action method

Bất kỳ public method nào trong một controller đều mặc định được coi là một action method — tức là một endpoint handler. Nếu bạn muốn một method không phải là endpoint, hãy đánh dấu nó bằng attribute [NonAction].

Controllers/DepartmentsController.cscsharp
using Microsoft.AspNetCore.Mvc;

namespace WebApp.Controllers
{
    public class DepartmentsController
    {
        // Public method → action method (endpoint handler)
        public string GetDepartments()
        {
            return "These are the departments.";
        }

        public string GetDepartmentById(int id)
        {
            return $"Department info for id: {id}";
        }

        // Không phải endpoint — chỉ là helper nội bộ
        [NonAction]
        public string BuildMessage() => "helper";
    }
}

Tại sao dùng MVC Controllers?

  1. Lịch sử & phổ biến: MVC là kiến trúc mặc định từ những phiên bản đầu của ASP.NET. Rất nhiều dự án hiện tại vẫn dùng MVC, nên đây là kiến thức bắt buộc.
  2. Hỗ trợ giao diện người dùng (HTML): Controllers kết hợp với Razor Views giúp framework tự sinh HTML — không cần "hard-code" HTML thủ công. Minimal API cũng có thể trả về HTML nhưng phức tạp hơn nhiều.
  3. Cấu trúc rõ ràng: Controller nhóm các endpoint liên quan, giúp dự án lớn dễ duy trì hơn. Minimal API linh hoạt hơn nhưng ít có quy ước hơn.

🎯Khi nào nên chọn? Dùng MVC Controller khi dự án cần sinh giao diện HTML phía server (web app truyền thống), hoặc khi đội nhóm muốn cấu trúc kiểu OOP rõ ràng. Dùng Minimal API khi ưu tiên sự tối giản và hiệu năng (microservice, API-only).

2 Attribute Routing

Bài 2 · Attribute Routing

Attribute routing là cách định tuyến bằng cách khai báo route template trực tiếp trên từng action method (hoặc trên class controller) thông qua các attribute như [Route], [HttpGet], [HttpPost]…

Cách ASP.NET Core nhận diện một Controller class

Middleware routing cần biết class nào là controller. Có hai cách:

  • Theo quy ước (convention): tên class kết thúc bằng hậu tố Controller (ví dụ: DepartmentsController).
  • Bằng attribute: đặt attribute [Controller] lên class — ít phổ biến hơn.

Đăng ký services và middleware

Để sử dụng controllers, cần đăng ký services và middleware trong Program.cs:

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);

// Đăng ký các services cần thiết cho controllers
builder.Services.AddControllers();

var app = builder.Build();

// Thêm routing middleware vào pipeline
app.UseRouting();

// Định nghĩa endpoints: quét toàn bộ controllers trong assembly
app.UseEndpoints(endpoints =>
{
    endpoints.MapControllers();
});

app.Run();

ℹ️AddControllers vs AddControllersWithViews: AddControllers() chỉ thêm dịch vụ cho API controller (không có View). AddControllersWithViews() thêm thêm các dịch vụ cần thiết để render Razor View — dùng khi xây dựng web app có giao diện HTML.

💡Top-level route registration: Thay vì dùng UseRouting + UseEndpoints, từ .NET 6 trở đi có thể viết gọn hơn: app.MapControllers(); — cách này tương đương và được khuyến nghị trong dự án mới.

Khai báo route trên action bằng attribute

Sau khi đăng ký, routing middleware đã biết các class controller, nhưng vẫn chưa biết URL nào map vào action nào — ta phải cung cấp thêm HTTP method và route template bằng attribute.

Controllers/DepartmentsController.cscsharp
using Microsoft.AspNetCore.Mvc;

namespace WebApp.Controllers
{
    public class DepartmentsController
    {
        // Cách 1: tách riêng [Route] và [HttpGet]
        [HttpGet]
        [Route("departments")]
        public string GetDepartments()
        {
            return "These are the departments.";
        }

        // Cách 2: kết hợp template vào [HttpGet] (ngắn gọn hơn)
        [HttpGet("departments/{id}")]
        public string GetDepartmentById(int id)
        {
            return $"Department info: {id}";
        }
    }
}

ℹ️Nếu chỉ đặt [Route("departments")] mà không có attribute HTTP method ([HttpGet], [HttpPost]…), thì action đó có thể nhận bất kỳ HTTP method nào — không bị giới hạn.

Route prefix ở cấp Controller class

Attribute [Route] đặt trên class sẽ trở thành prefix cho tất cả route bên trong controller đó:

Controllers/DepartmentsController.cscsharp
using Microsoft.AspNetCore.Mvc;

namespace WebApp.Controllers
{
    [Route("/api")]          // prefix cho toàn bộ controller
    public class DepartmentsController
    {
        [HttpGet("departments")]
        public string GetDepartments()
        {
            return "These are the departments.";
        }
        // → URL: GET /api/departments

        [HttpGet("departments/{id}")]
        public string GetDepartmentById(int id)
        {
            return $"Department info: {id}";
        }
        // → URL: GET /api/departments/100
    }
}

Token [controller] và [action]

Để tránh lặp tên controller/action khi viết route template, ASP.NET Core hỗ trợ các token thay thế:

  • [controller] — được thay bằng tên controller (bỏ hậu tố Controller), ví dụ Departments.
  • [action] — được thay bằng tên method, ví dụ GetDepartments.
Controllers/DepartmentsController.cscsharp
[Route("api/[controller]")]      // → /api/departments
public class DepartmentsController
{
    [HttpGet]                    // → GET /api/departments
    public string GetDepartments() => "These are the departments.";

    [HttpGet("{id:int}")]        // → GET /api/departments/42
    public string GetDepartmentById(int id) => $"Department info: {id}";

    [HttpPost]                   // → POST /api/departments
    public string Create() => "Created a new department";

    [HttpDelete("{id:int}")]     // → DELETE /api/departments/5
    public string Delete(int id) => $"Deleted department: {id}";
}

Route constraints (ràng buộc tham số route)

Có thể ràng buộc kiểu dữ liệu của tham số route ngay trong template để tránh nhận request sai kiểu:

ConstraintÝ nghĩaVí dụ
:intPhải là số nguyên{id:int}
:guidPhải là GUID hợp lệ{id:guid}
:minlength(n)Độ dài tối thiểu{name:minlength(3)}
:range(min,max)Phạm vi số{age:range(1,120)}
:alphaChỉ chữ cái{slug:alpha}

💡Mọi kiến thức về route template đã học ở Minimal API (route parameter, optional parameter ?, route constraint) đều áp dụng y hệt cho attribute routing của controller.

3 Conventional Routing

Bài 3 · Conventional Routing

Conventional routing (định tuyến theo quy ước) là cách định nghĩa một route template chung (master template) áp dụng cho toàn bộ ứng dụng. Thay vì đặt attribute riêng lên từng action, framework sẽ tự động ánh xạ URL vào controller/action dựa trên quy ước đặt tên.

Kích hoạt Conventional Routing

Thay MapControllers() bằng MapControllerRoute() và cung cấp pattern:

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();

var app = builder.Build();

app.UseRouting();

app.UseEndpoints(endpoints =>
{
    endpoints.MapControllerRoute(
        name: "default",
        pattern: "{controller=Home}/{action=Index}/{id?}"
    );
});

app.Run();

Giải mã master template

SegmentÝ nghĩaVí dụ URL → Đích
{controller=Home}Tên controller, default là Home/ → HomeController
{action=Index}Tên action method, default là Index/Departments → action Index của DepartmentsController
{id?}Tham số tùy chọn/Departments/Details/5 → action Details, id=5

Controller tuân theo quy ước

Khi dùng conventional routing, tên action method phải khớp với segment thứ 2 trong URL. Tên thường theo chuẩn CRUD:

Controllers/DepartmentsController.cscsharp
using Microsoft.AspNetCore.Mvc;

namespace WebApp.Controllers
{
    public class DepartmentsController
    {
        // GET /Departments  hoặc  GET /Departments/Index
        public string Index()
        {
            return "These are the departments.";
        }

        // GET /Departments/Details/5
        public string Details(int? id)
        {
            return $"Department info: {id}";
        }

        // POST /Departments/Create
        [HttpPost]
        public string Create()
        {
            return "Created a new department";
        }

        // POST /Departments/Edit/5
        [HttpPost]
        public string Edit(int? id)
        {
            return $"Updated department: {id}";
        }

        // POST /Departments/Delete/5
        [HttpPost]
        public string Delete(int? id)
        {
            return $"Deleted department: {id}";
        }
    }
}

ℹ️Tại sao Create/Edit/Delete dùng [HttpPost]? Trong web app truyền thống (có HTML form), trình duyệt chỉ gửi được GET hoặc POST. Nút Submit của form luôn tạo ra request POST, vì vậy các action thay đổi dữ liệu phải chấp nhận POST. Với API thuần túy (không có HTML form), bạn nên dùng [HttpPut]/[HttpDelete] cho đúng ngữ nghĩa REST.

HomeController — xử lý route mặc định

Vì master template có default controller=Home, action=Index, cần tạo HomeController để xử lý request vào / (trang chủ):

Controllers/HomeController.cscsharp
namespace WebApp.Controllers
{
    public class HomeController
    {
        // Xử lý GET /  (vì controller=Home, action=Index là default)
        public string Index()
        {
            return "Hello World!";
        }
    }
}

Attribute routing ghi đè Conventional routing

Trong dự án dùng conventional routing, vẫn có thể ghi đè route của một action cụ thể bằng attribute. Khi action có attribute route, nó không còn tham gia vào conventional routing nữa — chỉ phản hồi đúng URL đã khai báo trong attribute.

Controllers/DepartmentsController.cs (kết hợp)csharp
public class DepartmentsController
{
    // Dùng conventional routing: GET /Departments/Index
    public string Index() => "These are the departments.";

    // Ghi đè bằng attribute routing: chỉ nhận GET /Department/{id?}
    // Conventional routing KHÔNG còn map vào action này nữa
    [HttpGet("/Department/{id?}")]
    public string Details(int? id) => $"Department info: {id}";
}

So sánh hai kiểu routing

Attribute RoutingConventional Routing
Khai báo routeTrực tiếp trên action/classMột template chung trong Program.cs
Phù hợp vớiWeb API (REST), route linh hoạtWeb app truyền thống (MVC + Views)
Độ rõ ràngCao — nhìn vào action biết ngay URLThấp hơn — phải biết quy ước ngầm
Kết hợpCó thể dùng song song: attribute ghi đè conventional cho action cụ thể

⚠️Lưu ý: Một action vừa có attribute route vừa nằm trong dự án dùng conventional routing sẽ chỉ phản hồi URL trong attribute. Nếu người dùng truy cập URL theo quy ước conventional, họ sẽ nhận 404.

💡Scaffold nhanh: Trong Visual Studio, bạn có thể click chuột phải vào thư mục Controllers → Add Controller → chọn template MVC Controller with read/write actions. Framework sẽ tự sinh sẵn các action Index / Details / Create / Edit / Delete theo đúng quy ước conventional routing.
