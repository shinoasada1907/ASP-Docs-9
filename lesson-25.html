CHƯƠNG 25

Configuration trong ASP.NET Core

Hệ thống cấu hình ứng dụng — cơ chế hoạt động theo mô hình provider, cách đọc giá trị bằng nhiều phương thức, thứ tự ưu tiên khi nhiều nguồn xung đột, và cách dùng Options pattern để truy cập cấu hình theo kiểu strongly typed.

1 Cơ chế hoạt động của Configuration

Bài 1 · How Configuration Works

Configuration (cấu hình) là tập hợp các giá trị dữ liệu điều chỉnh hành vi của ứng dụng mà không cần sửa code. Dữ liệu này có thể đến từ nhiều nguồn khác nhau: file JSON, biến môi trường, command-line arguments, user secrets, cơ sở dữ liệu…

Mô hình provider

ASP.NET Core thiết kế hệ thống cấu hình theo mô hình plug-in: một lớp trung tâm gọi là ConfigurationManager chịu trách nhiệm nạp và tổng hợp dữ liệu từ nhiều configuration provider (nhà cung cấp cấu hình). Mỗi provider đọc một loại nguồn và đổ dữ liệu vào ConfigurationManager. Ứng dụng chỉ cần truy vấn một nơi duy nhất — ConfigurationManager — thay vì phải biết nguồn dữ liệu đến từ đâu.

Provider mặc địnhNguồn dữ liệu
JSON appsettings.jsonFile JSON gốc, có trong mọi project
JSON appsettings.{Environment}.jsonFile JSON theo môi trường (Development, Production…)
Secrets User SecretsFile bí mật cục bộ của nhà phát triển (không commit lên git)
ENV Environment VariablesBiến môi trường của hệ điều hành hoặc cloud
CLI Command-line ArgumentsTham số truyền khi chạy dotnet run

Khi bạn gọi WebApplication.CreateBuilder(args), ASP.NET Core tự động đăng ký tất cả các provider mặc định trên vào ConfigurationManager theo đúng thứ tự đó. ConfigurationManager được đưa vào DI container, giúp toàn bộ ứng dụng có thể lấy cấu hình thông qua Dependency Injection.

appsettings.jsonjson
{
  "AllowedHosts": "*",
  "ConnectionStrings": {
    "DefaultConnection": "Server=localhost;Database=MyDb;Trusted_Connection=True;"
  },
  "DisplaySettings": {
    "ShowDepartment": true,
    "ShowEmployees": false,
    "SubSettings": {
      "SubSetting1": 1,
      "SubSetting2": 10
    }
  }
}
appsettings.Development.jsonjson
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning"
    }
  },
  "DisplaySettings": {
    "SubSettings": {
      "SubSetting2": 5
    }
  }
}

ℹ️ Nếu muốn tuỳ chỉnh danh sách provider, bạn có thể xoá tất cả provider mặc định bằng builder.Configuration.Sources.Clear() rồi tự thêm từng provider theo ý muốn. Tuy nhiên với đa số dự án, danh sách mặc định là đủ.

Program.cs — tuỳ chỉnh provider (tuỳ chọn)csharp
var builder = WebApplication.CreateBuilder(args);

// Xoá tất cả provider mặc định rồi chỉ thêm provider JSON tự chọn
builder.Configuration.Sources.Clear();
builder.Configuration.AddJsonFile("appsettings.json", optional: false, reloadOnChange: true);
builder.Configuration.AddJsonFile(
    $"appsettings.{builder.Environment.EnvironmentName}.json",
    optional: true,
    reloadOnChange: true);
builder.Configuration.AddEnvironmentVariables();

var app = builder.Build();
app.Run();

🎯Điểm mấu chốt: ConfigurationManager implement interface IConfiguration. Đây là interface mà ứng dụng dùng để truy cập giá trị cấu hình — không phụ thuộc vào nguồn cụ thể nào.

2 Đọc giá trị cấu hình

Bài 2 · Access Configuration Values

ASP.NET Core cung cấp nhiều cách để đọc giá trị từ IConfiguration. Bạn có thể truy cập cấu hình ở hai giai đoạn trong Program.cs: trước builder.Build() (dùng builder.Configuration) và sau builder.Build() (dùng app.Configuration). Ngoài ra bạn có thể inject IConfiguration vào bất kỳ service hay endpoint handler nào.

Cách 1 — Indexer (dấu ngoặc vuông)

Đơn giản nhất: truyền key dưới dạng string. Với key lồng nhau, dùng dấu : làm phân cách cấp.

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Đọc trước Build() — dùng builder.Configuration
string? connStr = builder.Configuration["ConnectionStrings:DefaultConnection"];

// Đọc sau Build() — dùng app.Configuration
bool showDept = bool.Parse(app.Configuration["DisplaySettings:ShowDepartment"] ?? "false");

// Lồng nhiều cấp
string? sub1 = app.Configuration["DisplaySettings:SubSettings:SubSetting1"];

app.MapGet("/", () => $"ShowDepartment={showDept}");
app.Run();

Cách 2 — GetValue<T>

GetValue<T> tự động convert kiểu dữ liệu, tránh phải parse thủ công. Tham số thứ hai là giá trị mặc định khi key không tồn tại.

Program.cscsharp
bool showDept = app.Configuration.GetValue<bool>("DisplaySettings:ShowDepartment", defaultValue: false);
int sub1     = app.Configuration.GetValue<int>("DisplaySettings:SubSettings:SubSetting1", 0);

Cách 3 — GetSection

GetSection("SectionName") trả về một IConfigurationSection — bản thân nó cũng là IConfiguration nhưng có gốc từ section đó. Từ đây bạn có thể dùng tiếp indexer hoặc GetValue với key ngắn hơn.

Program.cscsharp
var displaySection = app.Configuration.GetSection("DisplaySettings");

bool showDept  = displaySection.GetValue<bool>("ShowDepartment");
bool showEmp   = displaySection.GetValue<bool>("ShowEmployees");

// Section lồng nhau
var subSection = displaySection.GetSection("SubSettings");
int sub2 = subSection.GetValue<int>("SubSetting2");

Cách 4 — GetConnectionString

Phương thức tiện ích để đọc nhanh chuỗi kết nối trong mục ConnectionStrings.

Program.cscsharp
// Tương đương: configuration["ConnectionStrings:DefaultConnection"]
string? connStr = builder.Configuration.GetConnectionString("DefaultConnection");

Inject IConfiguration vào endpoint và service

Trong Minimal API, bạn có thể khai báo IConfiguration trực tiếp trong tham số của endpoint handler. Trong Controller hay Razor Page, dùng constructor injection.

Program.cs — Minimal API endpointcsharp
app.MapGet("/info", (IConfiguration config) =>
{
    bool showDept = config.GetValue<bool>("DisplaySettings:ShowDepartment");
    bool showEmp  = config.GetValue<bool>("DisplaySettings:ShowEmployees");
    return Results.Ok(new { ShowDepartment = showDept, ShowEmployees = showEmp });
});
HomeController.cs — MVC Controllercsharp
public class HomeController : Controller
{
    private readonly IConfiguration _config;

    public HomeController(IConfiguration config)
    {
        _config = config;
    }

    public IActionResult Index()
    {
        bool showDept = _config.GetValue<bool>("DisplaySettings:ShowDepartment");
        ViewBag.ShowDepartment = showDept;
        return View();
    }
}

💡 Dùng dấu : để phân cách các cấp key lồng nhau khi truy cập bằng indexer hoặc GetValue. Ví dụ: key "DisplaySettings:SubSettings:SubSetting1" tương ứng với JSON { "DisplaySettings": { "SubSettings": { "SubSetting1": 1 } } }.

3 Ghi đè theo thứ tự nguồn

Bài 3 · Configuration Values Overriding

Khi nhiều provider cùng cung cấp một key, provider nào được đăng ký sau sẽ có độ ưu tiên cao hơn và ghi đè giá trị của provider trước đó. Đây là nguyên tắc cốt lõi giúp bạn cấu hình môi trường linh hoạt mà không cần sửa code.

Thứ tự ưu tiên mặc định (từ thấp đến cao)

  1. appsettings.json — giá trị gốc, cơ sở cho mọi môi trường.
  2. appsettings.{Environment}.json — ghi đè theo môi trường cụ thể (Development, Staging, Production…). Môi trường được xác định qua biến ASPNETCORE_ENVIRONMENT trong launchSettings.json hoặc hệ điều hành.
  3. User Secrets — chỉ kích hoạt trong môi trường Development; lưu thông tin nhạy cảm (mật khẩu, API key) trên máy local mà không commit lên source control.
  4. Environment Variables — ghi đè mọi file cấu hình; phù hợp cho production và CI/CD pipeline.
  5. Command-line Arguments — ưu tiên cao nhất; hữu ích để override tạm thời khi chạy thử.

⚠️ Khi đặt biến môi trường cho key lồng nhau trên Windows, dùng dấu __ (hai gạch dưới) thay vì : vì Windows không cho phép dấu hai chấm trong tên biến môi trường. Ví dụ: DisplaySettings__SubSettings__SubSetting2=500. Trên Linux/macOS, dùng : bình thường.

Ví dụ minh hoạ

Giả sử appsettings.json có DisplaySettings:SubSettings:SubSetting2 = 1, và appsettings.Development.json đặt giá trị đó là 10. Khi chạy ở môi trường Development, ứng dụng sẽ đọc được 10.

appsettings.jsonjson
{
  "DisplaySettings": {
    "ShowDepartment": true,
    "SubSettings": {
      "SubSetting2": 1
    }
  }
}
appsettings.Development.jsonjson
{
  "DisplaySettings": {
    "SubSettings": {
      "SubSetting2": 10
    }
  }
}
Terminal — ghi đè bằng biến môi trường (Windows)bash
# Đặt biến môi trường để ghi đè SubSetting2 thành 500
$env:DisplaySettings__SubSettings__SubSetting2 = "500"
dotnet run
Terminal — ghi đè bằng command-line argumentbash
dotnet run --DisplaySettings:SubSettings:SubSetting2=999

Ứng dụng thực tế: bảo mật connection string

Thay vì để username/password trong appsettings.json rồi commit lên git, hãy để appsettings.json chứa giá trị giả (placeholder), còn thông tin thật nằm trong biến môi trường hoặc User Secrets. Provider nào được nạp sau sẽ ghi đè, đảm bảo thông tin nhạy cảm không rò rỉ ra source control.

appsettings.json — chỉ có placeholder, an toàn khi commitjson
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=localhost;Database=MyDb;User Id=DEV_USER;Password=DEV_PASS;"
  }
}
Terminal — biến môi trường production (ghi đè toàn bộ)bash
export ConnectionStrings__DefaultConnection="Server=prod-sql;Database=ProdDb;User Id=sa;Password=S3cur3!"

🎯Quy tắc vàng: thông tin nhạy cảm (mật khẩu, API key) không bao giờ được commit lên source control. Dùng User Secrets (local development) hoặc biến môi trường / Azure Key Vault (production) để cung cấp giá trị thực, nhờ vào cơ chế ghi đè theo thứ tự nguồn.

4 Strongly Typed Configuration (Options Pattern)

Bài 4 · Strongly Typed Configuration Class

Việc đọc cấu hình bằng chuỗi key thủ công dễ dẫn đến lỗi typo và khó refactor. Options pattern (cấu hình strongly typed) giải quyết vấn đề này: bạn tạo một class C# có các property tương ứng với section trong appsettings.json, rồi bind class đó vào DI container. Từ đó, bất kỳ service nào cũng có thể inject class cấu hình trực tiếp mà không cần biết key string.

Bước 1 — Tạo class cấu hình

Property name của class phải khớp (case-insensitive) với key trong JSON. Không cần attribute hay kế thừa gì đặc biệt.

Settings/SubSettings.cscsharp
namespace MyApp.Settings;

public class SubSettings
{
    public int SubSetting1 { get; set; }
    public int SubSetting2 { get; set; }
}
Settings/DisplaySettings.cscsharp
namespace MyApp.Settings;

public class DisplaySettings
{
    // Tên section trong appsettings.json — dùng nameof() để tránh magic string
    public const string SectionName = "DisplaySettings";

    public bool ShowDepartment { get; set; }
    public bool ShowEmployees  { get; set; }

    // Property lồng nhau — tương ứng với sub-object trong JSON
    public SubSettings SubSettings { get; set; } = new();
}

Bước 2 — Đăng ký vào DI với Configure<T>

Program.cscsharp
using MyApp.Settings;

var builder = WebApplication.CreateBuilder(args);

// Bind section "DisplaySettings" trong appsettings.json vào class DisplaySettings
builder.Services.Configure<DisplaySettings>(
    builder.Configuration.GetSection(DisplaySettings.SectionName));

var app = builder.Build();

app.MapGet("/", () => "Hello World!");

// Endpoint sử dụng Options pattern
app.MapGet("/settings", (IOptions<DisplaySettings> opts) =>
{
    DisplaySettings s = opts.Value;
    return Results.Ok(new
    {
        s.ShowDepartment,
        s.ShowEmployees,
        SubSetting1 = s.SubSettings.SubSetting1,
        SubSetting2 = s.SubSettings.SubSetting2
    });
});

app.Run();

Bước 3 — Inject IOptions<T> hoặc IOptionsSnapshot<T>

InterfaceLifecycleKhi nào dùng
IOptions<T> Singleton — đọc một lần lúc khởi động Cấu hình không thay đổi trong suốt vòng đời ứng dụng
IOptionsSnapshot<T> Scoped — đọc lại mỗi request Cấu hình có thể thay đổi (reloadOnChange) và bạn muốn mỗi request dùng giá trị mới nhất
IOptionsMonitor<T> Singleton — theo dõi thay đổi real-time Background service cần phản ứng ngay khi cấu hình thay đổi (không cần restart)
Services/ReportService.cs — inject IOptions<T>csharp
using Microsoft.Extensions.Options;
using MyApp.Settings;

public class ReportService
{
    private readonly DisplaySettings _settings;

    public ReportService(IOptions<DisplaySettings> options)
    {
        _settings = options.Value;
    }

    public string GetReportSummary()
    {
        var parts = new List<string>();
        if (_settings.ShowDepartment) parts.Add("Department");
        if (_settings.ShowEmployees)  parts.Add("Employees");
        return parts.Count > 0
            ? $"Showing: {string.Join(", ", parts)}"
            : "Nothing to show.";
    }
}
Program.cs — đăng ký service và dùng IOptionsSnapshotcsharp
builder.Services.Configure<DisplaySettings>(
    builder.Configuration.GetSection(DisplaySettings.SectionName));

builder.Services.AddScoped<ReportService>();

// IOptionsSnapshot<T> — đọc lại mỗi request (scoped)
app.MapGet("/report", (IOptionsSnapshot<DisplaySettings> snap, ReportService svc) =>
{
    DisplaySettings current = snap.Value;
    return Results.Ok(new
    {
        Summary       = svc.GetReportSummary(),
        SubSetting2   = current.SubSettings.SubSetting2
    });
});

Bind thủ công (không dùng DI)

Nếu bạn cần đọc cấu hình strongly typed ngay trong Program.cs trước khi DI sẵn sàng, dùng phương thức Bind hoặc Get<T>:

Program.cs — bind thủ côngcsharp
// Cách 1: Bind vào instance
var displaySettings = new DisplaySettings();
builder.Configuration.GetSection(DisplaySettings.SectionName).Bind(displaySettings);

// Cách 2: Get<T> — ngắn gọn hơn
DisplaySettings? ds = builder.Configuration
    .GetSection(DisplaySettings.SectionName)
    .Get<DisplaySettings>();

Console.WriteLine($"ShowDepartment = {displaySettings.ShowDepartment}");

💡 Ưu tiên dùng IOptions<T> thay vì IConfiguration khi inject vào service. Options pattern giúp code tường minh hơn, dễ unit test hơn (mock một class đơn giản thay vì mock IConfiguration), và loại bỏ hoàn toàn magic string.

⚠️ Tên property của class C# phải khớp chính xác (case-insensitive) với key trong JSON. Nếu tên không khớp, property sẽ giữ giá trị mặc định và không báo lỗi — đây là lỗi thầm lặng khó debug.
