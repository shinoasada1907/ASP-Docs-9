CHƯƠNG 07

Tổ chức Code & Dependency Injection

Tìm hiểu cách tổ chức code cho Minimal API bằng extension method, giải quyết vấn đề tightly-coupled bằng Dependency Injection, và nắm vững ba vòng đời dịch vụ: Singleton, Scoped, Transient trong ASP.NET Core .NET 9.

1 Tổ chức Minimal API Endpoint

Bài 1 · Organize Minimal Api Endpoints

Khi số lượng endpoint tăng lên, nhồi nhét tất cả vào Program.cs sẽ làm file này trở nên cồng kềnh và khó bảo trì. ASP.NET Core cho phép tách các endpoint ra từng class riêng thông qua extension method trên WebApplication.

Vấn đề với Program.cs quá lớn

Nguyên tắc chung: Program.cs chỉ nên chứa cấu hình ứng dụng (đăng ký service, cấu hình middleware), không nên chứa bất kỳ logic nghiệp vụ hay định nghĩa endpoint nào. Khi có nhiều nhóm endpoint (employees, departments, products…), mỗi nhóm nên nằm trong một file riêng.

Giải pháp: Extension Method trên WebApplication

Tạo một thư mục Endpoints/, bên trong tạo các static class, mỗi class chứa một static method mở rộng WebApplication. Vì app trong Program.cs có kiểu WebApplication, ta có thể gọi trực tiếp phương thức mở rộng này lên app.

ℹ️Extension method phải nằm trong static class và tham số đầu tiên dùng từ khóa this để chỉ định kiểu được mở rộng.

Cấu trúc thư mục đề xuất

MyApp/
├── Program.cs
├── Endpoints/
│   ├── EmployeeEndpoints.cs
│   └── DepartmentEndpoints.cs
└── Repositories/
    └── EmployeeRepository.cs

Bước 1 — Tạo class chứa endpoint

Endpoints/EmployeeEndpoints.cscsharp
using Microsoft.AspNetCore.Http;

namespace MyApp.Endpoints;

public static class EmployeeEndpoints
{
    public static void MapEmployeeEndpoints(this WebApplication app)
    {
        app.MapGet("/employees", () =>
        {
            return EmployeeRepository.GetAll();
        });

        app.MapGet("/employees/{id}", (int id) =>
        {
            var employee = EmployeeRepository.GetById(id);
            return employee is null ? Results.NotFound() : Results.Ok(employee);
        });

        app.MapPost("/employees", (Employee employee) =>
        {
            EmployeeRepository.Add(employee);
            return Results.Created($"/employees/{employee.Id}", employee);
        });

        app.MapPut("/employees/{id}", (int id, Employee updated) =>
        {
            var result = EmployeeRepository.Update(id, updated);
            return result ? Results.NoContent() : Results.NotFound();
        });

        app.MapDelete("/employees/{id}", (int id) =>
        {
            var result = EmployeeRepository.Delete(id);
            return result ? Results.Ok() : Results.NotFound();
        });
    }
}

Bước 2 — Gọi extension method từ Program.cs

Program.cscsharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Thay vì định nghĩa hàng chục endpoint ở đây,
// chỉ cần gọi một dòng:
app.MapEmployeeEndpoints();

// Sau này thêm nhóm mới cũng rất gọn:
// app.MapDepartmentEndpoints();

app.Run();

💡Với cách này, mỗi nhóm endpoint nằm trong một file riêng biệt. Khi nhóm phát triển cần thêm endpoint mới cho Department, chỉ cần tạo file DepartmentEndpoints.cs và thêm một dòng gọi trong Program.cs mà không làm ảnh hưởng đến phần còn lại.

2 Vấn đề phụ thuộc (Dependency Problem)

Bài 2 · Dependency Problem

Sau khi tách endpoint ra file riêng, code trở nên gọn hơn về cấu trúc, nhưng vẫn còn một vấn đề kiến trúc nghiêm trọng: tightly coupling (phụ thuộc chặt chẽ) giữa tầng API và tầng dữ liệu.

Biểu hiện của tightly coupling

Trong ví dụ trước, EmployeeEndpoints gọi trực tiếp EmployeeRepository — một class cụ thể. Điều này tạo ra phụ thuộc trực tiếp:

Endpoints/EmployeeEndpoints.cs — vấn đềcsharp
// Tầng API biết tường tận về tầng dữ liệu cụ thể
// Đây là tightly coupling!
app.MapGet("/employees", () =>
{
    return EmployeeRepository.GetAll(); // gọi trực tiếp class
});

Hệ quả của tightly coupling

Vấn đềMô tả
Khó bảo trìNếu đổi tên hoặc refactor EmployeeRepository, phải sửa tất cả nơi đang dùng nó.
Không thể làm song songNhóm phát triển API phải chờ nhóm phát triển data layer xong mới làm được.
Khó thay thếMuốn đổi từ in-memory sang database thật phải sửa code ở nhiều chỗ.
Khó testKhông thể mock hoặc fake repository khi viết unit test.

⚠️Nguyên tắc: Tầng cao hơn (API definition) không nên biết về tầng thấp hơn (Repository cụ thể). Chúng ta cần một cơ chế để tách rời (decouple) hai tầng này. Đó là lý do ra đời của Dependency Injection.

Giải pháp — Dependency Injection

Dependency Injection (DI) được xây dựng dựa trên hai nguyên tắc:

  1. Dependency Inversion Principle (DIP) — Nguyên tắc đảo ngược phụ thuộc: cả hai tầng đều phụ thuộc vào một abstraction (interface), không phụ thuộc vào nhau trực tiếp.
  2. Inversion of Control (IoC) — Nguyên tắc đảo ngược điều khiển: việc tạo và quản lý vòng đời của đối tượng được trao cho framework, không phải code nghiệp vụ tự xử lý.

3 Nguyên tắc Đảo ngược Phụ thuộc (DIP)

Bài 3 · Dependency Inversion Principle

Dependency Inversion Principle (DIP) là chữ D trong SOLID. Nguyên tắc này phát biểu: các module cấp cao không nên phụ thuộc vào module cấp thấp — cả hai nên phụ thuộc vào abstraction.

Sơ đồ trước khi áp dụng DIP

Hiện tại cả hai component đều biết nhau trực tiếp:

EmployeeEndpoints  ──────depends on──────>  EmployeeRepository
   (higher component)                         (lower component)

Sơ đồ sau khi áp dụng DIP

Giới thiệu một interface làm trung gian — cả hai đều phụ thuộc vào interface, không phụ thuộc vào nhau:

EmployeeEndpoints  ──depends on──>  IEmployeeRepository  <──implements──  EmployeeRepository
   (higher)                          (abstraction)                            (lower)

Bước 1 — Định nghĩa Interface (Abstraction)

Repositories/IEmployeeRepository.cscsharp
namespace MyApp.Repositories;

public interface IEmployeeRepository
{
    IEnumerable<Employee> GetAll();
    Employee? GetById(int id);
    void Add(Employee employee);
    bool Update(int id, Employee updated);
    bool Delete(int id);
}

Bước 2 — Class cụ thể implement Interface

Repositories/EmployeeRepository.cscsharp
namespace MyApp.Repositories;

public class EmployeeRepository : IEmployeeRepository
{
    private readonly List<Employee> _employees = new()
    {
        new Employee { Id = 1, Name = "Alice", Salary = 80_000 },
        new Employee { Id = 2, Name = "Bob",   Salary = 90_000 },
        new Employee { Id = 3, Name = "Carol",  Salary = 75_000 },
    };

    public IEnumerable<Employee> GetAll() => _employees;

    public Employee? GetById(int id) =>
        _employees.FirstOrDefault(e => e.Id == id);

    public void Add(Employee employee)
    {
        employee.Id = _employees.Max(e => e.Id) + 1;
        _employees.Add(employee);
    }

    public bool Update(int id, Employee updated)
    {
        var existing = GetById(id);
        if (existing is null) return false;
        existing.Name   = updated.Name;
        existing.Salary = updated.Salary;
        return true;
    }

    public bool Delete(int id)
    {
        var existing = GetById(id);
        if (existing is null) return false;
        _employees.Remove(existing);
        return true;
    }
}

Bước 3 — Endpoint dùng Interface thay vì Class cụ thể

Endpoints/EmployeeEndpoints.cscsharp
namespace MyApp.Endpoints;

public static class EmployeeEndpoints
{
    public static void MapEmployeeEndpoints(this WebApplication app)
    {
        // IEmployeeRepository được inject vào, không tự new
        app.MapGet("/employees", (IEmployeeRepository repo) =>
            repo.GetAll());

        app.MapGet("/employees/{id}", (int id, IEmployeeRepository repo) =>
        {
            var emp = repo.GetById(id);
            return emp is null ? Results.NotFound() : Results.Ok(emp);
        });

        app.MapPost("/employees", (Employee emp, IEmployeeRepository repo) =>
        {
            repo.Add(emp);
            return Results.Created($"/employees/{emp.Id}", emp);
        });

        app.MapPut("/employees/{id}", (int id, Employee emp, IEmployeeRepository repo) =>
            repo.Update(id, emp) ? Results.NoContent() : Results.NotFound());

        app.MapDelete("/employees/{id}", (int id, IEmployeeRepository repo) =>
            repo.Delete(id) ? Results.Ok() : Results.NotFound());
    }
}

🎯Điểm mấu chốt: EmployeeEndpoints giờ chỉ biết về IEmployeeRepository (interface), không biết gì về EmployeeRepository (class cụ thể). Hai tầng đã được decouple. Nhưng ai sẽ tạo instance của EmployeeRepository và truyền vào? Đó là nhiệm vụ của IoC Container.

4 Nguyên tắc Đảo ngược Điều khiển (IoC)

Bài 4 · Inversion of Control principle

Inversion of Control (IoC) giải quyết câu hỏi còn lại sau khi áp dụng DIP: ai sẽ tạo ra instance của EmployeeRepository và truyền vào đúng chỗ?

Vấn đề nếu tự tạo instance

Nếu không có IoC, lập trình viên phải tự viết:

Tình huống không dùng IoC — saicsharp
// Higher component vẫn biết về EmployeeRepository (lower)
// vì phải tự new nó lên — vẫn tightly coupled!
IEmployeeRepository repo = new EmployeeRepository();

Cách làm trên vi phạm DIP vì tầng cao vẫn phụ thuộc vào tầng thấp (thông qua lệnh new). IoC giải quyết điều này bằng cách chuyển giao quyền tạo đối tượng cho framework.

IoC Container trong ASP.NET Core

ASP.NET Core tích hợp sẵn một DI Container (còn gọi là IoC Container). Lập trình viên đăng ký ánh xạ interface → implementation vào container. Sau đó, mỗi khi có component nào cần interface đó, framework tự động tạo instance phù hợp và inject (tiêm) vào.

Program.cscsharp
using MyApp.Endpoints;
using MyApp.Repositories;

var builder = WebApplication.CreateBuilder(args);

// Đăng ký vào DI Container:
// "Khi ai đó cần IEmployeeRepository, hãy tạo EmployeeRepository"
builder.Services.AddSingleton<IEmployeeRepository, EmployeeRepository>();

var app = builder.Build();

app.MapEmployeeEndpoints();

app.Run();

Cách ASP.NET Core inject vào Minimal API handler

Đối với Minimal API, framework sử dụng parameter injection: bất kỳ tham số nào của handler delegate có kiểu được đăng ký trong DI Container sẽ được tự động inject. Framework nhận biết qua attribute [FromServices] (có thể viết tường minh hoặc bỏ qua vì framework tự suy luận).

Inject tường minh vs ngầm địnhcsharp
// Ngầm định — framework tự nhận biết IEmployeeRepository là service
app.MapGet("/employees", (IEmployeeRepository repo) =>
    repo.GetAll());

// Tường minh — dùng [FromServices] khi cần rõ ràng
app.MapGet("/employees", ([FromServices] IEmployeeRepository repo) =>
    repo.GetAll());

Luồng hoạt động tổng thể

  1. Ứng dụng khởi động — DI Container được khởi tạo với các ánh xạ đã đăng ký.
  2. Client gửi request GET /employees.
  3. Framework tìm handler phù hợp (MapGet("/employees", ...)).
  4. Thấy handler cần IEmployeeRepository — container tạo (hoặc lấy) instance EmployeeRepository.
  5. Instance được inject vào tham số repo của handler.
  6. Handler thực thi, trả về response.

💡Khi dùng MVC Controller, cách inject khác một chút — sẽ học ở chương MVC. Với Minimal API, parameter injection là cơ chế chính và rất tiện lợi.

5 Quản lý vòng đời dịch vụ (Lifetime Management)

Bài 5 · Lifetime Management

Khi đăng ký dịch vụ vào DI Container, bạn phải quyết định vòng đời (lifetime) của nó — tức là DI Container sẽ tạo mới hay tái sử dụng instance khi có yêu cầu. ASP.NET Core cung cấp ba vòng đời:

MethodVòng đờiKhi nào dùng
AddSingleton<> Một instance duy nhất trong suốt vòng đời ứng dụng Dữ liệu in-memory, cache, configuration, shared state
AddScoped<> Một instance cho mỗi HTTP request DbContext (Entity Framework), Unit of Work, service cần dùng chung trong một request
AddTransient<> Tạo mới mỗi lần được inject Service nhẹ, không trạng thái, tính toán đơn thuần

Singleton — một instance suốt đời ứng dụng

Container tạo đúng một instance khi lần đầu tiên được yêu cầu và giữ nó trong bộ nhớ cho đến khi ứng dụng dừng. Tất cả request sau đều nhận cùng một instance.

Program.cs — Singletoncsharp
// EmployeeRepository lưu dữ liệu trong List<Employee> (in-memory)
// Phải dùng Singleton để tất cả request dùng chung một danh sách
builder.Services.AddSingleton<IEmployeeRepository, EmployeeRepository>();

⚠️Nếu dùng AddTransient cho EmployeeRepository, mỗi handler sẽ nhận một danh sách employees khác nhau. Thêm nhân viên qua POST /employees xong gọi GET /employees sẽ không thấy bản ghi vừa thêm — vì đây là hai instance khác nhau!

Transient — tạo mới mỗi lần inject

Mỗi lần DI Container cần inject service này, nó tạo một instance hoàn toàn mới. Instance được dùng xong thì dispose ngay.

Program.cs — Transientcsharp
// Phù hợp với service không giữ trạng thái
builder.Services.AddTransient<IEmailSender, SmtpEmailSender>();

Scoped — một instance cho mỗi HTTP request

Trong phạm vi một HTTP request, tất cả các component yêu cầu service này sẽ nhận cùng một instance. Khi request kết thúc (response được gửi đi), instance được dispose.

Program.cs — Scopedcsharp
// Entity Framework DbContext luôn đăng ký Scoped
// AddDbContext tự dùng Scoped cho bạn
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(connectionString));

// Hoặc service tùy chỉnh cần dùng chung trong một request:
builder.Services.AddScoped<IOrderService, OrderService>();

So sánh bằng ví dụ trực quan

Giả sử ba middleware (A, B, C) cùng xử lý một request và đều cần IMyService:

LifetimeA nhậnB nhậnC nhậnRequest tiếp theo
SingletonInstance #1Instance #1Instance #1Instance #1 (cùng)
ScopedInstance #1Instance #1Instance #1Instance #2 (mới)
TransientInstance #1Instance #2Instance #3Instance #4, #5, #6…

Ví dụ đầy đủ — Program.cs với DI hoàn chỉnh

Program.cscsharp
using MyApp.Endpoints;
using MyApp.Repositories;

var builder = WebApplication.CreateBuilder(args);

// Đăng ký dịch vụ với lifetime phù hợp
builder.Services.AddSingleton<IEmployeeRepository, EmployeeRepository>();
// builder.Services.AddScoped<IOrderService, OrderService>();
// builder.Services.AddTransient<IEmailSender, SmtpEmailSender>();

var app = builder.Build();

// Tổ chức endpoint ra file riêng qua extension method
app.MapEmployeeEndpoints();

app.Run();

🎯Quy tắc chọn lifetime:

  • Dùng Singleton khi dữ liệu/trạng thái phải chia sẻ toàn bộ ứng dụng (in-memory store, cache, cấu hình).
  • Dùng Scoped khi dữ liệu chỉ nên chia sẻ trong phạm vi một request (DbContext, Unit of Work).
  • Dùng Transient khi service không có trạng thái và mỗi lần dùng cần instance độc lập (helpers, validators, email senders).
